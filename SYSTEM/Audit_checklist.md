# Чеклист аудита заметки

> Единый источник правды для проверки заметок. Используется скиллами `/audit-note`, `/expand-stub`, `/new-note` и любыми тематическими create-скиллами.

## Шаг 1: Lint (механика)

Вызвать MCP-tool `vault_lint` с параметром `target: <файл>`.

Tool возвращает полный профиль файла (type, domain, tags, note_kind, links_out, lines) + массив issues. Использовать данные из ответа вместо повторного парсинга. Если lint нашёл issues — включить в отчёт как есть.

## Шаг 2: Прочитать governance-документы

1. **`SYSTEM/Metadata_schema.md`** — обязательные поля, допустимые значения, поля по `note_kind`.
2. **`SYSTEM/Tag_taxonomy.md`** — каноничный набор тегов, правила, маппинг.
3. **`SYSTEM/Linking_guidelines.md`** — правила WikiLinks, обязательные ссылки по `note_kind`.
4. **`SYSTEM/Naming_conventions.md`** — формат имён файлов.

## Шаг 3: Семантические проверки

Lint покрывает механику (наличие полей, лимит тегов, каноничность). Ниже — то, что **только LLM может проверить**.

### 3.1 Frontmatter

- Значения enum-полей сверять с `SYSTEM/enums.yaml::note_kinds.<note_kind>.<field>` — если значение в карточке отсутствует в списке, это **ERROR** (некаконичное значение). Не маппить и не «исправлять» молча — флагать в отчёте, пусть пользователь решает (либо поправить карточку, либо расширить enum структурированным multi-choice вопросом, как это делает `/new-anime` шаг 4).
- Если есть `note_kind` — присутствуют все специфичные поля для этого kind (из [[Metadata_schema]]).
- `created`/`updated` в формате `YYYY-MM-DD`.
- Рекомендуемые поля (`co_authored`, `quality`, `aliases`) — предупреждение если отсутствуют.

### 3.2 Теги — алгоритм подбора

Подбирать теги **по шагам**, а не интуитивно. Каждый шаг — отдельный проход по [[Tag_taxonomy]]:

1. **КАТЕГОРИЯ** — что это за заметка по сути? Если подходящего категорийного тега нет в таксономии — добавить.
2. **ТЕМЫ** — о чём содержание заметки? Только теги, которые добавляют ценность для фильтрации. Не дублировать `type`/`domain`.
3. **ОБЯЗАТЕЛЬНЫЕ по `note_kind`** (из Tag_taxonomy).
4. **НЕГАТИВНАЯ ПРОВЕРКА** — убрать невалидные:
   - Тег совпадает с именем/темой заметки-источника (правило 7 из Tag_taxonomy).
   - Тег дублирует `type` или `domain` — лишний.
   - Общее количество >10 — убрать наименее ценные.
5. **ПРОБЕЛЫ — ОБЯЗАТЕЛЬНОЕ ДЕЙСТВИЕ.** Для каждого нового тега, не присутствующего в [[Tag_taxonomy]]:
   1. Открыть `SYSTEM/Tag_taxonomy.md` (Read).
   2. Определить целевую группу (Категорийная / Тематическая / По `note_kind`).
   3. Добавить строку в таблицу группы через Edit. Формат: `\| tag-name \| <Краткое описание области применения> \|`.
   4. Если таблица группы пуста (`\| \| \|` placeholder из template) — заменить placeholder на нормальную строку.
   5. После добавления — продолжить аудит.

   **Никаких «рекомендуется пользователю добавить»** в отчёте. Таксономия и теги на заметках — **синхронны по контракту**. Governance documents (`Tag_taxonomy`, `Metadata_schema`, `Vault_architecture`) — автоуправляемый журнал, не статичный справочник. Если скилл нашёл нужду — скилл сам и записал.

### 3.3 Ссылки

- Обязательные ссылки по `note_kind` (из [[Linking_guidelines]]).
- Нет дубликатов (один target = один WikiLink на заметку).
- Регистр совпадает с именем файла.
- Все target существуют (проверить через Glob).
- Не менее 1 исходящей, не более 15.
- **Реципрокность связей** — если `note_kind` заметки образует пару в `reciprocity_pairs` (`.claude/vault-manifest.yaml`; для Anime_Base — `[character, character]`): вызвать `vault_asymmetric_links(sourceKind=<note_kind>, targetKind=<note_kind>)`. Любая запись в `pairs[]`, где `source` **или** `target` = эта заметка — **ERROR** (односторонняя связь). Чинить добавлением обратного навигационного линка в карточку `missingReverse.from` (gallery-формат по [[Linking_guidelines]]). Политика волта: связь в разделе `## Связи` **строго двунаправленная**, односторонних быть не должно.

### 3.4 Имя файла

- Соответствует [[Naming_conventions]] (`PascalCase_With_Underscores`, без пробелов, без кириллицы).

## Шаг 4: Контентный аудит

Этот шаг — главное отличие `/audit-note` от механического lint. LLM проверяет и **улучшает содержание** заметки.

### 4.0 Масштаб темы

Перед работой с контентом — оценить масштаб раскрываемой темы:
- **Узкий концепт** — 30–60 строк.
- **Средняя тема** — 80–150 строк.
- **Большая тема** — 150–200+ строк.

Размер заметки должен соответствовать объёму темы. Не сжимать большую тему в 50 строк и не раздувать узкий концепт до 200.

### 4.1 Очистка языкового мусора

Выполнить перед любой контентной правкой — удаление артефактов, не связанных со смыслом заметки:

- **CJK-вкрапления**: китайские иероглифы, японские хирагана/катакана, корейские хангыль — если это не осознанное название на оригинальном языке. Характерный признак генерации от китайских моделей.
- **Unicode-артефакты**: zero-width characters, неразрывные пробелы, полноширинные ASCII-аналоги (全角空格, 全角逗号, и т.п.). Заменить на обычные ASCII.
- **Битая markdown-разметка**: лишние/недостающие `*`, `_`, неверные уровни заголовков, битые ссылки, лишние пустые строки в списках.
- **Опечатки и стилистика**: повторы слов подряд, лишние пробелы перед знаками препинания, двойные пробелы, опечатки в технических терминах.

### 4.2 Проверка фактов

- **Web search** по ключевым утверждениям: даты, цифры, названия, версии.
- Если факт не подтверждается — **удалить или пометить** как неподтверждённый.
- **Никогда не удалять факт без web search подтверждения что он неверен.** Собственная неуверенность модели — не основание для удаления. Сначала проверь, потом удаляй.
- Если факт устарел — **обновить** до актуального состояния.

### 4.3 Удаление шума

- Убрать **домыслы и предположения** без источника («вероятно», «скорее всего», «по слухам»).
- Убрать **разговорный стиль** где неуместен (→ нейтральный информативный тон).
- Убрать **дублирование** информации внутри заметки.

### 4.4 Дополнение содержания

- Если заметка тонкая (<30 строк контента) — **дописать** недостающие секции по типу заметки.
- Если пропущен важный аспект темы — **добавить** (но не раздувать: каждая секция должна нести ценность).
- Добавить **WikiLinks** на существующие заметки волта, где уместно (по правилам из [[Linking_guidelines]]).

### 4.5 Качество текста

- Формулировки точные и лаконичные.
- Нет повторов одной мысли разными словами.
- Таблицы и списки предпочтительнее стен текста.
- Каждая секция имеет конкретную ценность для читателя.

### 4.6 Type-conditional проверки

Если в этом файле определены секции для конкретных `note_kind` — выполнить шаги из соответствующей секции.

> **Расширь под свой волт.** Здесь добавляются проверки, специфичные для конкретных `note_kind` твоего волта (например: для `note_kind: review` — проверка наличия ссылки на обозреваемый объект; для `note_kind: project` — проверка `status` field).

#### Шаблон секции

```markdown
### note_kind: <твой_kind>

#### <Описание проверки>

<Шаги проверки специфичной логики>
```

### note_kind: character

#### Source-сверка `featured_in` (КРИТИЧНО — ловит дрейф Mizuno-класса)

`featured_in` — source-anchored поле ([[Metadata_schema]] § «Source-якорь»). Consistency-машинерия (`vault_asymmetric_links`, reverse-ноги create-скиллов) лишь **тиражирует** его, но не проверяет правдивость — поэтому аудит обязан ре-верифицировать против источника.

1. Прямой web fetch Animeography персонажа по иерархии с fallback: **MAL** (`myanimelist.net/character/<id>` либо каст тайтлов `<mal_url>/characters`) → **AniList** (`Character.media`) → **Wikipedia/Fandom** тайтла.
2. Сверить множество `featured_in` карточки с тайтлами, где источник числит персонажа:
   - Тайтл в `featured_in`, но источник его НЕ числит → **флаг (вероятно лишний).** Если у тайтла есть карточка в волте — проверить каст этого тайтла на источнике; персонажа там нет → предложить удалить из `featured_in` + снять обратную связь в `## Тайтлы`. Это ровно кейс Mizuno→3rd (поле врало, поймал только MAL).
   - Источник числит тайтл, у которого есть карточка в волте, но его НЕТ в `featured_in` → **флаг (недостающий)** + предложить добавить.
3. **Ни один источник недоступен** → сверку пропустить с явной пометкой в отчёте «`featured_in` не верифицирован — источники недоступны», поле **не трогать**. Источники **противоречат** → флаг, не решать молча.
4. Правки `featured_in` тянут синхронное обновление `## Тайтлы` и обратной стороны (`ANIME/<title>::## Персонажи`) — выполнять как reverse-ногу `/new-anime` шаг 7.5 (или отметить в отчёте, если это вне зоны правки одной заметки).

#### Фактчек ОПИСАНИЙ связей в `## Связи` (не только их существования)

`featured_in` source-сверка (выше) проверяет, в каких тайтлах персонаж появляется, — но **не** правдивость самих формулировок связей в `## Связи`. Слабые модели фабрикуют **смысл** связи (кейс Cattleya: `Magnolia_Clara`/`Magnolia_Ann` описаны как «коллеги по CH Postal Company», хотя они клиенты из эп.10 — умирающая мать и дочь, — без канонической связи с персонажем). Механический lint это не видит (ссылка резолвится), `featured_in`-сверка — тоже (это про тайтлы, не про межперсонажные связи).

Для **каждой** записи в `## Связи` (`### В каноне` и `### За кулисами`) проверить **смысловую правдивость** описания:
1. Если у цели есть карточка — сверить с её `## Связи`/`## Биография`: упоминает ли цель этого персонажа и в той же ли роли. Карточка цели **не упоминает** персонажа + роль не подтверждается → **флаг фабрикации**.
2. Если карточки нет — web-проверить связь по источнику (Fandom/Wikipedia тайтла), как факты в §4.2.
3. **Сфабрикованная связь** (роль ложная / связи нет в каноне) → **WRONG_FORMAT**: удалить запись (или исправить роль, если связь реальна, но описана неверно). Не пропагировать в реципрокность — закрывать `vault_asymmetric_links` только по **подтверждённым** связям.

То же зеркально для `note_kind: anime` (описания в каст-таблице) и `person` (`## Ключевые работы`): фактчек содержания записи, не только существования ссылки.

### note_kind: anime

#### Source-сверка каст-таблицы `## Персонажи`

Симметрично character-проверке — тот же инвариант `C ∈ каст(title) ⟺ title ∈ featured_in(C)`.

1. Прямой web fetch каста тайтла: **MAL** `<mal_url>/characters` → **AniList** `Media.characters` → **Wikipedia/Fandom**.
2. Сверить персонажей таблицы `## Персонажи`, **имеющих карточку**, с кастом источника:
   - Персонаж в таблице, но источник его в этом тайтле НЕ числит → флаг (вероятно лишний); если у него есть карточка — строка создаёт ложную связь, предложить убрать.
   - Источник числит персонажа, **у которого есть карточка в волте**, но его нет в таблице → флаг (недостающий) + добавить (forward-ячейка). Порог — наличие карточки, не нарративный вес: любой персонаж с карточкой обязан быть в каст-секции (см. [[Linking_guidelines]] → «Взаимность с anime»).
3. Источники недоступны → пропустить с пометкой, таблицу не трогать.

### 4.7 Пост-редакционная валидация

После применения всех контентных правок — **повторно** проверить ссылки и заголовки:

- Вызвать MCP-tools `vault_broken_links` и `vault_duplicate_links` — правки могли сломать или продублировать ссылки.
- Проверить: нет ли WikiLinks в заголовках (`##`, `###`) — перенести в тело текста.
- Для каждой ссылки `[[Target|display text]]` убедиться, что Target действительно описывает концепцию из display text.

### 4.8 Результат

После всех правок установить:
- `co_authored: <ID текущей модели>`.
- `quality: draft` — **всегда draft**, пользователь подтвердит через `/verify`.
- `updated: <сегодня>`.

## Формат отчёта

```
## Аудит: <имя файла>

### Ошибки (нужно исправить)
- [frontmatter] Отсутствует поле `updated`
- [tags] Тег `xxx` не в каноническом наборе
- [links] Дубликат ссылки: [[Target]] встречается 3 раза

### Предупреждения (рекомендуется)
- [tags] Тема "..." обсуждается в тексте, но соответствующий тег не проставлен
- [frontmatter] Нет поля `co_authored`

### OK
- Имя файла корректно
- Теги в пределах лимита (5/10)
```

Если ошибок и предупреждений нет — краткое: `<имя файла>: все проверки пройдены`.
