# Таксономия тегов

## Правила

1. Все теги — `kebab-case`: только строчные латинские буквы и дефисы.
2. Никакого нелатинского текста в тегах.
3. Никаких имён конкретных сущностей в тегах — использовать `aliases` во frontmatter.
4. Никаких команд, кода или предложений в тегах.
5. Максимум 10 тегов на заметку.
6. Тег должен описывать **содержание** заметки, а не её провенанс или статус.
7. **Тег-имя-сущности не ставится на заметку-источник** — карточка `<Subject>.md` не получает тег `<subject-tag>`. Тег нужен на **других** заметках, связанных с данной по смыслу.

## Структура каноничного набора

> **Эта таксономия — автоуправляемый журнал.** Строки в таблицах добавляются скиллами `/new-note`, `/audit-note`, `/expand-stub` автоматически по мере появления новых тегов в заметках волта (см. Audit_checklist.md, Шаг 3.2 пункт 5). Пользователю руками заполнять **не нужно**.

### Контент-рейтинг / adult-дескрипторы

> **Применять ТОЛЬКО при `content_rating: nsfw` или `explicit`** (карточки `anime`/`manga`). Дескрипторы взрослого контента — классификационные ярлыки для фильтрации/исключения из выборок, **не** приглашение к explicit-описанию в теле. На `sfw`-карточки не ставить. Набор пополняется скиллами по мере появления тегов в карточках (как и прочие категории).

### Обязательные теги по `note_kind`

> Здесь определяются обязательные теги для каждого `note_kind`. Скилл `/audit-note` (Шаг 3.2 → пункт 3) проверяет их наличие. Заполняется тематическими create-скиллами при первом использовании нового `note_kind`.
>
> **Принцип:** не дублировать поля frontmatter в тегах. Если значение однозначно определяется enum-полем (`personal_status`, `format`, `status` и т.п.) — оно туда и записано, тег не нужен. Тег уместен только когда несёт **дополнительное** измерение, которого нет во frontmatter.
