# Соглашения об именовании файлов

## Имена файлов

| Правило | Пример |
| --- | --- |
| Формат: `PascalCase_With_Underscores.md` | `Token_Bucket_Algorithm.md` |
| Одиночные слова/аббревиатуры — как есть | `MCP.md`, `LSP.md` |
| Без пробелов | — |
| Без двойных расширений | — |
| Только латиница | — |
| Кириллические/иные нелатинские названия → `aliases` | `aliases: [Опенок луговой]` |

## Глобальная уникальность basename (ЖЁСТКОЕ правило)

**Имя файла (basename без `.md`) должно быть уникальным по ВСЕМУ волту, а не только внутри своей папки.** Obsidian резолвит `[[WikiLink]]` по basename: две заметки с одинаковым именем в разных папках (`ANIME/Violet_Evergarden.md` и `CHARACTERS/Violet_Evergarden.md`) делают каждую ссылку `[[Violet_Evergarden]]` неоднозначной — граф ломается молча.

**Все create-скиллы (`/new-anime`, `/new-character`, `/new-person`, `/new-studio`, `/new-note`) обязаны** перед записью файла проверить basename через Glob по **всему волту** (`**/<Имя>.md`), не только по целевой папке. Если файл с таким basename уже существует где угодно:

1. **НЕ создавать** клэшащий файл и **НЕ перезаписывать** существующий.
2. Остановиться и предложить пользователю disambiguated-имя (структурированным multi-choice вопросом). Кандидаты в порядке приоритета:
   - **Surname-first / каноничный порядок** — для именованных сущностей: `Violet Evergarden` (персонаж) ↔ `Violet_Evergarden` (тайтл) → персонаж становится `Evergarden_Violet`. Японские персонажи и так в порядке «фамилия→имя» (`Kamado_Tanjiro`), что обычно само разводит коллизию с одноимённым тайтлом.
   - **Суффикс-квалификатор `_(<kind>)`** — для мононимов и случаев, где переупорядочивание не помогает: `Naruto` (персонаж) ↔ `Naruto` (тайтл) → `Naruto_(character)`.
3. Имя файла = финальное (disambiguated) значение; `name_romaji` / `name_primary` при этом сохраняет каноничную форму, расхождение с basename — осознанное (зафиксировать в `aliases` при необходимости).

Коллизия basename = коллизия имени вложения (см. ниже), поэтому уникальность basename закрывает обе.

## Вложения (`attachments/`)

| Правило | Пояснение |
| --- | --- |
| Имя вложения = `<basename заметки>_<роль>.<ext>` | `Kamado_Tanjiro_cover.jpg`, `Miki_Itou_gallery_1.jpg`. Роль — `cover` / `gallery_N` / др. |
| **Расширение СОВПАДАЕТ с реальным типом содержимого** | Скачали WebP → файл `*.webp`, не `*.png`. Определять тип по сигнатуре/`Content-Type`, а не хардкодить расширение. `.png` с JPEG-байтами внутри — баг. |
| **НИКОГДА не перезаписывать существующий файл молча** | Перед сохранением скачанного ассета проверить, занят ли путь (`Test-Path` / `ls`). Если занят и это не тот же управляемый ассет — **стоп**: это коллизия (обычно симптом неуникального basename — см. правило выше). Не клобберить. |

## Имена папок

| Правило | Пример |
| --- | --- |
| `UPPER_CASE` для корневых папок | `SYSTEM/`, `GUIDE/` |
| `PascalCase` для вложенных | `GUIDE/Safety_Procedures/` |

## Специальные случаи

- **WikiLinks после переименования**: обновлять все `[[старое имя]]` → `[[новое имя]]` или `[[новое имя\|отображаемый текст]]`. Скилл `/rename-note` делает это автоматически.
- **Aliases**: при изменении основного имени старое автоматически попадает в `aliases` (для обратной совместимости поиска).
