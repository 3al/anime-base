# Правила связывания заметок (WikiLinks)

## Жёсткие правила

Эти правила обязательны для всех заметок и проверяемы автоматически (`vault_lint`, `vault_broken_links`, `vault_duplicate_links`).

### 1. Одна ссылка — одна цель

Каждый target встречается как WikiLink **максимум один раз** на заметку. Линкуется **первое смысловое упоминание**, повторные — plain text. При исправлении дубликатов — оставлять ссылку на **первом** вхождении, убирать на последующих.

### 2. Где НЕ ставить ссылки

- Внутри YAML frontmatter (там используются имена файлов как plain string).
- В заголовках (`#`, `##`, ...).
- Внутри code blocks (`` ` `` и `` ``` ``).
- В URL и внешних ссылках.

### 3. Минимум исходящих ссылок

Каждая заметка (кроме мета-файлов в `SYSTEM/` и индексных заметок с Dataview-запросами) **должна** иметь хотя бы одну исходящую WikiLink.

### 4. Исходящие ссылки — без жёсткого потолка

**Политического лимита нет.** В формализованном волте gallery/cast-секции (каст тайтла, тайтлы персонажа, ключевые работы студии) штатно дают десятки структурных ссылок — это норма, а не запах. Движок `vault_lint` пока эмитит `too-many-links` WARN при >15 (хардкод в framework-модуле vault-index) — для anime/cast-карточек это **известно-benign шум**, не повод декомпозировать заметку. Запрошена доработка: сделать чек отключаемым per-vault через манифест (как `reciprocity_pairs` / `asymmetry_severity`); **сам движок не патчим** — правка потеряется при обновлении модуля. Для свободных заметок ориентир «слишком много ссылок → возможно, заметка делает слишком много» остаётся здравым смыслом.

### 5. Регистр должен совпадать

WikiLink должен точно совпадать с именем файла по регистру.

### 6. Не линковать несуществующие файлы

Все WikiLinks должны указывать на существующие заметки. Если целевой заметки нет — использовать plain text.

## Когда ставить ссылку

Ссылка уместна, когда:
- Упоминаемый концепт **раскрыт** в другой заметке волта.
- Читатель захочет **перейти** к связанной теме для углубления.
- Заметки в отношениях **«часть — целое»**.

Ссылка НЕ уместна, когда:
- Концепт упомянут **мимоходом** и не является частью темы заметки.
- Связь **тривиальна**.
- Ссылка не помогает навигации — читатель и так знает, куда идти.

**Правило большого пальца:** если убрать ссылку и заметка ничего не потеряет для навигации — ссылка лишняя.

## Обязательные ссылки по типу заметки

Скиллы создания/аудита заметок используют эту секцию для автоматической проверки. Селектор — поле `note_kind` из frontmatter (см. [[Metadata_schema]]). Если `note_kind` не указан — применяются только общие правила по `type`.

> **Расширь под свой волт.** Здесь определяются обязательные ссылки для конкретных `note_kind` твоего волта. Например, для `note_kind: project` можно требовать ссылку на статус-документ; для `note_kind: review` — ссылку на обозреваемый объект. Скиллы `/new-note`, `/audit-note`, `/expand-stub` читают эту секцию и применяют правила.

### Шаблон секции

```markdown
### note_kind: <твой_kind>

Заметка типа `<kind>` **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| ... | ... |
```

### note_kind: anime

Карточка аниме **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| Карточка студии из поля `studio` (`STUDIOS/`) | Граф «студия → её работы», обратная навигация |
| Каждый элемент `related_titles[]` (`ANIME/`) | Связи внутри франшизы: сиквел/приквел/спинофф |
| Каждый `staff[].person` (`PERSONS/`) | Граф «человек → его работы», обратная навигация по ролям |
| Каждый `manga_source[]` (`MANGA/`) | Манга-первоисточник экранизации; обратная навигация к карточке манги (реверс `manga.anime_adaptation`) |
| Персонажи в специальной секции (`## Персонажи` / `## Главные персонажи`) (`CHARACTERS/`) | Навигация к карточкам персонажей |

**WikiLinks на персонажей — только в секции персонажей.** Имена персонажей в других секциях карточки тайтла (синопсис, сюжет, описание команды) остаются **plain text**, даже если карточка персонажа существует. Это намеренно: одно место для навигации, ноль дублей, чистый граф. Скилл `/new-character` соблюдает это правило в reverse-check.

**Структура секции `## Персонажи` — 2-колоночная Markdown-таблица.** Формат: `| Персонаж | Сэйю |` — cast-list в стиле MAL/AniList. Каждая колонка — отдельная gallery-точка с собственным reverse-flow:

| Колонка | Содержимое | Forward-fill | Reverse-upgrade |
| --- | --- | --- | --- |
| **Персонаж** | `[60] [[<File>\|<Имя>]]` для существующих карточек `CHARACTERS/`; plain bold для отсутствующих | `/new-anime` шаг 2 — Glob `CHARACTERS/` + Read frontmatter для cover | `/new-character` шаг 7.5a — найти строку с plain-text именем, апгрейдить col 1 in-place |
| **Сэйю** | `[60] [[<File>\|<Имя>]]` для существующих карточек `PERSONS/<va>` (VA-роль); plain text имени сэйю иначе; `—` если VA вообще не известен | `/new-anime` шаг 2 — Glob `PERSONS/` для каждого VA из MAL/AniList | `/new-person` шаг 7.5c — найти строку с этим персонажем в col 1, апгрейдить col 2 in-place |

**Описание ролей в таблицу не выносится.** Контекст по конкретному персонажу — на его character-карточке (`CHARACTERS/<имя>::## Описание` для открытой информации + `## Биография` с block-level spoiler-callout для плот-арки). Anime-карточка содержит общий синопсис в `## Краткий синопсис` и плот в `## Сюжет` (под `> [!warning]- Спойлеры`). Дублировать описание в таблицу — не нужно; таблица это узкий cast-канал.

**Почему не 3-col `Персонаж | Сэйю | Роль` с inline-спойлерами.** Пробовали обернуть плот-ревелейшены в col 3 через `||...||` (плагин obsidian-inline-spoilers). Markdown-таблицы используют `|` как разделитель колонок; `||spoiler||` ломает структуру (table-formatter режет ячейки на лишние колонки), pipe-escaping `\|\|` ломает детект плагином. Tested → не работает. Окончательно: 2-col, описания на character-карточках.

**Двунаправленный flow для col 1 (персонажи):**

| Триггер | Покрывает случай | Что делает |
| --- | --- | --- |
| `/new-anime` (forward, шаг 2) | Карточка `CHARACTERS/<имя>.md` уже есть на момент создания тайтла | Пишет gallery-запись с миниатюрой (если cover у персонажа есть) или WikiLink без миниатюры |
| `/new-character` (reverse, шаг 7.5a) | Карточка персонажа создаётся позже тайтла, к которому привязана через `featured_in[]` | Находит plain-text запись в col 1 и заменяет на gallery-формат in-place. Col 2 и col 3 не трогает |

**Двунаправленный flow для col 2 (сэйю):**

| Триггер | Покрывает случай | Что делает |
| --- | --- | --- |
| `/new-anime` (forward, шаг 2) | Карточка `PERSONS/<va>.md` уже есть на момент создания тайтла | Пишет gallery-запись сэйю в col 2 |
| `/new-person` (reverse, шаг 7.5c) | VA-карточка создаётся позже, чем anime/character | Для каждой `works[].character` находит соответствующую строку в `ANIME/<title>.md::## Персонажи`, апгрейдит col 2 in-place |

Так покрываются все ситуации, и таблица кастинга в карточке тайтла отражает текущее состояние волта без ручного backfill'а. Backfill через `/audit-note` остаётся fallback'ом для исторических заметок (legacy bullet-формат до перехода на table — конвертация только вручную или через явный аудит-пас).

**Pipe-escaping (vault-wide правило для Markdown-таблиц).** Все `[[X|Y]]` и `![[X|N]]` внутри ячеек таблицы пишутся с экранированием pipe: `[[X\|Y]]`, `![[X\|N]]`. Без экранирования Markdown интерпретирует pipe как разделитель колонок и ломает структуру таблицы.

**Взаимность для `related_titles`:** если A ссылается на B как related, то B должна ссылаться на A. Скилл `/new-anime` проверяет и при необходимости проставляет обратную ссылку (шаг 7.5).

**Взаимность для `staff[].person`:** каждый person из staff[] должен иметь обратную запись в своём `works[]` со ссылкой на этот тайтл и теми же ролями. Скилл `/new-anime` поддерживает это автоматически в cross-update шаге 7.5.

**Поля frontmatter ↔ WikiLinks в теле.** `studio: "Ufotable"` в frontmatter — это plain string foreign-key; саму WikiLink `[[Ufotable]]` обязательно поставить в теле (в секции «## Студия и команда»). Аналогично для `related_titles`, главных персонажей и людей из `staff[]` (упоминать в секции «## Студия и команда»).

### note_kind: person

Карточка person **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| Каждый `works[].title` (`ANIME/` или `MANGA/`) | Граф «человек → его работы» (аниме и манга); источник для запроса «все работы X» |
| Каждый `works[].character` для записей с `roles: [voice-actor]` (`CHARACTERS/`) | Граф «сэйю → озвученные персонажи»; ВА-роль определяется парой `title + character`, а не только title |
| Каждая `STUDIOS/<S>.md` с этим персонажем в `founders[]` или в gallery «## Основатели и ключевые люди» | Граф «человек ↔ студии»; материализуется в gallery-секции «## Студии» в карточке персоны. Сборка через `vault_backlinks(note="<thisPerson>")` + фильтр по `sourcePath`. |
| `birth_place` (если есть карточка локации) | Граф «город/регион → известные уроженцы» |

**Взаимность с anime.staff[]** (production-роли). Записи в `works[]` с production-ролями (всё кроме `voice-actor`) должны соответствовать `staff[]` тех тайтлов: если `works` person'а упоминает `Nausicaa_of_the_Valley_of_the_Wind` с ролью `director`, то в `ANIME/Nausicaa_of_the_Valley_of_the_Wind.md::staff[]` должна быть запись `{person: <этот>, role: director}`. Поддерживается `/new-anime` шаг 7.5 и `/new-person` шаг 7.5a.

**Взаимность с character.voice_actors[]** (VA-роль). Каждая запись `works[]` с `roles: [voice-actor]` и `character: <C>` должна иметь зеркальную запись `{anime: <title>, person: <thisPerson>}` в `CHARACTERS/<C>.md::voice_actors[]`. Поддерживается `/new-person` шаг 7.5b и `/new-character` шаг 7.5c. **Voice-actor НЕ кладётся в `anime.staff[]`** — производственный каст и VA-каст хранятся раздельно (см. `enums.yaml::staff_role` комментарий).

**Взаимность с manga.authors[]** (авторство манги). Записи `works[]` с MANGA-тайтлом и ролью `mangaka`/`novelist` зеркальны `manga.authors[]`: автор с карточкой → запись в `works[]` + строка в `## Ключевые работы` наравне с аниме-работами. Create-time — `/new-manga` reverse-нога; forward (person создан позже манги) — backstop (`/fix-links asymmetric`). Пары `[manga, person]`/`[person, manga]` в `reciprocity_pairs` (ERROR).

**Минимум исходящих.** Person без `works[]` — это stub. Рассматривать как кандидат на `/expand-stub` или удаление.

### note_kind: character

Карточка персонажа **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| Каждый `featured_in[]` (`ANIME/` или `MANGA/`) | Граф «персонаж → тайтлы/произведения, в которых появляется»; обратная навигация из карточки аниме/манги |
| Каждая запись `voice_actors[].person` (`PERSONS/`) | Граф «персонаж → сэйю»; материализуется в gallery-секции `## Сэйю`. VA-роль определяется парой `anime + person` (один сэйю мог озвучивать только в части `featured_in[]`) |
| Упоминания других персонажей в теле — родственники, союзники, антагонисты (`CHARACTERS/`) | Граф «персонаж ↔ персонаж»: семейные и нарративные связи внутри тайтла и франшизы |
| Упоминания производственных персон в теле — character designer, дизайн-вдохновитель и т.п. (`PERSONS/`) | Опционально, если связь исторически значима для персонажа (Навсикая ↔ Миядзаки) |

**Взаимность с anime.** Если `featured_in` содержит ссылку на `ANIME/X`, то карточка `X` **обязана** упоминать персонажа в каст-секции «## Персонажи» — для **любого** `character_role`. Порог — **наличие карточки персонажа в волте**, а не его нарративный вес: раз персонаж удостоился карточки, он отражается на карточке тайтла (симметрично тому, как `featured_in` ведёт от персонажа к тайтлу). Решение «заводить ли карточку эпизоднику» принимается раньше и отдельно — но появившаяся карточка обязывает к обратной ссылке. Forward-fill — `/new-anime` шаг 2 (Glob `CHARACTERS/`); reverse — `/new-character` шаг 7.5a (для всех ролей).

**Взаимность с manga (то же правило, что с anime).** `featured_in[]` может содержать и `MANGA/`-карточки (персонаж появляется в манге-первоисточнике). Симметрия та же: манга ∈ `featured_in(C)` ⟺ `C` в `## Персонажи` манги. Forward-направление (создаётся anime/character после манги) держит backstop — `vault_asymmetric_links` / `/fix-links asymmetric` / `/audit-note` по парам `[manga, character]`+`[character, manga]`; отдельных reverse-ног в `/new-anime`//new-character` для манги нет (осознанный per-vault выбор).

**Взаимность character↔character — ЖЕЛЕЗНАЯ.** Связь в `## Связи` (`### В каноне`) строго двунаправленна: если карточка A линкует B, то B **обязана** линковать A — односторонних быть не должно, даже если у B упоминания A не было вовсе. Это переопределяет общий принцип «ссылка не уместна, когда связь тривиальна»: для персонажных связей решает не вес связи, а факт её объявления. Дисциплина смещается вверх — в `## Связи` класть только связи, которые стоит делать взаимными, потому что каждая обязана размножиться в цель. Enforcement (три слоя поверх `vault_asymmetric_links`): create-time — `/new-character` §7.5d (обе стороны + само-проверка); quality-gate — `/audit-note` ([[Audit_checklist]] §3.3); vault-wide sweep — `/fix-links asymmetric`. Scope политики — `reciprocity_pairs` в `.claude/vault-manifest.yaml`.

**Поля frontmatter ↔ WikiLinks в теле.** `featured_in: ["Demon_Slayer"]` в frontmatter — это plain string foreign-key; саму WikiLink `[[Demon_Slayer]]` обязательно упомянуть в теле (обычно в lead-параграфе или в секции «## Биография»). `affiliations` остаются plain strings (фракции редко имеют свои карточки); если в будущем появится `note_kind: faction` — конвертировать в wikilinks.

**Минимум исходящих.** Character без `featured_in[]` — это stub. Карточка персонажа без привязки к тайтлу не имеет смысла.

### note_kind: studio

Карточка студии **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| `parent_company` (если есть в `STUDIOS/`) | Граф «холдинг → дочерние студии»; обратная навигация |
| `successor` (для `status: defunct`, если есть в `STUDIOS/`) | Граф преемственности (Topcraft → Studio Ghibli); навигация от закрытой студии к продолжателю |
| Каждый из `founders[]` (если есть в `PERSONS/`) | Граф «основатель → студии»; обратная навигация. В frontmatter — plain strings; WikiLink ставится только в теле в gallery-секции «## Основатели и ключевые люди». |

**Backlinks из ANIME через поле `anime.studio`.** Это foreign-key из карточек тайтлов; в самой карточке студии материализуется gallery-секция «## Ключевые работы» (рендерится `/new-studio` при создании, поддерживается `/new-anime` cross-update'ом при добавлении нового тайтла). Per vault gallery-convention — этот единственный канал WikiLink на тайтл со стороны студии.

**Минимум исходящих.** Studio без `## Ключевые работы` — допустимый случай (только что созданная карточка для студии, у которой в волте ещё нет ни одного тайтла). По мере появления тайтлов через `/new-anime` секция заполняется автоматически.

### note_kind: manga

Карточка манги **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в волте):

| Ссылка | Зачем |
| --- | --- |
| Каждый `authors[].person` (`PERSONS/`) | Граф «автор → произведения»; мангака/сценарист/художник |
| Каждый `anime_adaptation[]` (`ANIME/`) | Манга → её аниме-экранизация; обратная навигация |
| Каждый `related_titles[]` (`MANGA/`) | Связи внутри франшизы манги: сиквел/приквел/спинофф |
| Персонажи в секции `## Персонажи` (`CHARACTERS/`) | Навигация к карточкам персонажей |

**WikiLinks на персонажей — только в секции `## Персонажи`** (single-channel, как у `anime`). В синопсисе/сюжете имена персонажей — plain text.

**Секция `## Персонажи` — gallery-список, НЕ 2-колоночная таблица.** В отличие от `anime` (там `Персонаж | Сэйю`), у манги нет озвучки — поэтому это просто gallery-bullet-список персонажей (миниатюра 60 + WikiLink на `CHARACTERS/`) по конвенции «Gallery-стиль для backlink-списков».

**Взаимность manga↔character — ЖЕЛЕЗНАЯ (ERROR), симметрично anime↔character.** Персонаж с карточкой, появляющийся в этой манге → обязан быть в `## Персонажи` манги, а манга — в его `featured_in[]` + `## Тайтлы`. Порог — наличие карточки персонажа. Пары `[manga, character]` + `[character, manga]` в `reciprocity_pairs`.

**Взаимность manga↔anime (адаптация, ERROR).** `manga.anime_adaptation` ↔ `anime.manga_source`: манга ссылается на экранизацию — тайтл обязан ссылаться обратно на мангу-первоисточник. Пары `[manga, anime]` + `[anime, manga]`.

**Взаимность manga↔person (автор, ERROR), зеркало `anime.staff[]↔person.works[]`.** `manga.authors[].person` ↔ `person.works[]` (MANGA-тайтл, роль `mangaka`/`novelist`): автор с карточкой в `PERSONS/` обязан иметь обратную запись в `works[]` + строку в `## Ключевые работы` (наравне с аниме-работами). Create-time покрывает `/new-manga` (reverse-нога автора); forward (person создан позже манги) — backstop. Пары `[manga, person]` + `[person, manga]`.

**Поля frontmatter ↔ WikiLinks в теле.** `authors`, `anime_adaptation`, `related_titles` — plain-string FK во frontmatter; сами WikiLinks обязательны в теле (gallery-секции `## Автор и публикация` / `## Связанные тайтлы`).

**Минимум исходящих.** Manga без `authors` и без связей — кандидат на stub.

## Gallery-стиль для backlink-списков

**Конвенция волта.** Везде, где в теле заметки строится **список ссылок на другие заметки** (галерея персонажей тайтла, список тайтлов персонажа, нарративные связи, ключевые работы персоны и т.п.), запись должна следовать единому паттерну:

```
- ![[<target>_cover.<ext>|<W>]] **[[<target>|<отображаемое имя>]]** — <одна строка описания>.
```

**Три случая, в порядке убывания полноты:**

| Состояние цели | Формат |
| --- | --- |
| Карточка есть **и** `images.cover` заполнен | Миниатюра + bold-WikiLink + описание |
| Карточка есть, **без** `images.cover` | bold-WikiLink + описание (без миниатюры) |
| Карточки **нет** | bold-имя + описание (без ссылки и миниатюры) |

**Ширина миниатюры:**
- `60` для карточек персонажей (компактно — таких записей в списке обычно 5–15).
- `80` для карточек тайтлов (чуть крупнее — обычно 1–5 записей, читателю важнее различить обложки).
- `60` по умолчанию для прочего (студии, персоны, локации) — оптимизировать только если возникнут ситуации, где это не работает.

**Ширина embed собственной обложки/постера карточки (hero-картинка под заголовком) — потолок `300`.** Это НЕ галерейная миниатюра, а главное изображение карточки: `![[<File>_cover.<ext>|300]]`. Кэп `≤300` обязателен (без `|N` Obsidian рендерит в полный размер исходника — постер на пол-экрана). Все create-скиллы и `/add-images` ставят `|300` при вставке cover-embed; меньше — допустимо, больше `300` — нет.

**Где применяется:**

| Скилл / секция | Цель |
| --- | --- |
| `/new-character` → `## Тайтлы` | `ANIME/*`, `MANGA/*` |
| `/new-character` → `## Связи → ### В каноне` | `CHARACTERS/*` |
| `/new-character` → `## Связи → ### За кулисами` | `PERSONS/*` |
| `/new-anime` → `## Персонажи` (2-колоночная table-format gallery; см. отдельную секцию выше) | `CHARACTERS/*`, `PERSONS/*` |
| `/new-anime` → `## Связанные тайтлы` | `ANIME/*` |
| `/new-anime` → `## Студия и команда` | `PERSONS/*`, `STUDIOS/*` |
| `/new-character` → `## Сэйю` (forward-check существующих VA-карточек; reverse-апгрейд `/new-person` шаг 7.5b) | `PERSONS/*` (voice-actor) |
| `/new-person` → `## Ключевые работы` | `ANIME/*` |
| `/new-person` → `## Студии` | `STUDIOS/*` |
| `/new-studio` → `## Ключевые работы` | `ANIME/*` |
| `/new-studio` → `## Основатели и ключевые люди` | `PERSONS/*` |

**Почему не frontmatter.** Можно было бы построить эти галереи через Dataview-запросы по `featured_in` / `staff[]` / `related_titles[]` — но это привязка к плагину, ломает рендеринг в plain Markdown и в exported HTML. Решение: материализуем gallery в теле, поддерживаем синхронность через cross-update в create-скиллах (`/new-character` шаг 7.5b).

**Принцип идемпотентности.** Если в галерее уже стоит запись с миниатюрой, и обновлять не нужно — Edit не делать.

**Когда у цели появляется/меняется `images.cover` — backfill миниатюр.** За это отвечает тот скилл, который меняет cover (`/add-images`, шаг 8.5). Алгоритм:

1. После установки нового `images.cover` для заметки X — найти все WikiLinks на X через `vault_backlinks(note="X")`.
2. Отфильтровать gallery-style записи: паттерн `- **[[X|...]]**` (с любым отступом, опциональной миниатюрой слева). Если миниатюра уже стоит — пропустить (идемпотентность).
3. Для каждой оставшейся записи вставить `![[<X_cover>|<W>]] ` перед `**[[X|...]]**`. Ширина определяется `note_kind` **target'а** X (см. таблицу выше — `anime` → 80, прочее → 60). Source-тип не влияет: миниатюра одинакова во всех заметках, ссылающихся на X.
4. Обновить `updated` в затронутых заметках.

Это переносит ответственность с `/audit-note` на момент изменения cover'а — самый локальный и атомарный сигнал «нужно прокинуть миниатюру». Create-скиллы (`/new-character`, `/new-anime`, `/new-person`, `/new-studio`) этим **не занимаются**: на момент создания исходящих ссылок они смотрят на текущее состояние cover'ов цели. Backfill — ретроактивная починка, инициируемая только сменой cover'а.

`/audit-note` остаётся как fallback: если по какой-то причине backfill не сработал (исторические заметки, ручные правки cover'а без скилла) — он находит несовпадения и предлагает поправить.

**Gallery-секция — единственный канал WikiLink на цель.** Если карточка содержит gallery-секцию, ссылающуюся на target X, то любые **другие** упоминания X в этой же карточке (lead, синопсис, биография) остаются **plain text**. Это переопределяет общее правило «первое смысловое упоминание — линк»: для gallery-режима «единственное навигационное место — галерея». Симметрично правилу «WikiLinks на персонажей — только в секции персонажей» для карточек тайтлов.

## Синтаксис

| Ситуация | Формат |
| --- | --- |
| Имя файла = отображаемый текст | `[[Note_Name]]` |
| Нужен другой текст | `[[Note_Name\|отображаемый текст]]` |
| В таблице (экранирование pipe) | `[[Note_Name\|отображаемый текст]]` |

## Антипаттерны

- **Дубли**: одна и та же ссылка дважды на странице.
- **Линковка заголовков**: WikiLink внутри `## Заголовок`.
- **Спам-линковка**: превращать каждое существительное в ссылку.
- **Ссылки на пустые файлы**: сначала заполнить stub — потом линковать.
- **Циклы ради циклов**: A→B→C→A допустимы, только если каждая связь осмысленна.
