# /new-character — Создание карточки персонажа

Создаёт карточку вымышленного персонажа аниме/манги с заполнением всех обязательных полей frontmatter согласно `note_kind: character` и каркасом тела заметки.

## Аргумент

`$ARGUMENTS` — имя персонажа на русском, английском, ромадзи или в оригинальной письменности.

Примеры:
- `/new-character "Kamado Tanjiro"`
- `/new-character Танджиро`
- `/new-character Nausicaa`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек Персонажей (`note_kind: character`)». Полный список обязательных и опциональных полей с семантикой enum.
2. **`SYSTEM/enums.yaml`** — секция `note_kinds.character`. Машинно-читаемые множества значений для `character_role`, `gender`, `status`.
3. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: character`. Обязательные WikiLinks и взаимность с `anime`.
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Найти файл через Glob в `CHARACTERS/`. Если карточка уже существует:
- **Stub / тонкая (<20 строк контента)** — не создавать с нуля, а дополнить. Сохранить существующие данные.
- **Наполненная (>50 строк)** — сообщить пользователю. Спросить, что обновить.

### 2. Сбор данных

Найти актуальные данные через web search. Приоритет источников:
- **MyAnimeList** (`myanimelist.net/character/<id>`) — каноничные имена ромадзи/native, age, japanese voice actor, **URL портрета** (`cdn.myanimelist.net/images/characters/...`).
- **AniList** (`anilist.co/character/<id>`) — gender, dateOfBirth, описание, **URL портрета** (Character.image.large из GraphQL).
- **Fandom-вики тайтла** — биография, способности, affiliations, родственные/нарративные связи, **URL дополнительных изображений** если нужны.
- Wikipedia как fallback для известных персонажей.

**Численные/перечисляемые факты — через WebFetch конкретной страницы**, не из snippet'а WebSearch (см. `feedback_numeric_facts_from_fetch`).

Собрать данные для всех обязательных полей frontmatter (шаг 4) + вспомогательные для тела (биография, способности, сэйю) + **URL постера** (один, наиболее качественный — для шага 6.5).

### 3. Определить имя файла

Формат: `PascalCase_With_Underscores.md`, только латиница. Порядок частей имени — как в каноне источника:
- Японские персонажи — фамилия→имя в ромадзи: `Kamado_Tanjiro.md`, `Higashikata_Josuke.md`.
- Западные/моноимённые — как в каноне: `Nausicaa.md`, `Edward_Elric.md`.

Кириллические/иероглифические варианты — в `aliases`. Имя файла = значение `name_romaji` с заменой пробелов на `_`.

### 4. Собрать frontmatter

Заполнить ВСЕ обязательные поля для `note_kind: character`:

```yaml
---
type: reference
domain: anime
stability: stable
priority: medium
quality: draft
note_kind: character
tags:
  - <тематические теги по Tag_taxonomy>
aliases:
  - <русская локализация>
  - <альтернативные транслитерации>
created: <сегодня в YYYY-MM-DD>
updated: <сегодня в YYYY-MM-DD>
name_romaji: "<Фамилия Имя или просто Имя>"
name_native: "<кандзи/кана/латиница в зависимости от происхождения>"
featured_in:
  - "<Anime_Card_Filename>"
character_role: <protagonist | deuteragonist | antagonist | supporting | minor>
gender: <male | female | non-binary | other | unknown>
status: <alive | deceased | unknown>
# Рекомендуемые:
name_english: "<English name>"
age: "<число, диапазон или unknown>"
species: "<human / demon / elf / ...>"
affiliations:
  - "<фракция, организация, семья>"
personal_score: <1-10>
mal_url: "<https://myanimelist.net/character/...>"
anilist_url: "<https://anilist.co/character/...>"
images:
  cover: "attachments/<File_Name>_cover.<ext>"
---
```

**Теги подбирать пошагово** (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня (если есть)
2. ДОМЕН (обычно совпадает с `domain: anime`, тег не нужен)
3. ТЕМЫ содержания — что делает персонажа интересным (`tragic-hero`, `anti-hero`, `mentor-figure`, `coming-of-age`)
4. ОБЯЗАТЕЛЬНЫЕ по `note_kind: character` — нет (всё закрывается полями frontmatter)
5. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с `character_role`/`gender`/`species`/`status`
6. **ПРОБЕЛЫ в таксономии — ОБЯЗАТЕЛЬНО** добавить новый тег в `SYSTEM/Tag_taxonomy.md` если используешь тег, которого там ещё нет.

**Принцип:** теги — про **дополнительное** измерение, которого нет во frontmatter. `male`, `protagonist`, `human`, `alive` — это enum-поля, в теги не дублируем.

**НЕ ставить `co_authored` при создании.** Поле проставляется только через `/verify` или `/audit-note`, когда пользователь явно подтверждает контентный пас модели.

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <Name_Romaji> (<Native>)

![[<File_Name>_cover.<ext>|300]]

**<Имя на русском или ромадзи>** — <1-2 предложения: кто это, в каком тайтле, ключевая черта/роль>.

## Описание

<Внешность, личность, ключевая черта характера. 3-6 строк. Без сюжетных спойлеров — это «как персонаж выглядит / каким воспринимается».>

## Биография

> [!warning]- Спойлеры
>
> <Бэкграунд (предыстория) + ключевая арка в каноне. Если персонаж появляется в нескольких тайтлах — структурировать по тайтлам/аркам.>

## Способности

<Только если боевой/фэнтези/сай-фай — техники, дары, оружие, ограничения. Иначе секцию опустить.>

## Тайтлы

<Все произведения из `featured_in[]`. Формат — gallery-стиль (см. конвенцию в `Linking_guidelines.md`): миниатюра постера тайтла + WikiLink + одна строка с годом/жанром.>

```
- ![[<Anime_File>_cover.<ext>|80]] **[[<Anime_File>|<Название>]]** — <год>, <одна строка: жанр/завязка>.
```

<Если у карточки тайтла нет `images.cover` — миниатюру опустить, оставить только WikiLink. Если в `featured_in[]` только один тайтл — секция всё равно полезна как стандартизованный entry point.>

## Связи

<Разделить на две подсекции: нарративные связи внутри канона (другие персонажи) и внепроизведенческие (создатели, прототипы, реальные люди). Не мешать.>

### В каноне

<Семья, союзники, антагонисты, компаньоны — другие персонажи из произведения. Для каждого:
- Если карточка `CHARACTERS/<имя>.md` существует **и имеет `images.cover`** — gallery-стиль: `- ![[<X>_cover.<ext>|60]] **[[<X>|<Имя>]]** — <описание>.`
- Если карточка существует, но без постера — только WikiLink: `- **[[<X>|<Имя>]]** — <описание>.`
- Если карточки нет — plain bold: `- **<Имя>** — <описание>.`>

### За кулисами

<Создатели и реальные люди, значимо связанные с персонажем: режиссёр-автор образа, character designer, прототип/вдохновение, известный сэйю если связь именно для образа. Тот же gallery-формат, цели — `PERSONS/`:
- `- ![[<Person>_cover.<ext>|60]] **[[<Person>|<Имя>]]** — <роль и почему важно для образа>.`>

<Секцию `### За кулисами` опустить, если таких связей нет. Sole-`### В каноне` подзаголовок тогда тоже можно убрать и оставить плоский список под `## Связи`.>

## Сэйю

- **Japanese:** <имя сэйю>
- **English dub:** <имя актёра дубляжа>  <!-- опционально -->
- **Russian dub:** <имя актёра дубляжа>  <!-- опционально -->

## Личный отзыв

<Опционально — почему персонаж зашёл/не зашёл, чем зацепил. Если нет — секцию пропустить.>

## Ссылки

- [MyAnimeList](<mal_url>)
- [AniList](<anilist_url>)
- [Fandom](<url>)  <!-- опционально, если есть качественная вики -->
```

Размер заметки: 60–150 строк. Лучше короче и плотнее, чем длиннее и водянистее.

### 6. Проверить ссылки и собрать gallery-формат

Обязательные WikiLinks (по `### note_kind: character` в `Linking_guidelines.md`):

- **Каждый `featured_in[]`** — WikiLink на карточку аниме идёт **только в секцию `## Тайтлы`** (gallery-формат). В lead-параграфе тайтл упоминается **plain text**. Это применение vault-конвенции «gallery-секция — единственный канал WikiLink» (см. `Linking_guidelines.md`). Если карточки аниме нет в волте — секцию «## Тайтлы» не создавать, тайтл упомянуть только в lead как plain text.
- **Упоминания других персонажей** в «## Связи → ### В каноне» — WikiLinks на `CHARACTERS/<имя>.md` если карточки существуют.
- **Производственные персоны** в «## Связи → ### За кулисами» — WikiLinks на `PERSONS/<имя>.md`.

**Gallery-формат для всех WikiLink-записей в `## Тайтлы` и `## Связи`** (vault-wide конвенция, см. `Linking_guidelines.md` секция «Gallery-стиль для backlink-списков»):

Для каждого WikiLink — прочитать frontmatter цели (Read + парс YAML), посмотреть `images.cover`:

- **`images.cover` есть** → префиксировать запись миниатюрой: `- ![[<cover_filename>|<W>]] **[[<target>|<display>]]** — <описание>.` Ширина: 60 для персонажей, 80 для тайтлов.
- **`images.cover` нет** → только WikiLink: `- **[[<target>|<display>]]** — <описание>.`
- **Карточки цели не существует** → plain bold без ссылки: `- **<имя>** — <описание>.`

Оптимизация: если в волте мало карточек и почти у всех есть постеры, можно сразу строить gallery-формат и Edit'ить если что-то не подошло. Но дешевле всё-таки Read'нуть каждую цель один раз — это <5 запросов на типовую карточку.

Проверки:
- Нет дубликатов ссылок (одна цель — одна WikiLink на заметку, первое смысловое упоминание).
- Все target существуют — проверить через Glob (или vault-query).
- Все embed'ы миниатюр резолвятся в реальные файлы в `attachments/`.
- Если в `featured_in` нет ни одной существующей карточки аниме — предупредить пользователя: «карточка персонажа без `featured_in` будет stub'ом», предложить сначала создать карточку тайтла через `/new-anime`.

### 6.5. Скачать постер автоматически

**Правило:** карточка персонажа создаётся **с постером всегда, когда это возможно**. Это критично — без постера карточка не попадёт в галерею персонажей тайтла (см. шаг 7.5) и весь смысл reverse-update'а теряется.

**Алгоритм:**

1. **Взять URL** из шага 2 (приоритет: AniList Character.image.large > MAL character page > Fandom infobox).
2. **Определить расширение** — из URL (`.jpg` / `.jpeg` / `.png` / `.webp`). Если URL без расширения — взять из `Content-Type` после первого запроса.
3. **Скачать** в `attachments/<File_Name>_cover.<ext>` через PowerShell:
   ```powershell
   Invoke-WebRequest -Uri "<URL>" -OutFile "attachments/<File_Name>_cover.<ext>" -UserAgent "Mozilla/5.0"
   ```
   `-UserAgent` обязателен — некоторые CDN (особенно MAL) блокируют запросы без него.
4. **Проверить файл** — размер > 0, расширение совпадает с заявленным. Если фейл — попробовать следующий источник из приоритета.
5. **Прописать в frontmatter:**
   ```yaml
   images:
     cover: "attachments/<File_Name>_cover.<ext>"
   ```
6. **Вставить embed** в теле под заголовком: `![[<File_Name>_cover.<ext>|300]]`.

**Если ни один источник не дал годной картинки:**
- Опустить поле `images` и embed-строку.
- В финальном отчёте (шаг 9) явно сказать: «постер не найден — карточка не появится в галерее тайтла до ручного добавления через `/add-images`».
- **Не пытаться** генерировать заглушки или скачивать случайные картинки.

**Что НЕ скачивать:**
- Иконки/фавиконы MAL/AniList (часто попадают в HTML вместе с постером).
- Изображения < 100×100 px (вероятно превью галереи, не основной портрет).
- Изображения с water-mark на половину площади — лучше без постера, чем с мусором.

### 7. Создать файл

Записать в `CHARACTERS/<File_Name>.md`.

### 7.5. Reverse-check: упомянуть персонажа в карточках `featured_in`

**Принцип волта.** WikiLink на персонажа существует ровно в одном месте каждой карточки тайтла — в секции персонажей (типичные заголовки: `## Персонажи`, `## Главные персонажи`, `## Основные персонажи`). Plain-text упоминания персонажа в других секциях карточки тайтла (синопсис, биография, и т.п.) WikiLinks **не превращаем** — это намеренно, чтобы избежать дублей и шума.

Из этого принципа — **не делаем** vault-wide поиск упоминаний по всему волту. Проверяем ровно одну секцию в каждой `featured_in` карточке.

**Алгоритм для каждого `featured_in[i]` (если карточка `ANIME/<i>.md` существует в волте):**

1. **Прочитать карточку** через Read.
2. **Найти секцию персонажей.** Сканировать заголовки уровня `## `, искать (в этом порядке): `## Персонажи`, `## Главные персонажи`, `## Основные персонажи`. Брать первое совпадение.
3. **Если секции нет** — пропустить эту карточку с записью в финальный отчёт: «`ANIME/X.md`: секции персонажей нет, добавление пропущено». **Не создавать секцию** — это решение автора карточки тайтла, не подразумеваемый инвариант.
4. **Через Grep проверить состояние записи в этой секции.** Возможные исходы:
   - **a) Уже WikiLink** на эту карточку (`[[<File_Name>]]` или `[[<File_Name>|...]]`) — ничего не делать, в отчёт: «`ANIME/X.md`: уже есть, ОК». Пропустить остальные шаги.
   - **b) Plain-text упоминание** имени персонажа (одного из значений `name_romaji`, `name_english`, `aliases[]`) внутри секции — **это типовой случай**, поскольку `/new-anime` заполняет секцию текстовым списком всех основных персонажей при создании карточки тайтла. Перейти к шагу 5b (in-place upgrade).
   - **c) Ни WikiLink, ни упоминания** — перейти к шагу 5c (добавить новую запись).
5. **Что делать в зависимости от исхода шага 4:**

   **5b. In-place upgrade (text → WikiLink + миниатюра):**
   - Найти точную bullet-строку с именем персонажа. Заменить голое имя на bold-WikiLink, **сохранив описание и всё прочее без изменений**.
     - Было: `- **Танджиро** — главный герой, охотник на демонов.`
     - Стало: `- ![[Kamado_Tanjiro_cover.jpg\|60]] **[[Kamado_Tanjiro\|Танджиро]]** — главный герой, охотник на демонов.`
     - Если описания не было: `- Танджиро` → `- ![[Kamado_Tanjiro_cover.jpg\|60]] **[[Kamado_Tanjiro\|Танджиро]]**`.
   - Миниатюра `![[..._cover.jpg|60]]` добавляется только если постер успешно скачан в шаге 6.5. Если постера нет — добавить только WikiLink, без миниатюры.
   - Делать без явного prompt'а — это апгрейд существующей записи, не вторжение в чужой контент.

   **5c. Решить, добавлять ли новую запись:**
   - `character_role` ∈ {`protagonist`, `deuteragonist`, `antagonist`} → добавлять по умолчанию (показать «добавляю в `ANIME/X.md`», без prompt-ожидания).
   - `character_role` ∈ {`supporting`, `minor`} → через AskUserQuestion спросить: «`ANIME/X.md`: добавить упоминание `<Имя>` (роль: `<role>`) в секцию персонажей?», дефолт **«нет»**.

   Если решили добавлять — **отзеркалить существующий формат секции.** Прочитать одну-две уже существующие записи, повторить их стиль:
   - Если bullet с bold-WikiLink + тире + описание:
     ```markdown
     - ![[<File_Name>_cover.<ext>\|60]] **[[<File_Name>\|<Имя>]]** — <одна строка описания>.
     ```
   - Если другой формат — повторить его. **Не навязывать свой стиль** карточке тайтла.
   - Сортировка внутри секции: главные роли (`protagonist` → `deuteragonist` → `antagonist`) идут вначале, `supporting` / `minor` — после. Если порядок в существующей секции уже нарушен пользователем — не переупорядочивать, добавлять в конец соответствующей подгруппы (или в конец списка, если подгруппировки не видно).

6. **Обновить `updated`** в frontmatter карточки тайтла на сегодня (только если карточка реально менялась).

**Не делать:**
- Не вставлять упоминания в другие секции тайтла (синопсис, сюжет, биография) — даже если там персонаж очевидно фигурирует.
- Не превращать plain-text упоминания персонажа в других карточках волта в WikiLinks.
- Не переписывать существующие записи в секции персонажей (опечатки, форматирование — это задача `/audit-note`, не /new-character).

### 8. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** для **всех изменённых заметок** (созданная карточка персонажа + опционально обновлённая карточка аниме из шага 7.5). Если найдены ошибки — исправить. Если предупреждения — упомянуть в отчёте.

### 9. Сообщить результат

Показать: имя файла, путь, основное имя, число строк, ключевые WikiLinks, обновлённые связанные заметки (если были), результат аудита. Предложить `/verify` если устраивает.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **НЕ ставить `co_authored` при создании.** Только через `/verify` или `/audit-note`.
- Web research — обязательная часть фазы 2. Не выдумывать факты, не додумывать данные. Численные/перечисляемые факты (`age`, `mal_url`-id) — через WebFetch конкретной страницы.
- **Постер скачивать всегда, когда возможно.** Это часть контракта скилла: галерея персонажей в карточке тайтла заполняется через шаг 7.5b, и без постера апгрейд text → миниатюра+WikiLink становится text → WikiLink (без миниатюры — дыра в галерее).
