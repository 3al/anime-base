# /new-character — Создание карточки персонажа

Создаёт карточку вымышленного персонажа аниме/манги с заполнением всех обязательных полей frontmatter согласно `note_kind: character` и каркасом тела заметки.

## Аргумент

`$ARGUMENTS` — имя персонажа на русском, английском, ромадзи или в оригинальной письменности.

Флаг `--no-images` / `-n` (опционально) — пропустить всю работу с постером (шаг 6.5): карточка создаётся без изображения. Осознанный обход vision-gate для не-мультимодальной модели.

Примеры:
- `/new-character "Kamado Tanjiro"`
- `/new-character Танджиро`
- `/new-character Nausicaa --no-images`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек Персонажей (`note_kind: character`)». Полный список обязательных и опциональных полей с семантикой enum.
2. **`SYSTEM/enums.yaml`** — секция `note_kinds.character`. Машинно-читаемые множества значений для `character_role`, `gender`, `status`.
3. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: character`. Обязательные WikiLinks и взаимность с `anime`.
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Найти файл через Glob в `CHARACTERS/`. Если карточка уже существует:
- **Stub / тонкая (<20 строк контента)** — не создавать с нуля, а дополнить. Сохранить существующие данные.
- **Наполненная (>50 строк)** — сообщить пользователю. Спросить, что обновить.

### 2. Сбор данных

Найти актуальные данные через web search. Приоритет источников:
- **MyAnimeList** (`myanimelist.net/character/<id>`) — каноничные имена ромадзи/native, age, japanese voice actor, **URL портрета** (`cdn.myanimelist.net/images/characters/...`).
- **AniList** (`anilist.co/character/<id>`) — gender, dateOfBirth, описание, **URL портрета** (Character.image.large из GraphQL).
- **Fandom-вики тайтла** — биография, способности, affiliations, родственные/нарративные связи, **URL дополнительных изображений** если нужны.
- Wikipedia как fallback для известных персонажей.

**Численные/перечисляемые факты — через web fetch конкретной страницы**, не из snippet'а web search (см. `feedback_numeric_facts_from_fetch`).

Собрать данные для всех обязательных полей frontmatter (шаг 4) + вспомогательные для тела (биография, способности, сэйю) + **URL постера** (один, наиболее качественный — для шага 6.5).

**Сэйю — обязательно собрать per-anime.** На MAL у каждого персонажа есть блок «Voice Actors» (japanese + английский dub, если есть). На странице характера блок группируется по тайтлам — для каждого `featured_in[i]` извлечь имя японского сэйю. Это нужно для `voice_actors[]` (шаг 4) и forward-check'а на `PERSONS/` (шаг 4): если у уже-существующего сэйю есть карточка в `PERSONS/<va>.md`, запись в `voice_actors[]` собирается с этим foreign-key; если карточки нет — на этом этапе запись не создаётся, имя сэйю остаётся в plain-text в `## Сэйю` секции (апгрейдит `/new-person` при создании карточки VA — см. его шаг 7.5b).

### 3. Определить имя файла

Формат: `PascalCase_With_Underscores.md`, только латиница. Порядок частей имени — как в каноне источника:
- Японские персонажи — фамилия→имя в ромадзи: `Kamado_Tanjiro.md`, `Higashikata_Josuke.md`.
- Западные/моноимённые — как в каноне: `Nausicaa.md`, `Edward_Elric.md`.

Кириллические/иероглифические варианты — в `aliases`. Имя файла = значение `name_romaji` с заменой пробелов на `_`.

### 4. Собрать frontmatter

Заполнить ВСЕ обязательные поля для `note_kind: character`:

```yaml
---
type: reference
domain: anime
stability: stable
priority: medium
quality: draft
note_kind: character
tags:
  - <тематические теги по Tag_taxonomy>
aliases:
  - <русская локализация>
  - <альтернативные транслитерации>
created: <сегодня в YYYY-MM-DD>
updated: <сегодня в YYYY-MM-DD>
name_romaji: "<Фамилия Имя или просто Имя>"
name_native: "<кандзи/кана/латиница в зависимости от происхождения>"
featured_in:
  - "<Anime_Card_Filename>"
character_role: <protagonist | deuteragonist | antagonist | supporting | minor>
gender: <male | female | non-binary | other | unknown>
status: <alive | deceased | unknown>
# Рекомендуемые:
name_english: "<English name>"
age: "<число, диапазон или unknown>"
species: "<human / demon / elf / ...>"
affiliations:
  - "<фракция, организация, семья>"
personal_score: <1-10>
# voice_actors[] — список сэйю по тайтлам. Forward-check: каждая запись включается только
# если для VA существует карточка PERSONS/<va>.md. Сэйю без карточки в волте — на этом
# шаге не записываем; их апгрейдит /new-person при создании VA-карточки (его шаг 7.5b).
# Если у персонажа нет известных VA (манга-only до экранизации) — оставить [].
voice_actors:
  - anime: "<Anime_Card_Filename>"
    person: "<VA_Card_Filename>"
mal_url: "<https://myanimelist.net/character/...>"
anilist_url: "<https://anilist.co/character/...>"
images:
  cover: "attachments/<File_Name>_cover.<ext>"
---
```

**Теги подбирать пошагово** (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня (если есть)
2. ДОМЕН (обычно совпадает с `domain: anime`, тег не нужен)
3. ТЕМЫ содержания — что делает персонажа интересным (`tragic-hero`, `anti-hero`, `mentor-figure`, `coming-of-age`)
4. ОБЯЗАТЕЛЬНЫЕ по `note_kind: character` — нет (всё закрывается полями frontmatter)
5. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с `character_role`/`gender`/`species`/`status`
6. **ПРОБЕЛЫ в таксономии — ОБЯЗАТЕЛЬНО** добавить новый тег в `SYSTEM/Tag_taxonomy.md` если используешь тег, которого там ещё нет.

**Принцип:** теги — про **дополнительное** измерение, которого нет во frontmatter. `male`, `protagonist`, `human`, `alive` — это enum-поля, в теги не дублируем.

**Ставить `co_authored: <model-id>`** при создании. Значение — ID текущей модели в волтовом формате `claude-<family>-<major>.<minor>` (например `claude-opus-4.7`, `claude-sonnet-4.6`, `claude-haiku-4.5`). Источник истины — системный промпт, строка «The exact model ID is …» (последний дефис → точка: `claude-opus-4-7` → `claude-opus-4.7`). См. `feedback_co_authored`.

### 5. Написать содержание

**Политика канона (vault-wide).** Карточка персонажа — **одна на персонажа**, объединяющая все каноничные носители (аниме, манга, ранобэ). Не плодить отдельные карточки «X-фильм» и «X-манга».

- **Приоритет — аниме-канону.** Если у тайтла есть аниме-адаптация, основное описание (лид, `## Описание`, `## Биография` основной callout, `## Способности` основной список) даётся **по аниме**.
- **Манга/ранобэ — в помеченных блоках.** Существенные расхождения (расширенная роль, изменённый бэкграунд, дополнительные арки/сцены, физические/семейные детали, отсутствующие в аниме) выносятся в подзаголовки `### В манге` / `### В ранобэ` внутри `## Биография`, либо в callout `> [!note] В манге` внутри `## Способности`. Без пометки = аниме-канон.
- **Сюжетные ревизии (любого носителя) — всегда под спойлер-callout.** Если `### В манге` / `### В ранобэ` блок раскрывает сюжет, концовку или ключевые повороты — содержимое идёт внутри `> [!warning]- Спойлеры (манга)` / `> [!warning]- Спойлеры (ранобэ)`, как и аниме-биография. Если различия чисто описательные (внешность, способности, фоновые факты без plot-ревизий) — спойлер-callout не нужен, достаточно `> [!note]` или плоского списка.
- **Если аниме нет** (только манга / только ранобэ) — основное описание даётся по этому носителю, помечать раздельные блоки не нужно.
- **Если есть и аниме, и манга, и они полностью совпадают** — `### В манге` блок опустить.

Это применяется и к `## Связи → ### В каноне`: внутри одной bullet-записи допустимо указать «в манге — A, в фильме — B», если связь различается между носителями. Не разносить персонажа на отдельные записи.

Каркас тела заметки:

```markdown
# <Name_Romaji> (<Native>)

![[<File_Name>_cover.<ext>|300]]

**<Имя на русском или ромадзи>** — <1-2 предложения: кто это, в каком тайтле, ключевая черта/роль>.

## Описание

<Внешность, личность, ключевая черта характера. 3-6 строк. Без сюжетных спойлеров — это «как персонаж выглядит / каким воспринимается». Канон — аниме (см. блок «Политика канона» ниже).>

## Биография

> [!warning]- Спойлеры (аниме)
>
> <Бэкграунд + ключевая арка **по аниме-канону**. Если персонаж появляется в нескольких тайтлах одной франшизы — структурировать по тайтлам/аркам, но в пределах аниме-носителей.>

### В манге

<Опционально. Заполнять, только если манга существенно расходится с аниме: расширенная роль, другие сюжетные арки, дополнительные сцены, изменённый бэкграунд, физические/семейные детали, отсутствующие в фильме/сериале. Bullet-список различий, без пересказа всего фильма.>

> [!warning]- Спойлеры (манга)
>
> <Если блок раскрывает сюжет манги — содержимое (bullet-список различий) идёт **внутри** этого callout'а, как и аниме-биография. Если различия чисто описательные (внешность, фоновые факты, не плот) — callout можно опустить и оставить плоский список.>

### В ранобэ

<Опционально. Так же, только для существенных расхождений с аниме-каноном. То же правило: сюжетные ревизии — внутрь `> [!warning]- Спойлеры (ранобэ)` callout'а.>

## Способности

<Только если боевой/фэнтези/сай-фай — техники, дары, оружие, ограничения. Иначе секцию опустить. По умолчанию — аниме-канон. Если в манге/ранобэ есть способности, отсутствующие в аниме, — добавить callout `> [!note] В манге` / `> [!note] В ранобэ` с этими дополнениями.>

## Тайтлы

<Все произведения из `featured_in[]`. Формат — gallery-стиль (см. конвенцию в `Linking_guidelines.md`): миниатюра постера тайтла + WikiLink + одна строка с годом/жанром.>

```
- ![[<Anime_File>_cover.<ext>|80]] **[[<Anime_File>|<Название>]]** — <год>, <одна строка: жанр/завязка>.
```

<Если у карточки тайтла нет `images.cover` — миниатюру опустить, оставить только WikiLink. Если в `featured_in[]` только один тайтл — секция всё равно полезна как стандартизованный entry point.>

## Связи

<Разделить на две подсекции: нарративные связи внутри канона (другие персонажи) и внепроизведенческие (создатели, прототипы, реальные люди). Не мешать.>

### В каноне

<Семья, союзники, антагонисты, компаньоны — другие персонажи из произведения. Для каждого:
- Если карточка `CHARACTERS/<имя>.md` существует **и имеет `images.cover`** — gallery-стиль: `- ![[<X>_cover.<ext>|60]] **[[<X>|<Имя>]]** — <описание>.`
- Если карточка существует, но без постера — только WikiLink: `- **[[<X>|<Имя>]]** — <описание>.`
- Если карточки нет — plain bold: `- **<Имя>** — <описание>.`>

### За кулисами

<Создатели и реальные люди, значимо связанные с персонажем: режиссёр-автор образа, character designer, прототип/вдохновение, известный сэйю если связь именно для образа. Тот же gallery-формат, цели — `PERSONS/`:
- `- ![[<Person>_cover.<ext>|60]] **[[<Person>|<Имя>]]** — <роль и почему важно для образа>.`>

<Секцию `### За кулисами` опустить, если таких связей нет. Sole-`### В каноне` подзаголовок тогда тоже можно убрать и оставить плоский список под `## Связи`.>

## Сэйю

<Список японских сэйю на основе `voice_actors[]` (frontmatter). Формат — gallery-стиль с двумя case'ами в зависимости от того, есть ли карточка VA в волте:

- **VA-карточка существует** (`PERSONS/<va>.md` найдена, запись попала в `voice_actors[]`) → gallery-запись (миниатюра 60 при наличии `images.cover` у persona; WikiLink иначе):
  - Если этот же VA озвучивает персонажа во всех `featured_in[]`:
    - `- ![[<VA>_cover.<ext>|60]] **Japanese (все тайтлы):** [[<VA>|<Имя>]].`
  - Если по тайтлам разные VA или озвучивает только часть:
    - `- ![[<VA>_cover.<ext>|60]] **Japanese ([[<Anime>|<Title>]]):** [[<VA>|<Имя>]].`
- **VA-карточки нет** → plain text (запись не входит в `voice_actors[]`, но имя в теле упоминается):
  - `- **Japanese ([[<Anime>|<Title>]]):** <Имя сэйю>` — будущий `/new-person` для этого VA апгрейдит запись через свой шаг 7.5b.

Опционально:
- **English dub:** <имя актёра дубляжа>
- **Russian dub:** <имя актёра дубляжа>

Эти строки структурно не привязываются к `voice_actors[]` (там только japanese-каст, как на MAL/AniList).>

## Личный отзыв

<Опционально — почему персонаж зашёл/не зашёл, чем зацепил. Если нет — секцию пропустить.>

## Ссылки

- [MyAnimeList](<mal_url>)
- [AniList](<anilist_url>)
- [Fandom](<url>)  <!-- опционально, если есть качественная вики -->
```

Размер заметки: 60–150 строк. Лучше короче и плотнее, чем длиннее и водянистее.

### 6. Проверить ссылки и собрать gallery-формат

Обязательные WikiLinks (по `### note_kind: character` в `Linking_guidelines.md`):

- **Каждый `featured_in[]`** — WikiLink на карточку аниме идёт **только в секцию `## Тайтлы`** (gallery-формат). В lead-параграфе тайтл упоминается **plain text**. Это применение vault-конвенции «gallery-секция — единственный канал WikiLink» (см. `Linking_guidelines.md`). Если карточки аниме нет в волте — секцию «## Тайтлы» не создавать, тайтл упомянуть только в lead как plain text.
- **Упоминания других персонажей** в «## Связи → ### В каноне» — WikiLinks на `CHARACTERS/<имя>.md` если карточки существуют.
- **Производственные персоны** в «## Связи → ### За кулисами» — WikiLinks на `PERSONS/<имя>.md`.

**Gallery-формат для всех WikiLink-записей в `## Тайтлы` и `## Связи`** (vault-wide конвенция, см. `Linking_guidelines.md` секция «Gallery-стиль для backlink-списков»):

Для каждого WikiLink — прочитать frontmatter цели (Read + парс YAML), посмотреть `images.cover`:

- **`images.cover` есть** → префиксировать запись миниатюрой: `- ![[<cover_filename>|<W>]] **[[<target>|<display>]]** — <описание>.` Ширина: 60 для персонажей, 80 для тайтлов.
- **`images.cover` нет** → только WikiLink: `- **[[<target>|<display>]]** — <описание>.`
- **Карточки цели не существует** → plain bold без ссылки: `- **<имя>** — <описание>.`

Оптимизация: если в волте мало карточек и почти у всех есть постеры, можно сразу строить gallery-формат и Edit'ить если что-то не подошло. Но дешевле всё-таки Read'нуть каждую цель один раз — это <5 запросов на типовую карточку.

**Существование целей — детерминированно, не «по памяти авторинга».** Не полагаться на то, что при написании тела (шаг 5) ты «помнишь», у кого из упомянутых соседей есть карточка. Для **каждого** имени персонажа/персоны в bullet'ах `## Связи` явно проверить наличие карточки — Glob по `CHARACTERS/`/`PERSONS/` либо `vault_query(noteKind=character/person)` одним списком + матч имён против `aliases`/`name_romaji`. Если карточка существует — запись **обязана** быть gallery-WikiLink, а не plain-bold. Plain-bold допустим только когда карточки действительно нет в волте. Это **форвард-нога реципрокности** (см. 7.5d): пропуск здесь = вмороженная однонаправленная связь, которую `vault_text_mentions` потом уже не подсветит.

Проверки:
- Нет дубликатов ссылок (одна цель — одна WikiLink на заметку, первое смысловое упоминание).
- Все target существуют — проверить через Glob (или vault-query).
- Все embed'ы миниатюр резолвятся в реальные файлы в `attachments/`.
- Если в `featured_in` нет ни одной существующей карточки аниме — предупредить пользователя: «карточка персонажа без `featured_in` будет stub'ом», предложить сначала создать карточку тайтла через `/new-anime`.

### 6.5. Скачать и провалидировать постер (vision-gate)

**Правило:** карточка создаётся **с постером всегда, когда возможно** — без него она не попадёт в галерею персонажей тайтла (шаг 7.5), и смысл reverse-update'а теряется. **Но** подбор и валидация изображения требуют **реального восприятия картинки**: текстовая (не-мультимодальная) модель этого не умеет и, оценивая постер по имени файла / URL / подписи, тихо выдаёт брак. Поэтому блок проходит через **активный vision-gate**. Канонический источник правил для всех image-скиллов волта — `.claude/skills/add-note-kind/templates/image-handling.claude-fragment.md`; ниже — его применение к постеру персонажа.

**Флаг `--no-images` / `-n`** (если задан в аргументах) — весь блок 6.5 пропустить: постер не качать, `images`/embed опустить, пометить в отчёте, идти дальше. Gate не нужен.

**Алгоритм:**

1. **URL** из шага 2 (приоритет: AniList `Character.image.large` > MAL character page > Fandom infobox). Расширение — из URL (`.jpg`/`.jpeg`/`.png`/`.webp`) или из `Content-Type`.
2. **Скачать** в `attachments/<File_Name>_cover.<ext>`:
   ```powershell
   Invoke-WebRequest -Uri "<URL>" -OutFile "attachments/<File_Name>_cover.<ext>" -UserAgent "Mozilla/5.0"
   ```
   `-UserAgent` обязателен (MAL и часть CDN блокируют без него; Wikimedia требует descriptive UA — см. `Tools_reference`). *(Канон предпочитает `curl -L -f -A "…"` ради кросс-платформенности — выровнять при общем рефакторе image-скиллов.)*
3. **Провалидировать ДО любого `Read`** (битый файл отравляет контекст на всю сессию → `"Could not process image"` на всех последующих запросах):
   - **тип** — реальное изображение JPEG/PNG/GIF/WebP (по сигнатуре/магическим байтам или `file`). HTML/XML/ASCII/empty → удалить, взять следующий URL.
   - **размер** ≥ 1 КБ (мельче — почти наверняка не постер).
4. **Vision-gate — активный probe (не декларация).** Открыть валидный файл через `Read` и попытаться извлечь конкретный визуальный факт (цвет волос/глаз, поза, фон, характерная деталь персонажа).
   - **Воспринимаешь содержимое** → модель мультимодальна. Заодно подтвердить, что это **нужный** персонаж, а не фавикон / чужой кадр / watermark на пол-площади; мусор → удалить, próбовать следующий URL. → шаг 5.
   - **Не воспринимаешь** (Read без визуала / ошибка / только метаданные) → **СТОП на постере.** Оценивать картинку по имени/URL **запрещено** — это брак, не fallback. Удалить файл, `images`/embed **не** добавлять. В отчёт (шаг 9): «постер пропущен — модель не мультимодальна, визуальная проверка невозможна; добавить позже через `/add-images` на мультимодальной модели либо перезапустить с `--no-images` осознанно». **Текстовую часть карточки — frontmatter, тело, связи, reverse-check 7.5 — доделать полностью** (new-character не «целиком про фото»).
5. **Прописать** `images.cover: "attachments/<File_Name>_cover.<ext>"` во frontmatter и **вставить embed** под заголовком: `![[<File_Name>_cover.<ext>|300]]`.

**При `"Could not process image"` на шаге 4** — файл уже отравил контекст: повторно его **не** Read'ить, удалить, взять другой URL, заново провалидировать (шаг 3) перед probe.

**Если ни один источник не дал годной картинки** (или gate ушёл в non-multimodal-ветку): `images` и embed опустить, в отчёте явно сказать про отсутствие постера и путь через `/add-images`. Заглушки не генерировать, случайные картинки не качать.

### 7. Создать файл

Записать в `CHARACTERS/<File_Name>.md`.

### 7.5. Reverse-check: апгрейд plain-text упоминаний персонажа

**Принцип волта.** WikiLink на персонажа существует ровно в одном «навигационном» месте каждой карточки:
- В **карточке тайтла** — секция персонажей (`## Персонажи` / `## Главные персонажи` / `## Основные персонажи`). Plain-text упоминания персонажа в других секциях карточки тайтла (синопсис, биография) WikiLinks **не превращаем**.
- В **карточке другого персонажа** — секция `## Связи → ### В каноне` (или плоская `## Связи`). Это симметричный «навигационный» канал для нарративных связей.
- В **карточке персоны** — секция `## Ключевые работы` / аналог (редкий случай — character designer, прототип).

Этот шаг покрывает два потока:
- **(a) `featured_in[]` anime-карточки** — точечный Read+Edit (структура секции хорошо известна).
- **(b) sibling-карточки** (`CHARACTERS/*`, опционально `PERSONS/*`) — через `vault_text_mentions`, чтобы поймать text → WikiLink upgrade в их «связевых» секциях.

#### 7.5a. `featured_in[]` (anime-карточки)

**Алгоритм для каждого `featured_in[i]` (если карточка `ANIME/<i>.md` существует в волте):**

1. **Прочитать карточку** через Read.
2. **Найти секцию персонажей.** Сканировать заголовки уровня `## `, искать (в этом порядке): `## Персонажи`, `## Главные персонажи`, `## Основные персонажи`. Брать первое совпадение.
3. **Если секции нет** — пропустить эту карточку с записью в финальный отчёт: «`ANIME/X.md`: секции персонажей нет, добавление пропущено». **Не создавать секцию** — это решение автора карточки тайтла, не подразумеваемый инвариант.
4. **Определить формат секции.** Современная vault-конвенция — 2-колоночная таблица `| Персонаж | Сэйю |` (cast-list; см. `Linking_guidelines.md` → table-format gallery). Исторические карточки могут быть в bullet-list формате (до перехода на табличный layout). Алгоритм поддерживает оба:
   - Если первая непустая строка после заголовка `## Персонажи` — `| ... | ... |` (Markdown table header) → **табличный формат**.
   - Если bullet `- ...` → **bullet-формат** (legacy).
5. **Через Grep проверить состояние записи в этой секции.** Возможные исходы (одинаковые для обоих форматов):
   - **a) Уже WikiLink** на эту карточку (`[[<File_Name>]]` или `[[<File_Name>|...]]` / `[[<File_Name>\|...]]` для таблицы) — ничего не делать, в отчёт: «`ANIME/X.md`: уже есть, ОК». Пропустить остальные шаги.
   - **b) Plain-text упоминание** имени персонажа (одного из значений `name_romaji`, `name_english`, `aliases[]`) — **типовой случай**, `/new-anime` заполняет секцию plain-text для отсутствующих карточек. Перейти к шагу 6b (in-place upgrade ячейки/bullet).
   - **c) Ни WikiLink, ни упоминания** — перейти к шагу 6c (добавить новую запись).
6. **Что делать в зависимости от исхода шага 5:**

   **6b. In-place upgrade plain-text → WikiLink+миниатюра.**
   
   **Табличный формат (2-col `Персонаж | Сэйю`):**
   - Найти строку таблицы, где **первая ячейка** (col 1 Персонаж) содержит plain-text имя персонажа.
   - Заменить **только содержимое col 1** на gallery-формат с pipe-escaping. Col 2 (Сэйю) **не трогать** — она принадлежит `/new-person` reverse-flow.
     - Было: `| **Танджиро** | Natsuki Hanae |`
     - Стало: `| ![[Kamado_Tanjiro_cover.jpg\|60]] **[[Kamado_Tanjiro\|Танджиро]]** | Natsuki Hanae |`
   - Если постера нет — без миниатюры: `| **[[Kamado_Tanjiro\|Танджиро]]** | ... |`.
   - **Pipe-escaping обязательно** (`[[X\|Y]]`, `![[file.jpg\|60]]`) — иначе сломается структура таблицы.
   
   **Bullet-формат (legacy):**
   - Найти точную bullet-строку. Заменить голое имя на bold-WikiLink:
     - Было: `- **Танджиро** — главный герой, охотник на демонов.`
     - Стало: `- ![[Kamado_Tanjiro_cover.jpg|60]] **[[Kamado_Tanjiro|Танджиро]]** — главный герой, охотник на демонов.`
     - Если описания не было: `- Танджиро` → `- ![[Kamado_Tanjiro_cover.jpg|60]] **[[Kamado_Tanjiro|Танджиро]]**`.
   - Pipe-escaping не нужен (bullet — не table).
   
   Миниатюра добавляется только если постер успешно скачан в шаге 6.5. Если постера нет — только WikiLink. Делать без явного prompt'а.

   **6c. Решить, добавлять ли новую запись:**
   - `character_role` ∈ {`protagonist`, `deuteragonist`, `antagonist`} → добавлять по умолчанию (показать «добавляю в `ANIME/X.md`», без prompt-ожидания).
   - `character_role` ∈ {`supporting`, `minor`} → структурированным multi-choice вопросом спросить: «`ANIME/X.md`: добавить упоминание `<Имя>` (роль: `<role>`) в секцию персонажей?», дефолт **«нет»**.

   Если решили добавлять — **отзеркалить существующий формат секции.** Прочитать одну-две уже существующие записи, повторить их стиль:
   - **Табличный формат (2-col):** добавить новую строку. Col 2 (Сэйю) — если известен из MAL и есть `PERSONS/<va>.md` → gallery-формат; если известен без карточки → plain text; если не известен → `—`.
     ```markdown
     | ![[<File>_cover.<ext>\|60]] **[[<File>\|<Имя>]]** | <ячейка сэйю> |
     ```
   - **Bullet-формат:** повторить стиль.
   - Сортировка строк: protagonist → deuteragonist → antagonist → supporting → minor. Если порядок уже нарушен — не переупорядочивать, добавлять в конец подгруппы (или в конец, если подгрупп не видно).

7. **Обновить `updated`** в frontmatter карточки тайтла на сегодня (только если карточка реально менялась).

#### 7.5b. Sibling CHARACTERS/PERSONS — через `vault_text_mentions`

После 7.5a — проверить, не упоминают ли уже существующие карточки персонажей/персон создаваемого героя plain-text'ом (типовой случай: они создавались **до** этой карточки, поэтому имя стояло голым текстом).

**Алгоритм:**

1. Вызвать `vault_text_mentions(note="<File_Name>", noteKinds=["character", "person"])`. Tool вернёт `mentions[]` с `{source, sourcePath, line, lineText, matchedTerm}`, **уже отфильтрованный** от строк, где WikiLink на эту карточку уже стоит (идемпотентно при повторных прогонах).
2. Если `count == 0` — пропустить, в отчёт «sibling-карточек с plain-text упоминанием нет».
3. Для каждого `mention`:
   - **Прочитать** sourcePath через Read.
   - **Локализовать строку.** Проверить, что `mention.line` действительно содержит `mention.matchedTerm` plain-text'ом в bullet-стиле (`- **<имя>** — ...` или вариация). Если строка не bullet «связи», а часть прозы (например, в lead-параграфе или биографии) — **пропустить** (наша конвенция: navigation только в специализированных секциях).
   - **In-place upgrade.** Заменить голое имя на gallery-WikiLink, сохранив описание:
     - `- **<имя>** — <описание>.` → `- ![[<File_Name>_cover.<ext>\|60]] **[[<File_Name>\|<имя>]]** — <описание>.`
     - Если постера нет — без миниатюры: `- **[[<File_Name>\|<имя>]]** — <описание>.`
   - **Обновить `updated`** в frontmatter затронутой карточки на сегодня.
4. Лог в финальный отчёт: список затронутых файлов и строк, либо «sibling-апгрейдов не было».

**Что НЕ делать в 7.5b:**
- Не апгрейдить упоминания в lead/синопсисе/биографии sibling-карточек — даже если tool их вернул. Только bullet-записи в секциях связей (`## Связи`, `### В каноне`, `### За кулисами`, `## Ключевые работы` и аналоги). При сомнении — пропустить, отметить в отчёте.
- Не создавать новых записей в sibling-карточках, если упоминания нет вообще — это not implied; пользователь сам решит при следующем `/audit-note` или вручную.

**Не делать (общее для 7.5a/7.5b):**
- Не вставлять упоминания в другие секции тайтла (синопсис, сюжет, биография) — даже если там персонаж очевидно фигурирует.
- Не переписывать существующие записи в секциях связей (опечатки, форматирование — это задача `/audit-note`, не /new-character).

#### 7.5c. VA-cross-update — сэйю-карточки

Для каждой записи в `voice_actors[]` (все они уже прошли forward-check в шаге 4 — карточка `PERSONS/<va>.md` существует):

1. **Прочитать** `PERSONS/<va>.md`.
2. **Проверить `works[]`** — есть ли запись `{title: <anime>, roles: [voice-actor], character: <thisChar>}`. Если **отсутствует** — добавить во frontmatter. Если запись на этот `title` есть, но `character` — другой, или `voice-actor` нет в `roles[]` — это **отдельная** запись (см. правило в Metadata_schema: VA-запись определяется парой title+character, мешать с production-ролями нельзя). Не модифицировать существующую — добавить новую.
3. **В секции `## Ключевые работы`** VA-карточки добавить запись в gallery-формате (если ещё нет; см. `/new-person` шаг 6). Запись формируется по тайтлу: ширина 80 для cover тайтла, описание включает имя персонажа в plain text (карточка персонажа — текущая, навигация на неё с VA-карточки не нужна, потому что mutual link уже есть в `voice_actors[]` и `## Сэйю`):
   ```markdown
   - ![[<anime>_cover.<ext>|80]] **[[<anime>|<Title>]]** (<год>) — voice-actor (<Имя персонажа>), <одна строка контекста>.
   ```
4. **Идемпотентность.** Не добавлять дубль `{title, roles:[voice-actor], character}` в `works[]`. Не добавлять вторую gallery-запись на тот же тайтл с тем же VA-role.
5. **Минимальный патч.** Не трогать поля frontmatter за пределами `works[]`. Не «попутно чинить».

Если для какого-то VA из `voice_actors[]` карточки нет — этот шаг к нему неприменим (это противоречит инварианту forward-check). Если такая ситуация обнаружена — баг forward-check'а в шаге 4, исправить там.

#### 7.5d. Reciprocity closure — двунаправленное замыкание `character↔character` через `vault_backlinks`

Связь персонажей в `## Связи` обязана быть **строго двунаправленной** — односторонних линков быть не должно. Этот шаг замыкает реципрокность новой карточки в **обе стороны** и сам себя верифицирует.

Почему одного шага 6 мало: реверс-нога (7.5a–7.5c, на `vault_text_mentions`) апгрейдит карточки соседей, которые **уже упоминают** новую. Но два направления остаются дырявыми: (A) сосед упоминал новую, а forward-нога шага 6 не проставила обратный линк в новой карточке; (B) новая карточка **сама объявила** связь на соседа, который её не упоминал — реверс-нога его не тронет (нечего апгрейдить), и `vault_text_mentions` это не подсветит (он исключает строки с уже-стоящим линком). Оба направления видит только `vault_backlinks` + граф-детектор.

Единственный tool, который видит WikiLink независимо от истории строки, — `vault_backlinks`. §7.5d закрывает реципрокность `character↔character` для новой карточки **в обе стороны**: связь в `## Связи` — **строго двунаправленная**, односторонних быть не должно (политика волта). Это **create-time enforcement** (карточка рождается реципрокной); vault-wide подметание накопленного долга по всем парам из `reciprocity_pairs` — `/fix-links asymmetric` поверх `vault_asymmetric_links`.

Scope — **только `CHARACTERS/`** (`character↔character`). PERSONS не трогаем: `character↔person` by-design односторонняя (character→person в `## Сэйю` / `### За кулисами`; обратная сторона — FK `voice_actors[]↔works[]`, чинится шагом 7.5c). Anime/studio — FK-backed. То же, что `reciprocity_pairs: [[character, character]]` в манифесте — см. `docs/Asymmetric_Links_Spec.md §5`.

**Направление A — кто ссылается НА новую карточку** (бэкстоп форвард-ноги шага 6):

1. `vault_backlinks(note="<File_Name>")` → источники из `CHARACTERS/`, ссылающиеся на новую карточку.
2. Для каждого S: в `## Связи → ### В каноне` новой карточки должен быть обратный WikiLink на S. Уже есть → пропустить (идемпотентность); plain-bold → апгрейд до gallery-WikiLink; нет вовсе → добавить запись (gallery-формат шага 6).

**Направление B — кого новая карточка линкует САМА** (закрывает кейс «новая объявила связь, а цель не ответила»):

3. Собрать все WikiLink на `CHARACTERS/*` из `## Связи` **самой новой карточки**. Для **каждой** цели T прочитать `CHARACTERS/<T>.md`, проверить её связевую секцию (`## Связи` / `### В каноне`):
   - обратный WikiLink на новую карточку уже есть → ОК;
   - имя новой карточки стоит plain-bold → апгрейдить до WikiLink;
   - упоминания нет вовсе → **добавить** обратную запись на новую карточку (миниатюра — только если у новой карточки есть `images.cover`, иначе WikiLink без миниатюры; описание связи — по смыслу, с точки зрения T) и **обновить `updated`** карточки T на сегодня.
   - **Это override** консервативного правила 7.5b «не создавать записей в карточках без упоминания»: для связей, которые новая карточка **явно объявила**, обратная запись **обязательна** (политика «связь двунаправленна железно»). Дисциплина смещается вверх: в `## Связи` новой карточки кладём только связи, которые реально стоит делать двунаправленными — всё, что там есть, размножится в цели.

**Само-проверка (обязательно).** После A+B вызвать `vault_asymmetric_links(sourceKind="character", targetKind="character")`. В `pairs[]` не должно быть **ни одной** записи, где `source` **или** `target` = новая карточка. Если есть — петля не закрыта, доделать до нуля.

4. Лог в финальный отчёт: обратные связи, добавленные/апгрейженные в обе стороны, + результат само-проверки («asymmetric для новой карточки = 0»).

**Не делать:** не трогать backlink/упоминание, если оно в прозе/синопсисе/биографии цели, а не в связевой секции (навигация живёт только в `## Связи`). При сомнении — отметить в отчёте, не трогать.

### 8. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** для **всех изменённых заметок** (созданная карточка персонажа + опционально обновлённая карточка аниме из шага 7.5). Если найдены ошибки — исправить. Если предупреждения — упомянуть в отчёте.

### 9. Сообщить результат

Показать: имя файла, путь, основное имя, число строк, ключевые WikiLinks, обновлённые связанные заметки (если были), результат аудита. Предложить `/verify` если устраивает.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **Ставить `co_authored: <model-id>`** при создании. Формат `claude-opus-4.7` / `claude-sonnet-4.6` / `claude-haiku-4.5` (источник — системный промпт, последний дефис → точка). `/verify` это поле не трогает.
- Web research — обязательная часть фазы 2. Не выдумывать факты, не додумывать данные. Численные/перечисляемые факты (`age`, `mal_url`-id) — через web fetch конкретной страницы.
- **Постер скачивать всегда, когда возможно — но через vision-gate (шаг 6.5).** Часть контракта скилла: галерея персонажей в карточке тайтла заполняется через шаг 7.5b, и без постера апгрейд text → миниатюра+WikiLink становится text → WikiLink (без миниатюры — дыра в галерее). **Однако** подбор/валидация картинки требует мультимодальности: не-мультимодальная модель проходит gate → пропускает постер с пометкой в отчёте и доделывает текстовую часть (НЕ оценивает картинку по имени/URL — это брак). Пропуск также если ни один источник не дал годной картинки или задан `--no-images`. Во всех случаях — явно в отчёте.
- При обнаружении новых тегов, которых нет в `SYSTEM/Tag_taxonomy.md` — автоматически добавить их туда.
- Карточка персонажа имеет смысл только в связке с тайтлом: пустой `featured_in` — повод спросить пользователя «а карточка тайтла есть?» до сохранения.
- `character_role` — про доминирующую роль в основном произведении. Если персонаж главный в S1 и эпизодический в S3 — `protagonist`. Не пытаться смоделировать роль-per-тайтл.
- `status: unknown` использовать только если канон явно не даёт ответа (открытый финал, незавершённый тайтл). Для «персонажа в открытом сериале, который сейчас жив» — `alive`.
