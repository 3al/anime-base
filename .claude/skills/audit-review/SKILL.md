# /audit-review — Аудит секции «Личный отзыв»

Полирует текст личного отзыва на карточке и извлекает мета-сигналы (оценка, статус) для синхронизации frontmatter. Узкий scope — **только** секция `## Личный отзыв`, остальное тело и frontmatter не трогаются (для широкого аудита есть `/audit-note`).

## Аргумент

`$ARGUMENTS` = `<note>` — имя карточки без `.md`, относительный путь от корня волта, или абсолютный путь.

Примеры:
- `/audit-review Nausicaa_of_the_Valley_of_the_Wind`
- `/audit-review ANIME/Demon_Slayer`
- `/audit-review PERSONS/Hayao_Miyazaki` — работает с любым kind, у которого есть `## Личный отзыв`

## Источники правил

1. `SYSTEM/Metadata_schema.md` — `personal_score`, `personal_status`, `opening_score`/`ending_score` (anime); аналогичные поля для будущих kinds.
2. `SYSTEM/enums.yaml` — допустимые значения `personal_status` для kind карточки.
3. `SYSTEM/Tag_taxonomy.md` — обязательный тег статуса для kind (например, anime).

## Алгоритм

### 1. Резолв заметки

1. Если `<note>` содержит `/` или `\` — нормализовать к относительному пути; убедиться, что файл существует.
2. Иначе — Glob `**/<note>.md`:
   - Один матч → использовать.
   - Несколько → структурированный multi-choice вопрос с выбором.
   - Ноль → ошибка.
3. Прочитать файл, извлечь frontmatter и тело.

### 2. Найти секцию

Поиск **строгого** заголовка `## Личный отзыв` (case-sensitive, точное совпадение). Если не найдено — ошибка: «В карточке нет секции `## Личный отзыв`. Если хотите аудит всей заметки — используйте `/audit-note`».

Найти границы секции: от строки `## Личный отзыв` до следующей строки `## ...` (любой H2) или до конца файла.

### 3. Detect stub

Если содержимое секции (без заголовка, trimmed) совпадает с автогенерированной заглушкой `/new-anime` или подобной:

- Точное совпадение или содержит фразу «Карточка создана со статусом» + «Личные впечатления добавятся после просмотра».
- Стаб-приглашение `/new-character`: содержит «Заполните личный отзыв» (курсивная строка «_Заполните личный отзыв в Obsidian: чем персонаж зацепил или почему не зашёл…_»).
- Или содержимое < 30 символов осмысленного текста.

→ Прервать выполнение с сообщением: «Секция отзыва ещё не заполнена пользователем. Заполните содержание в Obsidian, затем запустите `/audit-review` снова». Frontmatter не трогать.

### 4. Polish phase (механические правки)

Применяются автоматически без подтверждения (обратимо через git/undo). Только **в пределах секции отзыва**.

Конкретные правки:

| Что | Из | В |
|---|---|---|
| Дефис как тире в прозе | ` - ` (с пробелами по бокам) или `--` | ` — ` (em dash + пробелы) |
| Двойные пробелы между словами | `  ` | ` ` |
| Двойные кавычки в RU-контексте | `"text"` (Latin double) внутри кириллицы | `«text»` |
| Пробел перед знаком препинания | `слово ,` / `слово .` | `слово,` / `слово.` |
| Утроенный знак препинания | `...!` `???` `!!!` | `…` / `?` / `!` (одиночный) |
| Опечатки | очевидные опечатки в русском/английском | исправлено |
| Markdown — незакрытые маркеры | `**жирный` без закрытия | `**жирный**` |
| Markdown — лишние пустые строки | `\n\n\n+` | `\n\n` |

**Не делать:**
- Не переписывать стиль/перефразировать.
- Не трогать прямые цитаты (внутри `« »`, `>` блоков, code fences).
- Не «улучшать» формулировки — это `/audit-note`.

### 5. Metadata extraction (consertvative)

Сканировать **только** секцию отзыва на явные паттерны. Намёки/настроение не угадывать.

#### 5.1. Оценка → `personal_score`

Паттерны (regex-уровень):
- `\b(10|[1-9])\s*/\s*10\b` — «9/10», «10 / 10»
- `(?:оценка|ставлю|поставил[аи]?|даю)[:\s]+(\b10|\b[1-9])\b(?!\s*/)` — «оценка: 8», «ставлю 9»
- `\b(10|[1-9])\b\s*из\s*10` — «8 из 10»

Если паттерн найден — извлечь число `N`. Сравнить с текущим `frontmatter.personal_score`:
- Совпадает → ничего не делать.
- Отличается / отсутствует → собрать в `extracted_changes.personal_score = N`.

Если несколько паттернов с разными значениями — выбрать **последнее по позиции** (предполагаем, что итоговая оценка — в конце отзыва).

#### 5.2. Оценки опенинга и эндинга → `opening_score` / `ending_score`

**Применяется только к kind'ам, чья схема определяет эти поля** (сейчас — `anime`). Если в `Metadata_schema.md` для kind карточки полей `opening_score`/`ending_score` нет (или в самой карточке их нет во frontmatter) — пропустить этот шаг целиком, не предлагать.

Это **независимые** оси от `personal_score`: оценку OP/ED **нельзя** выводить из общей оценки или настроения отзыва. Извлекать **только** при явном совпадении двух условий в одном предложении/клаузе:

1. **Маркер элемента** (что именно оценивается):
   - Опенинг: `опенинг`, `опен(?!\w)`, `\bOP\b`, `заставк`, `вступительн(ая|ую|ой) (тема|песня|композиция)`, `открывающ(ая|ую) (тема|песня|композиция)`.
   - Эндинг: `эндинг`, `эндин(?!\w)`, `\bED\b`, `закрывающ(ая|ую) (тема|песня|композиция)`, `финальн(ая|ую) (песня|тема|композиция)`.
2. **Балл рядом** — те же score-паттерны, что в 5.1 (`\b(10|[1-9])\s*/\s*10\b`, `(?:оценка|ставлю|даю)[:\s]+...`, `\b(10|[1-9])\b\s*из\s*10`, плюс `(?:опенинг|эндинг|опен|OP|ED)\D{0,15}\bна\s+(10|[1-9])\b` для «опенинг на 10»).

**Осторожно с «концовка» / «финал».** Эти слова в отзыве чаще означают **сюжетную** развязку, а не ED-песню. Засчитывать их в `ending_score` **только** если в той же клаузе явно стоит музыкальный контекст (`песн`, `тема`, `композиц`, `музык`, `саундтрек`, `OST`). Иначе — игнорировать (это не про эндинг как элемент).

Извлечь `N` для каждого найденного элемента, сравнить с текущим значением:
- Совпадает → ничего не делать.
- Отличается / `null` / отсутствует → собрать в `extracted_changes.opening_score = N` / `extracted_changes.ending_score = N`.

Если для одного элемента несколько разных баллов — выбрать **последний по позиции** (как в 5.1).

#### 5.2b. Оси art / story / originality → `art_score` / `story_score` / `originality_score`

**Применяется к kind'ам, чья схема определяет эти поля** (`anime`, `manga`). Если полей нет в схеме kind или во frontmatter карточки — шаг пропустить. Это **независимые** оси (как OP/ED): балл нельзя выводить из общей оценки или настроения отзыва. Извлекать **только** при явном совпадении маркера оси + балла рядом в одной клаузе (те же score-паттерны, что 5.1, плюс `<маркер>\D{0,15}\bна\s+(10|[1-9])\b` для «рисовка на 9»):

- **art_score** — маркеры: `рисовк`, `прорисовк`, `анимаци`, `арт(?!\w)`, `визуал`, `график`, `картинк`, `стиль рисунка`.
- **story_score** — маркеры: `сюжет`, `сценари`, `истори(я|ю|и)`, `повествован`, `драматург`, `арк[аи]`.
- **originality_score** — маркеры: `оригинальн`, `необычн`, `свеж(ий|есть|о)`, `эксперимент`, `новатор`, `нестандартн`.

Извлечь `N` для каждой найденной оси, сравнить с текущим (совпадает → ничего; отличается/`null`/нет → `extracted_changes.<ось>_score = N`). Несколько баллов на одну ось — последний по позиции (как 5.1). Не угадывать по настроению — только явный «<ось> на N» / «N/10 за <ось>».

#### 5.3. Статус → `personal_status`

Паттерны (case-insensitive, кириллица):
- «(досмотрел|закончил|пересмотрел|посмотрел) (все|до конца|целиком|залпом)», «всё (\d+ серий|серии)» → candidate `completed`
- «бросил(а)?», «дроп(нул(а)?)?», «не дос(мотрел(а)?|илил)» → candidate `dropped`
- «отлож(ил(а)?|ено)», «приостанов(ил(а)?|лено)» → candidate `on-hold`
- «любим(ое|ая)», «топ-?\d?», «в избранное», «лучшее что я смотрел» → candidate `favorite` **только** если уже есть индикатор завершения или текущий status уже `completed`.

Если найдены конфликтующие сигналы (например, «бросил, но любимое» — редкий случай) — пометить конфликт и спросить пользователя без автоматического выбора.

Сравнить кандидата с `frontmatter.personal_status`:
- Совпадает → ничего не делать.
- Отличается → собрать в `extracted_changes.personal_status = <candidate>`.

#### 5.4. Валидация по enums.yaml

Перед предложением изменения — проверить, что кандидат `personal_status` присутствует в `enums.yaml::note_kinds.<kind>.personal_status`. Если нет (например, скрипт нашёл `dropped` в карточке `note_kind: person`, где такого статуса нет) — пропустить кандидата.

### 6. Подтверждение изменений

Если `extracted_changes` непустой — для **каждого** изменения отдельно структурированным multi-choice вопросом:

> «В отзыве упоминается оценка 9 (паттерн «9/10»). Обновить `personal_score`?»
>
> - **Да, обновить** — выставить новое значение во frontmatter.
> - **Нет, оставить** — не трогать (текущее значение `null` / 7).

Если пользователь отказывается — оставить frontmatter как есть, но **полировка из шага 4 остаётся** (это бесспорные правки).

**Если в `tags[]` обнаружен тег-дубль значения `personal_status`** (например, `tags: [plan-to-watch, ...]` при `personal_status: plan-to-watch`) — упомянуть в отчёте как замечание о tag-status дубле и предложить убрать. Не делать автоматически, не блокировать выполнение скилла. По принципу Tag_taxonomy не дублировать поля frontmatter в тегах.

### 7. Применение

1. **Edit секции** — записать polished текст. Один Edit на всю секцию.
2. **Edit frontmatter** — точечно изменить только согласованные поля (`personal_score`, `opening_score`, `ending_score`, `art_score`, `story_score`, `originality_score`, `personal_status`, `updated`). `tags[]` не трогать.
3. **Bump `updated`** — на сегодняшнюю дату, **только если** были изменения (полировка или мета). Если ни секция, ни frontmatter не менялись — не трогать.

**Минимальный патч.** Не нормализовывать другие поля, не сортировать ключи. Только нужные строки.

**`co_authored` НЕ трогать.** `/audit-review` — ревью-пас (полирует пользовательский отзыв + извлекает метаданные), не автор сути карточки. Перезапись на судью уничтожила бы origin-автора — субъект оценки качества модели (леджер). `co_authored` ставят только производящие контент скиллы (`/new-*` / `/expand-stub` / `/vault-rag`). См. `feedback_co_authored` + `audit-by-creator/references/ledger-protocol.md §6`.

### 8. Самоаудит

Запустить `vault_lint` (vault-index MCP `vault_lint`) с `target` = путь к заметке. Issues в отчёт дословно.

### 9. Отчёт

```
## /audit-review: <относительный путь>

Polish phase:
  • <N> typo fixes
  • <M> punctuation normalizations
  • <K> spacing fixes
  ... (или «без изменений», если ничего не найдено)

Metadata extraction:
  • Найдено: personal_score=9 (паттерн «9/10»)         → принято / отклонено
  • Найдено: opening_score=10 (паттерн «опенинг на 10») → принято / отклонено
  • Найдено: personal_status=completed (паттерн «...») → принято / отклонено
  ... (или «не найдено явных мета-сигналов»)

Frontmatter обновления:
  • personal_score: null → 9
  • opening_score: null → 10
  • personal_status: plan-to-watch → completed
  • tags: [plan-to-watch, ...] → [completed, ...]
  • updated: <today>

Самоаудит (vault_lint):
  <warnings/errors дословно, либо «без замечаний»>
```

## КРИТИЧНО

1. **Scope строгий — только секция `## Личный отзыв`.** Не трогать другие секции, даже если в них видно опечатки. Для широкого аудита — `/audit-note`.
2. **Метаизменения — только структурированным multi-choice вопросом.** Никакого автоматического обновления `personal_score` или `personal_status`, даже если extraction уверен на 100%.
3. **Tags не трогать.** `personal_status` живёт в frontmatter; дублирующих status-тегов в `tags[]` быть не должно (правило Tag_taxonomy «не дублировать поля frontmatter»). Если такой тег обнаружен — флагать в отчёте, но не править автоматически.
4. **Polish — бесспорные правки.** Не превращать audit-review в литературный редактор. Если хочется переписать стиль — это явный запрос пользователю, а не дефолт скилла.
5. **Не трогать секцию-stub.** Скилл — для зрелых отзывов, не для заглушек.
6. **Source-of-truth для значений status — `enums.yaml`.** Если паттерн вытащил статус, которого нет в enum для kind карточки — пропустить кандидата (не предлагать пользователю расширить enum здесь; это задача `/new-anime`/`/add-note-kind`).
7. **`co_authored` НЕ трогать** — `/audit-review` ревьюит пользовательский отзыв, не авторствует суть карточки (см. шаг 7 / `ledger-protocol.md §6`: аудит не авторствует, иначе теряется субъект оценки модели). Origin-автор переживает ревью. `/verify` тоже не трогает.
