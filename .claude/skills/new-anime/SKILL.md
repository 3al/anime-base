# /new-anime — Создание карточки аниме

Создаёт карточку аниме с заполнением всех обязательных полей frontmatter согласно `note_kind: anime` и каркасом тела заметки.

## Аргумент

`$ARGUMENTS` — название аниме на русском, английском или оригинальном языке.

Примеры:
- `/new-anime "Demon Slayer"`
- `/new-anime Kimetsu no Yaiba`
- `/new-anime Атака титанов`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек Аниме (`note_kind: anime`)». Полный список обязательных и опциональных полей + семантика значений enum.
2. **`SYSTEM/enums.yaml`** — раздел `note_kinds.anime`. **Машинно-читаемый** список допустимых значений для всех enum-полей kind'а (`format`, `status`, `season`, `source`, `personal_status`, `genres`). **Это единственный источник истины** — не подставлять значения из памяти/SKILL.md, если они не присутствуют в `enums.yaml`.
3. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: anime`. Какие WikiLinks обязательны.
4. **`SYSTEM/Tag_taxonomy.md`** — обязательные теги по `note_kind: anime`.
5. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Найти файл через Glob в `ANIME/`. Если карточка уже существует:
- **Stub / тонкая (<20 строк контента)** — не создавать с нуля, а дополнить. Сохранить существующие данные.
- **Наполненная (>50 строк)** — сообщить пользователю. Спросить, что обновить.

### 2. Сбор данных

Найти актуальные данные через web search. Приоритетные источники для аниме:
- **MyAnimeList** — главный каталог: формат, эпизоды, год, сезон, студия, жанры, средняя оценка. Актуальный домен: **`https://myanimelist.net/`**
- **AniList** (anilist.co) — структурированные данные, related_titles
- **Shikimori** — русскоязычное описание, средняя оценка по RU-аудитории. Актуальный домен: **`https://shikimori.io/`** (раньше был `shikimori.one` / `shikimori.org` — сайт периодически переезжает, при недоступности проверять свежий зеркальный домен через поиск).
- **Wikipedia** — fallback для синопсиса и культурного контекста

**КРИТИЧНО про числовые поля (`mal_score`, `shikimori_score`, episodes, duration, year).** Брать **только из прямого web fetch конкретной страницы** этого аниме на соответствующем сайте. **Не использовать** числа из summary поисковика (web search) — там бывают вырванные из контекста значения с других сайтов / Kodik / случайных обзоров; провенанс не отслеживается. Для каждой такой оценки в карточке должна быть конкретная страница, из которой она извлечена, в качестве источника. Если web fetch на нужный домен упал (403/404/таймаут) — лучше **пропустить** поле (оно опциональное), чем подставить число из неоднозначного источника. Этот же принцип распространяется на любые будущие источники оценок (Kinopoisk, IMDB, Rotten Tomatoes и т.п.).

**Замечание о доменах.** MAL и Shikimori периодически меняют домены (Shikimori особенно — мигрирует между `.one`, `.io`, `.org`). При записи `mal_url` во frontmatter и ссылки на Shikimori в секции «## Ссылки» — использовать тот домен, что **актуален на момент создания**. Если URL в существующей карточке указывает на устаревший домен — это нормально, не «чинить» автоматически; обновлять только при явном пользовательском запросе.

Собрать данные для всех обязательных полей frontmatter (см. шаг 4). Если ищется не очень популярный/нишевый тайтл — проверить несколько источников, разногласия флагать в отчёте.

**Дополнительно — список персонажей.** На MAL странице тайтла есть раздел Characters & Staff (`<mal_url>/characters`), на AniList — поле `characters` в GraphQL Media. Извлечь **всех основных** (роли `Main` + `Supporting` в терминах MAL), пропустить эпизодических (`Minor`/`Background`). Для каждого собрать: имя ромадзи, роль (`Main` → `protagonist`/`deuteragonist`/`antagonist` — определить по сюжету; `Supporting` → `supporting`), 1 строка краткого описания (кто это в тайтле, ключевая черта). Это понадобится в шаге 5 для секции `## Персонажи`.

**Сэйю — обязательно собрать per-character.** На `<mal_url>/characters` для каждого main+supporting персонажа указан Japanese voice actor. Извлечь имя сэйю как **отдельное поле** для каждого персонажа (понадобится в шаге 5 для второй колонки таблицы `## Персонажи`). Если у MAL-роли стоит несколько VA (старый/новый каст по сезонам) — брать **первого** в списке (обычно canonical для этого тайтла). Английский/русский dub в anime-карточке **не отображается** — это character-side данные.

**Forward-check на существующие карточки персонажей и сэйю.** Для каждого собранного имени — двойной Glob:
- `CHARACTERS/<PascalCase_имя_персонажа>.md` — для col 1 таблицы.
- `PERSONS/<PascalCase_имя_сэйю>.md` — для col 2 таблицы.

Если карточка **уже есть** — прочитать её frontmatter, запомнить `images.cover` (если есть). Эти данные используются в шаге 5: для существующих карточек ячейка пишется сразу в gallery-формате (миниатюра 60 + WikiLink), для остальных — plain text. Reverse-апгрейд: persона-плейн → WikiLink делает `/new-character` (для col 1) и `/new-person` (для col 2 — через свой шаг 7.5b в анимешные таблицы). См. `Linking_guidelines.md` секция «Gallery-стиль» и `### note_kind: anime`.

**Данные для секции «Резонанс» (опционально, для законченных тайтлов).** Если `status: finished` — собрать данные о приёме тайтла для секции `## Резонанс` (шаг 5). Источники:

- **MAL stats page** (`<mal_url>/stats`) — `Ranked #`, `Popularity #`, `Members`, `Favorites`. Структурно, стабильный путь через web fetch.
- **Wikipedia EN** (`en.wikipedia.org/wiki/<Title>`) — секция «Reception» / «Awards» / «Sales»: премии, цитаты критиков, BD/box office, влияние на исходник.
- **Crunchyroll Anime Awards / Tokyo Anime Award / Newtype Awards** — обычно агрегируются на Wikipedia, отдельно лезть не нужно.
- **Box Office Mojo** (`boxofficemojo.com`) — **только для `format: movie`**, числовые сборы.

**Cutoff правила** (когда секцию НЕ создавать):
- `status != finished` (онгоинги, on-hiatus, upcoming, cancelled) → секцию опустить, ранги волатильны.
- MAL `members < 50000` **И** Wikipedia 404 → нишевый тайтл без данных, секция не создаётся.
- Wikipedia есть, но MAL `members < 50000` → допустимо создать секцию (жанровое влияние / нишевый культурный вес — валидный контент даже без массовой популярности).
- Только MAL stats, без Wikipedia/наград → допустимо создать секцию с одним подразделом «Оценки и популярность» (1 строка), остальные опустить.

**Числовые факты приёма (BD-тиражи, box office, число копий исходника) — только через прямой web fetch конкретной страницы** (Wikipedia, Box Office Mojo, Oricon). Не подставлять числа из web search snippet'ов — тот же принцип, что для `mal_score` / `shikimori_score`. Если конкретики нет — формулировать качественно («один из самых высоких BD-тиражей года», «закрепил статус флагмана студии»), без вымышленных цифр.

### 3. Определить имя файла

PascalCase с подчёркиваниями вместо пробелов, латиница, без диакритики. По умолчанию — `title_romaji` с заменой пробелов на `_`. Примеры: `Demon_Slayer.md`, `Fullmetal_Alchemist_Brotherhood.md`. Если ромадзи слишком длинный или менее узнаваем — английский официальный заголовок: `Attack_on_Titan.md`. Избегать спецсимволов (`:`, `!`, `?`) — заменять пропуском или дефисом.

В `aliases` минимум: английский официальный заголовок (если основное имя в ромадзи), кириллическая транслитерация японского или официальная русская локализация. Дополнительно — известные фанатские сокращения (FMA, AoT, JJK).

### 4. Собрать frontmatter

**Прежде чем подставлять значения** — прочитать `SYSTEM/enums.yaml` (раздел `note_kinds.anime`) и убедиться, что каждое подставляемое значение enum-поля присутствует в соответствующем списке.

**Если из web-research пришло значение вне enum'а** (например, MAL вернул жанр `Award Winning` или новый source `picture-book`, которого в `enums.yaml::source` нет) — **не вписывать молча** и **не маппить на ближайший существующий**. Структурированным multi-choice вопросом спросить пользователя:

> «В `enums.yaml::note_kinds.anime.<field>` отсутствует значение `<новое>`, но оно есть в источнике (`<источник, например MAL>`). Добавить в enums.yaml?»
>
> - **Добавить** — Edit `SYSTEM/enums.yaml`, дописать значение в нужный список (сохранить порядок: append в конец, не сортировать). Затем использовать в карточке.
> - **Пропустить** — значение в карточку **не вписывать**. Если поле обязательное и нет fallback'а — отказать создавать карточку, попросить пользователя определиться с маппингом вручную.
> - **Заменить на существующее** — пользователь указывает, какое из существующих значений использовать как маппинг. В карточку идёт указанное.

Не считать «`Award Winning` это очевидно `award-winning`» — для пользователя это может быть мусорным значением, которое не нужно тащить в формализованный enum. Решение всегда за пользователем.

После Edit `enums.yaml` — продолжить заполнение карточки. В отчёте упомянуть факт расширения enum'а отдельной строкой (важно: это изменение governance-документа, пользователю полезно видеть).

Заполнить ВСЕ обязательные поля для `note_kind: anime`:

```yaml
---
type: reference
domain: anime
stability: stable
priority: medium
quality: draft
note_kind: anime
tags:
  - <теги по Tag_taxonomy>
aliases:
  - <альтернативные названия>
created: <сегодня в YYYY-MM-DD>
updated: <сегодня в YYYY-MM-DD>
title_romaji: "<Romaji-заголовок>"
title_original: "<Оригинальное название>"
format: <значение из enums.yaml::note_kinds.anime.format>
status: <значение из enums.yaml::note_kinds.anime.status>
episodes: <число>
year: <YYYY>
genres:
  - <значение из enums.yaml::note_kinds.anime.genres>
  - <значение из enums.yaml::note_kinds.anime.genres>
personal_status: <значение из enums.yaml::note_kinds.anime.personal_status>
# Рекомендуемые — ВСЕГДА присутствуют во frontmatter (политика «always-present»).
# Если данных нет — оставлять null / "" / [] (см. ниже). НЕ опускать поле целиком —
# пустой слот делает поля discoverable, пользователь видит, что можно заполнить.
title_english: "<English-заголовок или ''>"
season: <значение из enums.yaml::note_kinds.anime.season, или null>
studio: "<Имя_файла_студии или ''>"
source: <значение из enums.yaml::note_kinds.anime.source, или null>
episode_duration_minutes: <число или null>
personal_score: <1-10 или null>     # обычно null при создании — пользователь поставит после просмотра
times_watched: <число или null>     # общее число просмотров: 1 = смотрел один раз, 3 = трижды. null для plan-to-watch / dropped
last_episode_watched: <число или null>  # для completed/favorite = episodes; для watching/on-hold = текущий прогресс; для plan-to-watch = null
watched_with: []                    # list[enum] из enums.yaml::note_kinds.anime.watched_with. [] если смотрел один или ещё не смотрел
mal_score: <0.00-10.00 или null>
shikimori_score: <0.00-10.00 или null>
related_titles: []                  # список foreign-keys, [] если связей нет
staff: []                           # list[dict] из {person, role}, [] если состав не известен
mal_url: "https://myanimelist.net/anime/<id> или ''"
---
```

**Соглашения по пустым значениям:**
- Строки → `""` (пустая строка в кавычках) если данных нет.
- Числа/даты → `null` (YAML: явное `null` или просто `field:` без значения).
- Списки (`aliases`, `related_titles`, `staff`, `genres`) → `[]` если пусто. Для `genres` минимум 1 значение требуется, поэтому `[]` сюда не подходит — если жанры неизвестны, **не создавать карточку**, попросить уточнить.
- Перечисления (enum) → `null`, не `""`. `null` понимается как «не задано», `""` валидным enum-значением не является.

**`staff[]` — отдельные записи на каждую роль.** Если один человек выполнял несколько ролей (Хаяо Миядзаки = режиссёр + сценарист) — это **две** записи в `staff[]`, не объединённая `roles: [director, screenwriter]`. Сделано для упрощения фильтра «все режиссёрские работы X» — flat scan вместо list-into-list.

**Personal_status — только поле frontmatter, не тег.** Принцип Tag_taxonomy: не дублировать поля frontmatter в тегах. `personal_status` однозначно определяется enum-полем (см. `enums.yaml::note_kinds.anime.personal_status`); тег-дубликат тут не нужен. Если пользователь не задал статус — по умолчанию `plan-to-watch` (в поле, не в tags). Аналогично для `format` — это поле, не тег.

**Семантика `favorite`:** взаимоисключает `completed`. Если тайтл завершён и попал в фавориты — ставится `favorite`, не `completed`. Не ставить `favorite` для незавершённых тайтлов (`watching`/`plan-to-watch`/`on-hold`/`dropped`).

**Теги подбирать пошагово** (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня
2. ДОМЕН/ВЕНДОР
3. ТЕМЫ содержания
4. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с `format`/`personal_status`/`status`/`domain`/`source` и любыми другими enum-полями frontmatter (они уже там — тег-дубль нарушает Tag_taxonomy правило «не дублировать»).
6. **ПРОБЕЛЫ в таксономии — ОБЯЗАТЕЛЬНО** добавить новый тег в `SYSTEM/Tag_taxonomy.md` если используешь тег, которого там ещё нет.

**Ставить `co_authored: <model-id>`** при создании. Значение — ID текущей модели в волтовом формате `claude-<family>-<major>.<minor>` (например `claude-opus-4.7`, `claude-sonnet-4.6`, `claude-haiku-4.5`). Источник истины — системный промпт, строка «The exact model ID is …» (последний дефис → точка: `claude-opus-4-7` → `claude-opus-4.7`). См. `feedback_co_authored`.

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <title_romaji> (<title_original>)

**<Основное название>** — <1-2 предложения: жанр, основная завязка, ключевая характеристика, почему интересно>.

## Краткий синопсис

<1-2 абзаца: завязка, главный конфликт, тон. Без серьёзных спойлеров — это hook, остаётся видимым всегда.>

## Сюжет

> [!warning]- Спойлеры
>
> <Подробный разбор по сезонам или ключевым аркам. Здесь спойлеры допустимы. Каждая строка контента префиксируется `> ` — это Obsidian callout-синтаксис. Минус после `]` делает callout схлопнутым по умолчанию (`> [!warning]- ...`), без минуса — раскрыт.>
>
> <Несколько абзацев — все внутри callout. Раскрывается одним кликом по заголовку.>

## Персонажи

<Все основные персонажи тайтла из шага 2: protagonist → deuteragonist → antagonist → supporting (в этом порядке, главные сверху). Эпизодические (`minor`) — пропускать.>

<Формат — **2-колоночная Markdown-таблица**: `Персонаж | Сэйю`. Это vault-wide конвенция для секции `## Персонажи` на anime-карточке: cast-list в стиле MAL/AniList. Описание layout'а — `Linking_guidelines.md` → «Структура секции `## Персонажи`».>

<**Описание роли НЕ выносится в таблицу.** Контекст по конкретному персонажу — на character-карточке (`CHARACTERS/<имя>::## Описание` для открытой информации + `## Биография` с спойлерным callout для плот-арки). Anime-карточка содержит общий синопсис в `## Краткий синопсис` и детали сюжета в `## Сюжет` (под `> [!warning]- Спойлеры` callout). Дублировать описание в таблицу не нужно — это узкий cast-канал.>

<**Почему не 3-col с инлайн-спойлерами.** Пробовали `Персонаж | Сэйю | Роль` с `||...||` обёрткой плот-ревелейшенов через плагин obsidian-inline-spoilers. Markdown-таблицы используют `|` как разделитель колонок; `||spoiler||` ломает структуру, table-formatter режет ячейки. Pipe-escaping `\|\|` ломает детект плагином. Окончательно решено: cast-only таблица; контекст — на character-карточках.>

<**Pipe-escaping.** Все `[[X|Y]]` внутри ячеек таблицы пишутся как `[[X\|Y]]` (backslash перед pipe). То же для embed'ов: `![[X.jpg|60]]` → `![[X.jpg\|60]]`. Это требование Markdown — иначе pipe ломает структуру таблицы.>

<**Содержимое ячеек col 1 (Персонаж) и col 2 (Сэйю)** — зависит от forward-check'а из шага 2:>

<- **Карточка цели существует + `images.cover`** → `![[<File>_cover.<ext>\|60]] **[[<File>\|<Имя>]]**`>
<- **Карточка существует, без cover** → `**[[<File>\|<Имя>]]**`>
<- **Карточки нет** → `**<Имя>**` (plain bold)>

<Это единственная секция в карточке тайтла, где допустимы WikiLinks на персонажей и VA — конвенция single-channel.>

```
| Персонаж | Сэйю |
|---|---|
| ![[Kamado_Tanjiro_cover.jpg\|60]] **[[Kamado_Tanjiro\|Танджиро]]** | ![[Hanae_Natsuki_cover.jpg\|60]] **[[Hanae_Natsuki\|Natsuki Hanae]]** |
| **[[Kamado_Nezuko\|Незуко]]** | **[[Kito_Akari\|Akari Kitō]]** |
| **Зеницу** | Hiro Shimono |
| **Музан** | Toshihiko Seki |
```

<**Сортировка строк.** protagonist → deuteragonist → antagonist → supporting. Внутри supporting — порядок MAL (он обычно отражает значимость).>

<**Пустая VA-ячейка.** Если VA для персонажа не удалось определить из MAL (редкий случай — обычно для оригинальных персонажей-фонов) — поставить `—` (em dash) в ячейку. Не оставлять пустую (Markdown table требует непустой контент в ячейке для корректного рендера).>

## Связанные тайтлы
```

### 6.5. Постер (интерактивно)

Структурированным multi-choice вопросом спросить пользователя:

> «Найти постер тайтла в сети и добавить в карточку?»
>

> - **Нет, добавлю позже** — пропустить весь шаг 6.5; в финальном отчёте указать, что постер и любые дополнительные кадры (key visuals, скриншоты) можно добавить позже скиллом `/add-images <note> poster <file>` (главный постер) или `/add-images <note> gallery <file-or-folder>` (галерея).

При **отказе** — `images` во frontmatter не появляется, тело карточки остаётся без эмбеда. Переходим к шагу 7.

При **согласии** выполнить:

#### 6.5.1. Найти URL обложки

Источники в порядке приоритета:

1. **MAL og:image.** web fetch на `mal_url` с prompt'ом «Извлеки URL изображения из `<meta property="og:image">` и `<meta property="og:image:secure_url">`. Верни прямую ссылку на CDN MAL (`https://cdn.myanimelist.net/images/anime/...`).». Это самый стабильный путь — у MAL предсказуемая разметка.
2. **AniList GraphQL** (fallback). POST на `https://graphql.anilist.co` с телом:
   ```json
   {"query":"query($idMal:Int){Media(idMal:$idMal,type:ANIME){coverImage{extraLarge large medium}}}","variables":{"idMal":<id>}}
   ```
   `<id>` — числовой MAL id из `mal_url` (для Навсикаи это `572`). Через Bash + `curl -s -H "Content-Type: application/json" -d '<body>' https://graphql.anilist.co | jq -r '.data.Media.coverImage.extraLarge'`.
3. Если оба не дали URL — сказать пользователю что cover не удалось найти, дальше как при отказе на 6.5.

#### 6.5.2. Имя файла

`<basename>_cover.<ext>`, где:
- `<basename>` — имя файла карточки без `.md` (`Nausicaa_of_the_Valley_of_the_Wind`).
- `<ext>` — расширение из URL (обычно `.jpg`, иногда `.webp` / `.png`). Сохранять оригинальное.

#### 6.5.3. Папка `attachments/`

Проверить через Glob `attachments/`. Если папки нет — создать через Bash:
```bash
mkdir -p attachments
```

#### 6.5.4. Скачать

```bash
curl -L --fail -o "attachments/<filename>" "<cover_url>"
```

Проверка успеха:
- Exit code `curl` равен 0.
- Размер файла > 10 KB (защита от 404-страниц и серверных заглушек). Через Bash: `stat -c %s "attachments/<filename>"` (Linux/Git Bash) или fallback на `ls -la`. Если < 10 KB — удалить и сообщить о неудаче (далее как при отказе).

#### 6.5.5. Frontmatter

В карточке (`ANIME/<basename>.md`) после уже записанного блока полей добавить:
```yaml
images:
  cover: "attachments/<filename>"
```

Размещать после `mal_url`, до закрывающего `---`. Минимальный патч — только добавить ключ `images`, ничего больше не трогать.

#### 6.5.6. Эмбед в теле

Вставить эмбед **сразу после заголовка `# <Title>`, до lead-параграфа**:

```markdown
# <Title>

![[<filename>]]

**<Lead>** — <1-2 предложения>.

## Краткий синопсис
...
```

Это hero-image convention: cover виден первым при открытии заметки в Obsidian, не разбивает прозаический поток нижних секций. Не создавать отдельную секцию `## Галерея` — она будет добавляться будущим gallery-скиллом при наличии дополнительных кадров (screenshots, key visuals) и располагаться ближе к концу карточки.

В отчёте упомянуть имя скачанного файла, размер, источник URL.
