# /remove-gallery-image — Удалить картинку или всю галерею

Удаляет изображения из `## Галерея` карточки. Два режима:

- **`<index>`** — одно изображение по row-major позиции (счёт слева направо, затем построчно вниз). Пустые ячейки не считаются.
- **`all`** (case-insensitive) — все изображения в галерее целиком. Секция `## Галерея` удаляется вместе с таблицей (автогенерируемая, других данных не предполагается).

Сам определяет, нужно ли удалять файлы: если других эмбедов по волту нет — молча (per-file); если есть хотя бы у одного файла — один структурированный multi-choice вопрос с агрегированным списком per-file.

## Аргументы

`$ARGUMENTS` — парсить как `<note> <selector>`:

1. **`<note>`** — имя карточки без `.md` (`Evergarden_Violet`), относительный путь (`CHARACTERS/Evergarden_Violet.md`) или абсолютный путь. Имя без пути — Glob.
2. **`<selector>`** — один из:
   - **`<index>`** — целое ≥ 1, позиция картинки в галерее (row-major). Пустые ячейки не считаются.
   - **`all`** (case-insensitive: `all` / `All` / `ALL`) — удалить всю галерею.

Примеры:
- `/remove-gallery-image CHARACTERS/Evergarden_Violet 3` — удалить 3-ю картинку в галерее
- `/remove-gallery-image Evergarden_Violet 1` — то же по короткому имени
- `/remove-gallery-image ANIME/Demon_Slayer 7` — 7-я картинка в галерее тайтла
- `/remove-gallery-image Evergarden_Violet all` — удалить всю галерею Violet (секция `## Галерея` тоже удаляется)
- `/remove-gallery-image Evergarden_Violet ALL` — то же (case-insensitive)

## Алгоритм

### 1. Резолв заметки

1. Если `<note>` содержит `/` или `\` — нормализовать к относительному пути от корня волта; убедиться, что файл существует.
2. Иначе — Glob `**/<note>.md` от корня волта:
   - Один матч → использовать.
   - Несколько → структурированным multi-choice вопросом выбрать.
   - Ноль → ошибка «Заметка `<note>` не найдена».
3. Прочитать файл, извлечь frontmatter, тело, basename (без `.md`).
4. Если basename содержит shell-проблемные символы (пробелы, кавычки) — продолжать (мы не передаём basename в шелл), но упомянуть в отчёте.

### 2. Найти `## Галерея`

1. Найти секцию `## Галерея` (regex `^##\s+Галерея\s*$`).
2. Нет секции → ошибка: «В заметке нет секции `## Галерея`».
3. Границы секции — от её заголовка до следующего `^##\s+` любого уровня или до конца файла.

### 3. Распарсить таблицу

1. Внутри секции найти **первую** валидную Markdown-таблицу. Признак: последовательность строк, начинающихся с `|`, среди которых есть хотя бы одна строка-разделитель вида `| --- | ... |` (допустимы варианты `|---|`, `|:---:|`).
2. **Парсить только data-ряды** (ряды, идущие ПОСЛЕ разделителя). Header-ряд и delimiter-ряд не считаются и не содержат эмбедов.
3. Для каждого data-ряда — split по `|`, отбросить пустые крайние элементы (типичная split-семантика для таблиц с leading/trailing `|`). Каждая ячейка либо:
   - **Пустая** (whitespace only), либо
   - Содержит embed-синтаксис `![[<file>...]]`. Извлечь filename первым regex `\!\[\[([^\[\]\|\\]+)` (non-greedy по символам, исключающим `[`, `]`, `|`, `\` — это даёт имя файла даже при экранированном pipe `\|`).
4. Счёт row-major: сначала все ячейки 1-й data-строки слева направо, потом 2-й, и т.д. **Пустые ячейки пропускаются** при нумерации.
5. Сохранить два списка:
   - `index_map: [(row, col, filename), ...]` — для проверки индекса и редактирования.
   - `all_files: [filename, ...]` — для rebuild'а (порядок уже row-major).

Если таблица не найдена или в data-рядах нет ни одного `![[...]]` → ошибка: «В `## Галерея` не найдено ни одной картинки».

Если формат таблицы нестандартный (нет разделителя, ячейки в data-рядах без `|`) → ошибка: «Формат таблицы в `## Галерея` не распознан. Запустите /add-images для перевода в канонический формат».

### 4. Резолв `<selector>` → список целей

Парсить `<selector>` (case-insensitive):

- **Если `all`**: `target_files = all_files` (вся галерея), `target_mode = "all"`. Никакой валидации диапазона.

- **Иначе** — парсить как int. Ошибка парсинга → «селектор должен быть целым числом ≥ 1 или ключевым словом `all`».
  - Если `index < 1` или `index > len(index_map)` → ошибка: «В галерее всего N эмбедов, запрошен индекс K (допустимо 1..N)».
  - `target_files = [index_map[index-1].filename]`, `target_mode = "single"`.

### 5. Получить пути файлов

Для каждого filename в `target_files`:
- `target_path = "attachments/<target_filename>"` (по конвенции волта — единая папка `attachments/` в корне, Obsidian-настройка «Attachments folder» указывает на неё, все заметки используют её).
- Проверить существование файла на диске. Если не существует — **сообщить в отчёте**, но не блокировать (файл мог быть удалён вручную; всё равно убираем эмбед).

### 6. Проверить эмбеды в других заметках (для каждого файла)

**Зачем.** Если файл эмбедится где-то ещё (например, как миниатюра персонажа в `## Персонажи` карточки тайтла) — удаление молча сломает ту картинку. Нужно спросить.

**Алгоритм** — для каждого `target_filename` из `target_files`:

1. Grep по всему волту (только `.md` файлы, исключая `.obsidian/`, `.claude/`, `.trash/`) на pattern `\!\[\[<exact_filename>` где `<exact_filename>` — буквальное имя файла, метасимволы regex экранированы (`.` → `\.`).
2. Исключить из результата строки, относящиеся к редактируемой заметке.
3. Оставшийся список — **внешние эмбеды для этого файла**: `[(sourcePath, line), ...]`.

**Агрегировать результат** в `external_embeds_map: {filename: [(sourcePath, line), ...], ...}`. Только файлы с ненулевым списком попадают в map.

### 7. Решение по файлам

**Если `external_embeds_map` пуст (ни у одного файла нет внешних эмбедов)** → `delete_mode = "everywhere"`, молчаливое удаление всех файлов. Пропустить вопрос, перейти к шагу 8.

**Если хотя бы один файл имеет внешние эмбеды** → структурированным multi-choice вопросом:

```
Файлы галереи ещё эмбедятся в других заметках:

attachments/<file_1>.<ext>:
  - ANIME/<X>.md:42
  - CHARACTERS/<Y>.md:71
attachments/<file_2>.<ext>:
  - PERSONS/<Z>.md:15
...

Что сделать?
- Удалить во всех этих заметках + удалить файлы (Recommended)
- Удалить только в этой заметке, оставить файлы
- Ничего не трогать (отменить)
```

Запомнить выбор: `delete_mode ∈ {"everywhere", "only_here", "cancel"}`.

**Один вопрос на все файлы** (не per-file) — пользователь уже выбрал bulk-режим, не дёргаем его N раз. Чистые файлы (без внешних эмбедов) тоже подчиняются этому решению.

### 8. Удаление

**Режим `everywhere`:**

1. Собрать **все** внешние эмбеды из `external_embeds_map` (все файлы, все заметки).
2. Для каждой записи `(sourcePath, line)`:
   - Edit в `sourcePath`: убрать `![[<target_filename>...]]` из ячейки/строки. Минимальный патч:
     - **Внутри таблицы** (ячейка) — заменить ячейку на пустую (например, было `![[file.png]]`, стало пустая строка между `|`). Попутно обновить `updated` во frontmatter этого файла на сегодня.
     - **Вне таблицы** (bullet/list-item) — удалить весь embed-токен с одним trailing пробелом, оставив остальной текст строки. Попутно обновить `updated` в frontmatter.
3. Для каждого файла из `target_files` (независимо от того, были ли у него внешние эмбеды):
   - `rm "attachments/<target_filename>"` (Bash). Если файл уже отсутствует на диске — игнорировать (`rm -f` или проверить `Test-Path` заранее).
4. Перейти к шагу 9.

**Режим `only_here`:**

1. Файлы **не трогать**.
2. Перейти к шагу 9.

**Режим `cancel`:**

1. Прервать. Ничего не менять. Сообщить: «Отменено».

### 9. Перестроить (или удалить) таблицу `## Галерея`

1. Собрать **новый** список эмбедов:
   - `target_mode == "single"`: `all_files` минус `target_files[0]`.
   - `target_mode == "all"`: пустой список (вся галерея удаляется).
2. **Если новый список пуст** (в т.ч. режим `all`):
   - Удалить всю секцию `## Галерея` целиком (вместе с заголовком и trailing-пустой строкой). По конвенции `## Галерея` — автогенерируемая секция, других данных в ней не предполагается.
   - Перейти к шагу 10.
3. **Если не пуст** (только для `single` режима с оставшимися файлами) — перестроить таблицу по канонической форме (`/add-images` шаг 8):
   - `cols = min(6, N)`.
   - Header-ряд: `cols` пустых ячеек.
   - Delimiter-ряд: `cols` ячеек `---`.
   - Data-ряды: эмбеды слева направо, по `cols` в ряд. **Неполный последний ряд добить пустыми ячейками до `cols`.**
   - Каждая ячейка: `![[<filename>\|150]]` — фиксированная ширина **150** с экранированным pipe `\|`, как в `/add-images` шаг 6.6/8. **Bound обязателен**: без `|150` Obsidian рендерит картинку в нативном разрешении (напр. 1920px) → ячейка растягивается на всю ширину заметки и grid рассыпается. Aspect-uniformity (`/add-images` шаг 6.5) этого НЕ заменяет — она выравнивает высоты, а ширину держит именно bound.
4. **Замена всей секции**: один Edit со старой секцией (от `## Галерея` до следующего `##` или конца файла) → новая сборка. Минимальный патч, остальные секции не трогать.

### 10. Самоаудит

1. `vault_lint` (vault-index MCP) с `target` = путь к редактируемой заметке. WARN/ERROR — упомянуть дословно.
2. Grep по редактируемой заметке: для каждого `target_filename` должно вернуть 0 строк (sanity-check, что rebuild прошёл).
3. Если был `everywhere` режим — аналогичный Grep по всем `.md` файлам: должно вернуть 0 строк.

### 11. Отчёт

```
## /remove-gallery-image: <относительный путь заметки>

mode:            <single|all>
index:           <K> / <N>  (только для single)
targets:         <список target_filenames, или N=0 если all на пустой>
delete_mode:     <everywhere|only_here|cancelled>

Файлы:
  • удалено с диска: <M из N> | <список удалённых>
  • с внешними эмбедами: <K файлов> или «нет»
  • пропущено (already missing на диске): <L>

Тело:
  • ## Галерея: <rebuilt (M → M-K эмбедов, cols=C, rows=R) | удалена целиком>

Cross-update (если был, режим everywhere):
  • обновлено эмбедов в K внешних заметках: [список]
  • updated их frontmatter: [список]

Самоаудит (vault_lint):
  <warnings/errors дословно, либо «без замечаний»>
  • Grep после операции: все целевые файлы больше не упоминаются: <yes|no>
```

## КРИТИЧНО

1. **Считать индекс по эмбедам, не по ячейкам.** Пустые ячейки таблицы (появляются в последнем ряду при N не кратном `cols`) не занимают индекс — пользователь думает «3-я картинка», а не «3-я ячейка слева в 1-м ряду». Это базовая usability-гарантия.
2. **Grep-эвристика** ищет только `![[<file>...]]` (embed), не `[[<file>...]]` (WikiLink). WikiLink на картинку не рендерит картинку; сломанный WikiLink — не наша забота, это ad-hoc использование, не конвенция волта.
3. **Empty-after-removal → удалить секцию.** По конвенции `## Галерея` — автогенерируемая секция (только таблица, без другого контента). Пустой header без таблицы — мусор, удалить. Это относится и к режиму `all` (вся таблица удаляется, секция тоже). Другой контент в этой секции — нарушение конвенции, не наш кейс.
4. **Cover vs gallery.** `<basename>_cover.*` живёт в frontmatter (`images.cover`) + hero-эмбед под `# H1`. Gallery-эмбеды имеют суффикс `_gallery_<N>`. Если по индексу выпал файл с суффиксом `_cover` — это либо нарушение конвенции, либо пользователь перепутал. Сообщить и остановиться, **не удалять** (cover ломает куда больше — он ещё и в frontmatter). Для режима `all` — тот же стоп, поскольку gallery-файлы не должны включать cover.
5. **Порядок строк в Markdown-таблице после rebuild'а.** Сохранять row-major — старые картинки остаются в исходном порядке, удалённая просто исчезает, оставшиеся не «перепрыгивают» влево/вверх.
6. **Идемпотентность: повторный вызов на уже удалённый индекс — ошибка**, не тихий no-op. Если картинок стало 4, а пользователь зовёт индекс 5 — сказать «всего 4». Для режима `all` — повторный вызов на уже удалённую/отсутствующую секцию выдаёт «секции `## Галерея` нет» (шаг 2), не тихий no-op.
7. **Multi-choice вопрос — единственный момент для выбора.** Дальше скилл идёт по выбранной ветке. Никаких повторных подтверждений поверх. Для `all` — **один вопрос на все файлы**, не per-file (пользователь уже выбрал bulk-режим, не дёргаем его N раз).
8. **Frontmatter основной заметки не трогать.** Это согласуется с `/add-images` для gallery: операция с автогенерируемой секцией не отражается на review-метаданных. Если хочется бампнуть `updated` — отдельный вызов `/verify`.
9. **Внешние заметки (в режиме `everywhere`) — бампнуть `updated`.** Это согласуется с cross-update в `/add-images`: реальное изменение контента чужой заметки = её `updated` сдвигается. В режиме `all` бампается `updated` всех затронутых внешних заметок (может быть много — собрать уникальный список `sourcePath`, не дублировать если одна заметка затрагивается из-за нескольких файлов).
10. **Режим `all` — ядерная кнопка.** Удаляет всю секцию `## Галерея` и все её файлы (в режиме `everywhere`). Последствия, о которых пользователь должен осознавать до запуска: (a) `## Галерея` исчезнет целиком из заметки; (b) в режиме `everywhere` все эмбеды файлов галереи в чужих заметках будут удалены. Скилл делает **один** финальный multi-choice вопрос, который покрывает обе последствия, и затем идёт безусловно.

## Известные ограничения

- **Только канонический табличный формат.** Старый построчный формат (каждый эмбед на отдельной строке как bullet) — отказ с просьбой перевести через `/add-images` (повторный вызов на те же файлы перестроит секцию в таблицу).
- **Эмбеды вне секции `## Галерея` — вне scope.** Если картинка файла также используется в hero-эмбеде, секции «## Концепт-арт» и т.п. — Grep их найдёт, и пользователю придётся выбрать `everywhere` (рекомендованный вариант сам всё уберёт) или `cancel` и разобраться руками. Скилл не пытается угадывать, какие секции «настоящие» галерейные.
- **Grep может false-positive на похожих именах.** Pattern `\!\[\[<exact_filename>` (с экранированием `.` и других regex-метасимволов в имени файла) минимизирует это, но для уверенности sanity-check на шаге 10 (Grep после операции должен дать 0).
- **Bulk без частичного `everywhere`.** Режим `all` не поддерживает «удалить везде для одних файлов, только здесь для других» — это всегда одна и та же политика для всего списка. Если нужен mixed-режим — несколько вызовов по индексу.
