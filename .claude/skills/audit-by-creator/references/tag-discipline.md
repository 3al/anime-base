# Tag Discipline — правила тегирования + критерий приёма в канон

> **Единый источник правды** для правил тегирования волта. Тема-нейтрально: ни одного доменного хардкода — набор тегов живёт в волте (`SYSTEM/tag_taxonomy.yaml`), правила и критерий — здесь.
>
> На этот файл ссылаются: create-скиллы (`/new-*` — выбор тега из канона + гейт расширения), `audit-by-creator` (`audit_type: structural` — canon-membership, facet-пропагация), `audit-note` (`audit_type: content` — осмысленность тега). Любой скилл, ставящий или судящий теги, обязан следовать **этому** контракту.
>
> Цель — чтобы у каждого скилла **не было своей копии** правил и критерия. Спека одна, синхронизировать нечего. Раньше правила лежали в vault-owned `Tag_taxonomy.md` (тема-нейтральная governance в контент-файле — аномалия); промоутнуты в framework-слой (заказ Anime_Base §29).

Версия контракта: **`"v": 1`**.

## 1. Где что живёт (слои)

| Owns | Что |
| --- | --- |
| **Framework** (этот файл) | правила 1–9, критерий приёма (D), схема канон-файла, sync-контракт yaml→md, severity-политика детекторов |
| **Vault** | набор тегов-контент (`SYSTEM/tag_taxonomy.yaml`, yaml = SSOT) · конфиг-ручки в манифесте (`max_tags`, `tag_facet_fields`, adult-gating) · применение критерия (вердикт «широкая ось / сущность-узкий ярлык») |

**Канон — машиночитаемый.** `SYSTEM/tag_taxonomy.yaml` — SSOT (схема `{name, group, description, applies_to_kinds?, adult_gated?}`). `SYSTEM/Tag_taxonomy.md` — **генерируемый** человекочитаемый вид (правится только yaml; md перерисовывает `core/lib/tag_taxonomy_render.mjs`). Запись тега — структурной записью в yaml, **не** вставкой строки в md-таблицу (убирает источник `broken-table-row` в самой таксономии).

## 2. Правила тегирования (1–9)

1. **kebab-case** — только строчные латинские буквы и дефисы (`[a-z-]+`).
2. **Без нелатинского текста** в тегах. *(Framework-дефолт; волт со своей письменностью может переопределить — но детекторы/имена файлов и WikiLinks завязаны на латиницу.)*
3. **Без имён конкретных сущностей** — имя/синоним сущности кодируется `aliases` во frontmatter и реляционными полями (`featured_in`, `affiliations`), не тегом.
4. **Без команд, кода, предложений** — тег это ярлык-ось, не фраза.
5. **Максимум `max_tags` тегов** на заметку (manifest-ручка `max_tags`, дефолт **10**). Детерминированный бэкстоп — `too-many-tags` (structural-ERROR, `vault_lint`).
6. **Тег описывает содержание**, а не провенанс/статус заметки (провенанс — `co_authored`/`created`; статус — enum-поля).
7. **Тег-имя-сущности не ставится на заметку-источник** — карточка `<Subject>.md` не получает тег `<subject>`; такой тег нужен на **других** заметках, связанных по смыслу (а лучше — это вообще не тег, см. правило 3).
8. **Критерий приёма в канон (D)** — см. §3.
9. **Facet-пропагация** — добавляя тематический тег, проверь **фасет-сиблингов** (карточки, делящие значение `tag_facet_fields`): если ось реальна для кластера, дотегируй сиблингов; если нет — тег, вероятно, сущность-узкий (демоут). Бэкстоп — `under_tag_discord` (`vault_tag_health`, heuristic-кандидат).

## 3. Критерий приёма в канон (D)

**Тег достоин канона ⟺ переиспользуемая ось** (плаузибельно применима за пределами одной сущности) **И** ортогональна enum-полям (`type`/`domain`/`note_kind`/прочие закрытые множества) **И** описывает содержание, не провенанс/статус.

- **Критерий — «переиспользуемость по природе», НЕ порог по счёту.** Порог рубит легитимно-широкие оси, ещё не повторившиеся в малом корпусе (`post-apocalyptic`, `pacifist` могут быть синглтонами, оставаясь широкими). Счётчик (`singleton`/`ghost`) — **слабый сигнал**, ловит лишь **кандидата** на ревизию; вердикт «широкая ось / сущность-узкий ярлык» — суждение по содержанию, его выносит судья/автор, не счётчик.
- Сущность-специфичный ярлык **тегом не становится** — это уже кодируют реляционные поля (`featured_in`/`affiliations`).
- **Расширение канона — через гейт:** прежде чем завести новый тег, примени D. Прошёл → структурная запись в `tag_taxonomy.yaml` + дотегирование facet-сиблингов (правило 9). Не прошёл → не плодить тег.

## 4. Детекторы и severity (раскол structural/heuristic, Roadmap §23)

| Детектор | Где | Класс | Гейтит `structural_green`? |
| --- | --- | --- | --- |
| `non-canonical-tag` | `vault_lint` (per-card) | structural, детерминированный | **пока WARN** — фазовый флип в ERROR-gating (только при `canonSource='yaml'`, см. §5) |
| `too-many-tags` | `vault_lint` (per-card) | structural | да (ERROR) |
| `empty-tags` | `vault_lint` (per-card) | structural | нет (WARN — отсутствие, benign) |
| `ghost` (канон с 0 использований) | `vault_tag_health` (vault-wide) | детерминированный | нет (агрегатная гигиена) |
| `noncanon_summary` | `vault_tag_health` | детерминированный | нет (агрегат per-card WARN) |
| `singletons` (count==1) | `vault_tag_health` | **heuristic-кандидат** | нет |
| `under_tag_discord` | `vault_tag_health` | **heuristic-кандидат** | нет |

Детерминированное (бинарное per-card) и нечёткое (агрегат/кандидат) — в **разных** потоках; нечёткий шум не эродирует доверие к structural-сигналу. Только `non-canonical-tag` — кандидат на гейт (единственное детерминированное per-card тег-правило).

## 5. Admission-гейт: фазовая раскатка `non-canonical-tag` (Q6)

ERROR-gating `non-canonical-tag` — **зубы** admission-гейта (без него правило 8 advisory: слабая модель пропускает «добавь в канон», ничто не краснеет). Но раскатка фазовая, чтобы не сломать волты ложными срабатываниями:

1. **Завести yaml-канон**, засеять, перевести create-скиллы на запись в него.
2. `non-canonical-tag` — **structural-WARN один цикл**: подтвердить ~0 FP на реальной базе (краевые: casing-нормализация, `adult_gated`, conditional-by-kind через `applies_to_kinds`).
3. **Флип в ERROR** отдельным датированным коммитом — **только при `canonSource='yaml'`** (над markdown-fallback гейтить нельзя: ломкий вход → FP). Семантика ERROR: не «тег плохой», а «канон рассинхронен с карточкой — резолвни (зарегистрируй тег по критерию D / сними) до clean».

> Текущий статус: **фаза 2 (WARN)**. Не флипать `non-canonical-tag` в ERROR, пока не пройден WARN-цикл и дата флипа не зафиксирована.

## 6. adult-gating

Тег с `adult_gated: true` в каноне — под adult-гейтом волта. Контракт гейта (где допустимы adult-теги, как они скрываются/фильтруются) — vault-side политика; канон лишь маркирует тег. Аудит не штрафует отсутствие adult-тега на не-adult карточке.
