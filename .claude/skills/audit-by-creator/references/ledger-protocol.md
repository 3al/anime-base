# Ledger Protocol — SSOT записи качества моделей

> **Единый источник правды** для скиллов, которые оценивают качество модели-автора заметки и пишут результат в накопительный леджер.
>
> На этот файл ссылаются: `audit-by-creator` (`audit_type: structural`), `audit-note` (`audit_type: content`). Любой тематический vault-side скилл, который **судит качество модель-генерированной карточки** и хочет попасть в леджер, обязан следовать **тому же** контракту.
>
> Цель документа — чтобы у каждого скилла **не было своей копии** схемы записи, формулы оценки, нормализации ID и порядка операций. Спека одна, синхронизировать нечего.

Версия контракта: **`"v": 1`**. При несовместимой смене схемы — инкремент `v` + раздел миграции внизу.

---

## 1. Зачем леджер

Волт обслуживается параллельно разными моделями (часто слабее референс-судьи, через разные harness). Эмпирически хочется знать, **какие модели дают качественный результат в разрезе `note_kind` и оси аудита**. Авторство видно в `co_authored`; после аудита оценка модели уже выводима из находок → её надо сохранять **попутно**, как побочный эффект обычного вызова аудита.

Это **не** background-таски / hooks / cron. Леджер пополняется при обычном вызове аудита, который пользователь и так гоняет. Карточки, которых аудит не касался, остаются неоценёнными — это приемлемо: покрытие растёт естественно.

## 2. Хранилище

- Файл: **`SYSTEM/model_quality.jsonl`** (JSON Lines — одна запись на строку).
- Режим: **append-only**. Никогда не переписывать существующие строки (история динамики модели во времени).
- Почему JSONL: append-friendly, чистый git-line-diff, тривиальный парс. Файл — **vault-owned данные** (живёт в `SYSTEM/`), но его **схема и писатель** — фреймворк (этот протокол).
- Если файла нет — создать его первой записью.

## 3. Схема записи (`v: 1`)

Одна строка JSONL = один объект. Поля:

| Поле | Тип | Обяз. | Смысл |
| --- | --- | --- | --- |
| `v` | int | да | Версия схемы. Сейчас `1`. |
| `ts` | string (ISO-8601 UTC) | да | Момент оценки, напр. `2026-06-03T14:20:00Z`. |
| `note` | string | да | Путь карточки от корня волта, напр. `ANIME/Maria-sama_ga_Miteru.md`. |
| `note_kind` | string | да | `note_kind` карточки. |
| `audit_type` | string | да | Ось оценки: `structural` (audit-by-creator) / `content` (audit-note) / др. **Без него агрегаты смешают разные оси.** |
| `authored_model` | string | да | **Нормализованный** (§3.1) ID модели-автора = origin-автор из `co_authored` на входе. **Субъект оценки.** |
| `authored_model_raw` | string | да | Сырой ID автора как в `co_authored` (до нормализации) — для аудируемости нормализации. |
| `judge_model` | string | да | **Нормализованный** ID модели-судьи (текущая модель). Снимает single-judge-проблему на уровне запроса. |
| `score` | int 1–10 | да | Оценка (см. §4). |
| `max` | int | да | Потолок шкалы, всегда `10`. |
| `invariants_total` | int | да | Сколько инвариантов/проверок сверено (нормализация между карточками). |
| `findings` | object | да | Счётчики: `{ "ok": int, "missing": int, "wrong_format": int, "needs_external": int, "schema_drift": int }`. `schema_drift` — реклассифицированные epoch'ом (§4.1), в score не входят. |
| `reasons` | string[] | да | Короткий человекочитаемый список главных дефектов. Пустой `[]` если дефектов нет. |
| `content_hash` | string | да | Хэш контента карточки **на входе** (до правок аудита) — для дедупа (§5). |
| `fixes_applied` | bool | да | Применил ли **этот** прогон механические правки к карточке (`false` при `--report-only` / нет фиксабельного). |
| `pristine` | bool | да | Была ли карточка **свободна от любых прежних judge-фиксов** в момент этого замера (§5). Вычисляется. Вью сравнения моделей фильтрует `pristine=true`. |

Пример строки:

```json
{"v":1,"ts":"2026-06-03T14:20:00Z","note":"ANIME/Maria-sama_ga_Miteru.md","note_kind":"anime","audit_type":"structural","authored_model":"glm-5.1","authored_model_raw":"zai-coding-plan/glm-5.1","judge_model":"claude-opus-4.8","score":6,"max":10,"invariants_total":24,"findings":{"ok":18,"missing":2,"wrong_format":3,"needs_external":1,"schema_drift":1},"reasons":["нет ## Резонанс при status:finished","каст bullet вместо table"],"content_hash":"a1b2c3d4","fixes_applied":true,"pristine":true}
```

Поля писать в стабильном порядке (как в таблице) — облегчает чтение git-diff.

### 3.1 Нормализация ID модели (`authored_model`, `judge_model`)

**Зачем:** одна модель = один ключ агрегации. ID слабых моделей в `co_authored` несогласованы (`minimax-m3-free` vs `minimax-m3.free`; `glm-5.1` vs `zai-coding-plan/glm-5.1`) → без канонизации одна модель расколется на несколько ключей. Волтовая конвенция `claude-<family>-<major>.<minor>` покрывает только Claude.

Канонизировать **перед записью** (общий алгоритм, не хардкод-маппинг):

1. **Lowercase + trim.**
2. **Снять provider/namespace-префикс:** если в ID есть `/`, оставить только сегмент после последнего `/` (`zai-coding-plan/glm-5.1` → `glm-5.1`).
3. **Снять access-tier суффикс:** финальный маркер доступа из набора `{free, preview, beta}`, отделённый любым из `-`/`.`/`:`, отбросить (`minimax-m3-free` / `minimax-m3.free` / `minimax-m3:free` → `minimax-m3`). Набор маркеров **расширяем под волт** (если у волта есть свои tier-суффиксы — дополнить здесь).
4. Claude-ID уже канонические (`claude-opus-4.8`) — шаги 2–3 для них no-op.

Результат → `authored_model` / `judge_model`. Сырое значение `co_authored` сохранить в `authored_model_raw` (нормализацию можно перепроверить/откатить без потери данных).

> Нормализация — на стороне **писателя леджера** (этот протокол), не на простановке `co_authored`. Так `co_authored` остаётся человекочитаемым origin, а канонизируется только ключ агрегации, единообразно для всех скиллов-писателей.

## 4. Формула оценки (штраф от базы)

Оценка должна быть **воспроизводима у одного судьи** → якорится к находкам, не к «ощущению» модели. LLM-судья формирует `reasons`, но **число выводится из находок** механически.

1. База: `score = 10`.
2. За каждый `MISSING` обязательного инварианта: `score -= 2` (тяжёлый штраф — отсутствует требуемый артефакт).
3. За каждый `WRONG_FORMAT` (артефакт есть, но не по спеке): `score -= 1` (лёгкий штраф — содержание на месте, форма нарушена).
4. За каждый `NEEDS_NETWORK_OR_VISION`: **штрафа нет** — не вина автора.
5. Пол: `score = max(score, 1)`. Потолок `10`.

`score = clamp(10 − 2·missing − 1·wrong_format, 1, 10)`.

`SCHEMA_DRIFT` (§4.1) и `NEEDS_NETWORK_OR_VISION` в score **не входят** — это не дефекты автора.

> **Веса фиксированы здесь.** Тюнить — только тут, синхронно для всех скиллов; при существенной смене — инкремент `v`. `invariants_total` пишется отдельно для нормализации (карточка с 8 и с 30 инвариантами несёт разный вес одного дефекта).

### 4.1 Epoch-conformance: балл за качество, не за дрейф схемы

**Проблема.** Аудит судит карточку **текущей** спекой создателя, но `score` приписывается **автору-на-момент-создания** (`co_authored`, дата `created`). Если требование введено в спеку **после** `created`, автор физически не мог его проставить → штраф за пост-авторскую эволюцию схемы, не за качество. Смещение системное и злое: старые карточки = сильные модели (baseline), свежие = слабые → schema-drift занижает baseline, слабые на его фоне выглядят лучше — ровно то, что леджер должен мерить честно.

**Правило.** Перед скорингом **реклассифицировать** каждую находку `MISSING`/`WRONG_FORMAT`, чьё требование введено в спеку **позже** даты `created` карточки, в класс **`SCHEMA_DRIFT`**:
- `SCHEMA_DRIFT` **в score не входит** (как `NEEDS_EXTERNAL`) — это **vault-долг** (карточку стоит дотянуть до текущей спеки), но не вина автора. Репортится отдельно, может авто-фикситься как backfill.
- **Двусторонность:** требование, существовавшее **до** `created` и отсутствующее → честный author-defect (штрафуется). Свежая карточка слабой модели без поля, введённого раньше её `created`, → штрафуется. Epoch отсекает **только** несуществовавшее на момент авторства.

**Источник дат — `SYSTEM/spec_changelog.yaml`** (vault-owned контент, framework-defined схема, читается как `enums.yaml`):

```yaml
# SYSTEM/spec_changelog.yaml — когда введено каждое требование create-спеки.
# Контент — vault-side; схему задаёт фреймворк (ledger-protocol §4.1).
v: 1
kinds:
  anime:
    - requirement: online_url       # frontmatter-поле ИЛИ заголовок секции ("## Резонанс") ИЛИ стабильный label инварианта
      introduced: 2026-06-02         # YYYY-MM-DD — дата ввода требования в спеку
      kind_of: field                 # field | section | format (информативно)
  character:
    - requirement: "## В ранобэ"
      introduced: 2026-05-10
      kind_of: section
```

**Матчинг и дефолты:**
- Находку матчить к записи changelog по `requirement` = имя frontmatter-поля / точный заголовок секции / label инварианта.
- **Нет записи в changelog для требования** → считать его **пред-существовавшим** (introduced = −∞) → штрафуется как author-defect. Полнота changelog обеспечивается тремя механизмами:
  1. **spec-requirements манифест** в каждом `/new-<kind>/SKILL.md` — машиночитаемый managed-блок (`<!-- BEGIN: spec-requirements -->` … `END`), декларирующий обязательные требования kind (`{requirement, kind_of}`). Генерируется `/add-note-kind` из того же списка, что и записи changelog (1:1 by construction).
  2. **Детерминированный backstop** — tool `vault_spec_drift` (vault-index MCP): диффит каждый манифест против changelog, флагает `manifest-requirement-without-changelog` (требование объявлено, но не залогировано) и `missing-manifest` (create-скилл без блока — кандидат на бэкфилл). Harness-нейтрален, работает в CC и Opencode идентично. Это заменяет прежний статистический «белая ворона»-эвристик (он не ловил format-правила, живущие прозой).
  3. **Live-reminder при ручной правке** — в Claude Code PostToolUse-hook (harness-claude-code) при правке spec-requirements блока напоминает синхронизировать changelog тем же изменением. В Opencode live-reminder недоступен (upstream [#13574](https://github.com/anomalyco/opencode/issues/13574)) → там инвариант держит только backstop (#2).
- **Бэкфилл существующих create-скиллов:** на волтах, где `/new-*` скиллы появились до spec-requirements контракта, прогнать `vault_spec_drift` → находки `missing-manifest` перечислят скиллы без блока. Для каждого: извлечь обязательные требования из прозы скилла (обязательные поля + секции + format-правила), дописать managed-блок, сверить с существующими записями `kinds.<slug>` в changelog (добавить недостающие записи **исторической** датой ввода требования, не «сегодня», если требование реально вводилось раньше). Повторить `vault_spec_drift` до нуля ERROR.
- **У карточки нет/битый `created`** → epoch установить нельзя → реклассификацию не применять, в отчёте пометить «epoch unverifiable (no created)».
- **Новый волт** (нет `spec_changelog.yaml`) → дрейфа нет, реклассифицировать нечего → всё author-defect (корректно). Файл не обязателен; его отсутствие = пустой changelog.

## 5. Дедуп, контаминация, история

Леджер — append-history (видна динамика модели), с двумя защитами:

**A. Дедуп точных дублей.** Перед записью прочитать леджер. Если уже есть запись с тем же `note` **и** `judge_model` **и** `audit_type` **и** совпадающим `content_hash` → **пропустить** (тот же судья по той же оси уже оценил это самое состояние карточки; новая строка ничего не добавит). Если `content_hash` отличается (карточку правили) → писать новую строку.

**B. Контаминация judge-фиксами → флаг `pristine`.** Корень: судья, починив карточку, делает её состояние частично **своей** работой; последующий замер этого состояния нельзя честно записать на origin-автора (иначе автор получит балл за чужой ремонт — и в той же оси на ре-ране, и в **другой** оси, если первый аудит правил карточку до замера второй оси).

Решение — не выбрасывать запись, а **помечать**:
- `fixes_applied` = применил ли **этот** прогон правки.
- `pristine` вычисляется на записи: `true` ⟺ в леджере **нет более ранней записи по этой `note` c `fixes_applied=true`** (любой `audit_type` — контаминация кросс-осевая: первый же фиксящий аудит делает карточку non-pristine для всех последующих замеров любой оси).
- Собственный фикс текущего прогона на его же `pristine` **не влияет** (замер делается ДО фиксов — см. §6 порядок).

Так история сохраняется целиком, а **вью сравнения моделей фильтрует `pristine=true`** — в сравнение качества попадает только «чистая» работа автора, не judge-улучшенная.

**Вью** (отложенные, считаются на запросе, не часть контракта):
- «качество моделей» — `avg(score)` по `authored_model × note_kind × audit_type`, фильтр **`pristine=true` И `judge_model = <референс>`** (§7).
- «текущее состояние» — последняя запись по `(note, judge_model, audit_type)`.
- «тренд» — вся история по ключу (включая non-pristine).

## 6. Авторство: аудит НЕ авторствует (фикс кросс-ран потери субъекта)

**Принцип.** Скиллы делятся по тому, **производят** они контент или **ревьюят/оценивают** его:

| Класс | Скиллы | `co_authored` |
| --- | --- | --- |
| **Производят контент** | `/new-*`, `/expand-stub`, `/vault-rag` | **ставят** `co_authored` (они авторы сути) |
| **Ревьюят/оценивают** | `/audit-note`, `/audit-by-creator`, `/audit-review` | **НЕ трогают** `co_authored` (они судьи, не авторы) |

**Почему так (а не «co_authored = judge последним»).** Перезапись `co_authored` на судью **уничтожает субъект оценки** для всех последующих аудитов: пример — `/audit-note` починил факты и затёр `co_authored = opus`; затем `/audit-by-creator` читает `co_authored = opus` и записывает структурные косяки слабой модели **на судью**. Capture-at-entry это **не** спасает — он живёт один прогон. Единственное надёжное решение: **аудит не перезаписывает `co_authored` никогда**, оригинал-автор переживает любую последовательность аудитов. След судьи живёт в `ledger.judge_model`, а не в `co_authored`. Правка Opus'ом косяков glm **не делает Opus автором** карточки.

**Порядок операций аудита-писателя леджера:**

1. **На входе** — прочитать `co_authored` (origin-автор, субъект оценки) и `content_hash` по контенту на входе. `co_authored` далее **не модифицируется** этим скиллом.
2. Выполнить аудит (структурный / контентный / …).
3. Вычислить `score` из находок (§4).
4. Вычислить `pristine` и `fixes_applied` (§5), нормализовать ID (§3.1).
5. **Аппендить запись в леджер** (§2–§5): `authored_model` = нормализованный origin-автор, `judge_model` = нормализованная текущая модель, `audit_type` = ось скилла. Соблюсти дедуп (§5.A). Пропустить при опции «без леджера».
6. Применить правки (если политика скилла разрешает и правки есть).
7. **Метаданные ревизии — НЕ авторство.** Если на шаге 6 были правки: `quality: draft` (пользователь подтвердит через `/verify`), `updated: <сегодня>`. **`co_authored` не трогать.** Правок не было → frontmatter не менять вовсе.

> **Серая зона — дописывание контента в аудите** (напр. `/audit-note §4.4` «дописать секции»): даже там `co_authored` остаётся origin, а правка ставит `fixes_applied=true` → последующие замеры этой карточки становятся `pristine=false` и не засчитываются автору. Если карточке нужно **настоящее** авторское дописывание (тонкий стаб) — это путь `/expand-stub` (он производит контент и ставит `co_authored`), не аудит.

## 7. Консистентность судьи (без инфраструктуры)

Сравнение моделей (§1) осмысленно только при **консистентном судье** + **чистых записях**:

- Каждая запись несёт `judge_model` (нормализованный); вью «качество моделей» фильтрует по референс-судье (`judge_model = claude-opus-*`) **и** `pristine=true`.
- Практически судья и так референс: скиллы-писатели декларируют `model: opus` + пользователь гоняет аудит под Opus. Записи под слабой моделью / non-pristine просто отфильтруются на запросе, не портя датасет.

**Single-judge-bias оговорка:** один судья даёт **относительное** сравнение моделей-авторов (кто лучше относительно одной планки), не абсолютную истину. Для цели леджера этого достаточно. Грейдинг самого референса как baseline допустим (записи с `authored_model == judge_model`).

## История версий

- `v: 1` — первичная схема. Поля §3 (вкл. `authored_model_raw`, `fixes_applied`, `pristine`, `findings.schema_drift`), нормализация ID §3.1, формула §4, epoch-conformance §4.1, дедуп+контаминация §5, «аудит не авторствует» §6. Дизайн закрывает кросс-ран потерю субъекта (A), кросс-осевую контаминацию (B/pristine) и schema-drift мисатрибуцию (epoch). `schema_drift`-счётчик + `spec_changelog.yaml` добавлены аддитивно (леджер держался на паузе до epoch-фикса → миграции существующих записей не требуется).
