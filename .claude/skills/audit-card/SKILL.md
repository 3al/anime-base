# /audit-card — Полный аудит карточки по обеим осям (оркестратор)

Гоняет на одной карточке **обе** оси леджера качества — `structural` (`/audit-by-creator`) и `content` (`/audit-note`) — так, чтобы **каждая** ось записала **чистый** (`pristine=true`) замер качества модели-автора, после чего применяет фиксы по обеим осям и закрывает карточку.

**Зачем нужен.** При наивном порядке «один аудит фиксит → второй видит `fixes_applied=true` → `pristine=false`» вторая ось систематически не попадает в сравнение моделей (`ledger-protocol.md §5.B`, кросс-осевая контаминация). Особенно больно при детерминированном workflow «по бэклогу всегда обе оси, structural первой» — content-ось набирает ноль pristine-данных. Эта обёртка снимает проблему **порядком вызовов с `--report-only`**, без правки managed-слоя.

**Это тонкий оркестратор.** Сам он не аудитит — всю работу делают два managed-скилла. Любое их ужесточение (новые инварианты, смысловой фактчек, lint-бэкстопы) обёртка наследует автоматически. Это **мост**: когда фреймворк сольёт обе оси в один проход, тело §«Алгоритм» схлопнется в один вызов.

## Аргумент

`$ARGUMENTS` = `<note> [--no-images]`.

- `<note>` — имя карточки без `.md`, относительный путь от корня волта, или абсолютный путь.
- `--no-images` / `-n` — проброс в `/audit-by-creator` (пропустить визуальный под-чек постера). На content-ось не влияет.

Примеры:
- `/audit-card CHARACTERS/Marlborough_Luculia`
- `/audit-card Hodgins_Claudia`
- `/audit-card CHARACTERS/Dietfried_Bougainvillea --no-images`

## Контракт, на который опирается обёртка (ledger-protocol)

Стабильные инварианты managed-скиллов — обёртка их **только использует**, не дублирует:

1. `--report-only` — **пишет ряд в леджер**, но **не мутирует** карточку (`audit-by-creator/SKILL.md`, `audit-note/SKILL.md`).
2. `--no-ledger` — выполняет аудит/фиксы **без** записи ряда.
3. Порядок каждого скилла: замер → запись в леджер → фикс (`ledger-protocol.md §6`). Замер всегда по контенту **на входе**.
4. `pristine=true` ⟺ в леджере нет более раннего ряда по этой `note` c `fixes_applied=true` (любая ось) (`ledger-protocol.md §5.B`).
5. Дедуп (`§5.A`): ряд пропускается, если совпали `note` + `judge_model` + `audit_type` + `content_hash`.

## Алгоритм

### 1. Резолв заметки

1. Снять флаг `--no-images`/`-n` из `$ARGUMENTS`, остаток = `<note>`.
2. Если `<note>` содержит `/` или `\` — нормализовать к относительному пути; убедиться, что файл существует.
3. Иначе — Glob `**/<note>.md`:
   - Один матч → использовать.
   - Несколько → структурированным multi-choice вопросом дать выбрать.
   - Ноль → ошибка.
4. Зафиксировать **резолвнутый путь** `P` — дальше передавать всем под-вызовам **именно его** (чтобы каждый под-скилл не переспрашивал резолв).

### 2. Предусловия

1. Прочитать frontmatter `P`: `note_kind`, `co_authored`, `created`.
2. **Structural-ось применима**, только если для `note_kind` существует создающий скилл `/new-<kind>` (его читает `/audit-by-creator`). Если создателя нет (например, системная заметка) — structural-ось пропустить, прогнать **только** content (шаг 4) и отметить в отчёте «structural N/A — нет /new-<kind>».
3. Запомнить факт: **изменилась ли карточка за прогон** (для флага края, шаг 6) — сравнивать контент `P` до шага 3 и после шага 5.

### 3. Ось A — structural-замер (report-only, без мутации)

Запустить **`/audit-by-creator --report-only P`** (+ `--no-images`, если задан).

- Пишет **row A**: `audit_type=structural`, `pristine=true`, `fixes_applied=false`. Это **чистый замер** structural-качества автора на нетронутом оригинале.
- Карточка **не меняется**.
- Удержать в контексте его **список механических фиксов** и находки — переиспользуются на шаге 5 (не передеривировать с нуля).

### 4. Ось B — content-замер + починка контента

Запустить **`/audit-note P`** (дефолт, без флагов).

- К этому моменту в леджере только row A (`fixes_applied=false`) → content-замер выходит **`pristine=true`**.
- Скилл чинит контент (проза/факты/язык/теги) и пишет **content-row**.
- В обычном случае (карточка битая по контенту) content-row несёт `fixes_applied=true` — это и есть маркер контаминации для будущих ре-аудитов.

### 5. Ось A — применить structural-фиксы (без второго ряда)

Запустить **`/audit-by-creator --no-ledger P`** (дефолт-режим минус леджер).

- Скилл ре-деривирует structural по **текущему** (уже контент-починенному) состоянию `P` и применяет безопасную механику. Ре-деривация против реального состояния **снимает риск stale line-edit** (камень 1): фиксы строятся по факту, а не по оригиналу шага 3.
- `--no-ledger` → второй structural-ряд **не пишется** (row A остаётся единственным structural-замером, и он чистый).
- Наследует fix-policy скилла (включая fix-scoping §22, когда приедет): чужие со-локализованные дефекты он флагнёт как vault-долг, а не починит.

### 6. Проверка остаточного края (контаминация-guard)

Один квадрант ломает guard: **контент был чист (шаг 4 ничего не починил, `content-row.fixes_applied=false`), а структура битая (шаг 5 применил фиксы)**. Тогда дедуп (`§5.A`) душит ряд шага 5 (хеш входа == row A), и в леджере **нет** ни одного `fixes_applied=true` ряда, хотя карточка судьёй изменена.

- Детект: `карточка изменилась за прогон` **И** в леджере по `P` нет ни одного ряда с `fixes_applied=true`.
- Если сработало → **флаг в отчёте** (не авто-маркер): «⚠ Структура починена при чистом контенте; контаминация не записана в леджер. При будущем ре-аудите этой карточки выставить `pristine=false` вручную» (та же дисциплина, что [[feedback_audit_fixes_scoped_to_subject_author]] для Marlborough).
- Если не сработало (обычный случай) — ничего не делать.

### 7. Самоаудит и отчёт

1. Прогнать `vault_lint` (vault-index MCP, tool `vault_lint(target=P)`). Issues — дословно в отчёт.
2. Сводный отчёт:

```
## /audit-card: <относительный путь>

Structural (audit-by-creator):
  • замер: score N/10 (pristine=true, row A)   [или: N/A — нет /new-<kind>]
  • фиксы: <что применено на шаге 5>            [или: без правок]

Content (audit-note):
  • замер: score M/10 (pristine=<true|false>)
  • фиксы: <что применено на шаге 4>            [или: без правок]

Леджер:
  • structural row (pristine=true) + content row (pristine=<...>)
  • <флаг края, если сработал шаг 6>

Самоаудит (vault_lint):
  <issues дословно, либо «без замечаний»>
```

## КРИТИЧНО

1. **Тонкий оркестратор — не аудитить руками.** Всю проверку/починку делают `/audit-by-creator` и `/audit-note`. Обёртка только задаёт порядок и флаги. Не дублировать их логику фиксов (иначе разойдётся с §22–24 при апдейте managed-слоя).
2. **Порядок строгий: A(report-only) → B(content) → A(no-ledger fix).** Он гарантирует оба `pristine=true`. Менять порядок нельзя — если structural починит до content-замера, content-ось станет `pristine=false` (ровно проблема, которую обёртка решает).
3. **`--report-only` на шаге 3 обязателен.** Без него шаг 3 замутит карточку → ось B потеряет pristine.
4. **Шаг 5 — `--no-ledger`.** Иначе второй structural-ряд (`pristine=false`) засорит леджер; чистый замер уже взят на шаге 3.
5. **Резолвить заметку один раз** (шаг 1), передавать резолвнутый путь под-вызовам — иначе тройной multi-choice на неоднозначном имени.
6. **`co_authored` никто не трогает** — оба под-скилла судьи, не авторы (`ledger-protocol.md §6`). Обёртка тем более.
7. **Это мост.** Когда фреймворк сольёт обе оси в один проход (Roadmap §17, вариант B) — заменить тело §«Алгоритм» (шаги 3–6) на один вызов слитого скилла. Остаточный край (шаг 6) тогда исчезнет by construction.
