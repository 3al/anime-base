# /new-person — Создание карточки персоны

Создаёт карточку человека из аниме-индустрии (режиссёр, сценарист, композитор, продюсер, аниматор, mangaka и т.д.) с заполнением полей frontmatter по `note_kind: person` и каркасом тела заметки.

**Сэйю — этот же kind.** Голосовые актёры — `note_kind: person` с `roles: [voice-actor]`. Отдельного kind для них нет. Особенности: записи в `works[]` обязательно содержат поле `character` (foreign-key в `CHARACTERS/`), потому что VA-роль определяется не только тайтлом, но и конкретным персонажем; зеркальная запись поддерживается в `character.voice_actors[]`. См. шаг 7.5b.

## Аргумент

`$ARGUMENTS` — имя человека на русском, английском, ромадзи или в родной письменности.

Примеры:
- `/new-person "Hayao Miyazaki"`
- `/new-person Хаяо Миядзаки`
- `/new-person 宮崎駿`

## Источники правил

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек Персон (`note_kind: person`)». Полный список обязательных и опциональных полей + семантика.
2. **`SYSTEM/enums.yaml`** — раздел `note_kinds.person`. Машинно-читаемый источник допустимых значений (`status`, `role`). **Не подставлять значения, отсутствующие в enums.yaml.**
3. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: person`. Какие WikiLinks обязательны.
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Glob по `PERSONS/*.md` — поискать совпадение по `name_primary` и `aliases` (если карточка с этим человеком уже есть под другим именем). Если найдено:
- **Stub (<20 строк)** — не создавать с нуля, дополнить.
- **Наполненная (>50 строк)** — сообщить пользователю.

### 2. Сбор данных

Web search → найти актуальные сведения. Приоритетные источники:
- **MyAnimeList** (`https://myanimelist.net/people/`) — основные роли, биография.
- **AniList** (`https://anilist.co/staff/`) — структурированные роли, works.
- **Wikipedia** — биография, даты, культурный контекст.
- **Anime News Network** (`https://www.animenewsnetwork.com/encyclopedia/people.php`) — энциклопедический справочник, оригинальные имена кандзи.

**КРИТИЧНО про числовые/датовые поля** (`birth_year`, `birth_date`, `death_date`). Брать **только из прямого web fetch конкретной страницы**, не из summary поисковика. Если в источнике нет — оставить пустым (поле опциональное), не угадывать. См. также `~/.claude/projects/D--Anime-Base/memory/feedback_numeric_facts_from_fetch.md`.

### 3. Определить имя файла

PascalCase с подчёркиваниями вместо пробелов, латиница. По умолчанию — `name_primary` (для японцев — ромадзи) с заменой пробелов на `_`. Примеры: `Hayao_Miyazaki.md`, `Joe_Hisaishi.md`, `Isao_Takahata.md`.

Кириллица, кандзи и иероглифы — в `name_native` и `aliases[]`, **не в имя файла**.

### 4. Собрать frontmatter

**Прежде чем подставлять значения** enum-полей (`status`, `roles[]`) — прочитать `SYSTEM/enums.yaml::note_kinds.person` и проверить каждое. Если из web-research пришла роль вне enum (например, `editor-in-chief`) — структурированным multi-choice вопросом спросить пользователя: Добавить / Пропустить / Заменить на существующее. При «Добавить» — Edit `enums.yaml`, append в `note_kinds.person.role` (или в anchor `&production_role` — он шарится с `anime.staff_role`). Упомянуть факт расширения enum'а в отчёте отдельной строкой.

Заполнить ВСЕ обязательные поля:

```yaml
---
type: reference
domain: anime
stability: stable
priority: medium
quality: draft
note_kind: person
tags:
  - <теги по Tag_taxonomy>
aliases:
  - <альтернативные написания>
created: <сегодня>
updated: <сегодня>
name_primary: "<Основное имя (ромадзи для японцев / латиница как есть для западных)>"
name_native: "<Имя в родной письменности>"
roles:
  - <значение из enums.yaml::note_kinds.person.role>
country: "<Страна>"
birth_year: <YYYY>
status: <alive|deceased>
# Рекомендуемые — ВСЕГДА присутствуют во frontmatter (политика «always-present»).
# Если данных нет — null / "" / []. НЕ опускать поле целиком.
birth_date: <YYYY-MM-DD или null>
death_date: <YYYY-MM-DD или null>     # только если status: deceased, иначе null
birth_place: "<Город, страна или ''>"
mal_url: "<URL или ''>"
anilist_url: "<URL или ''>"
wikipedia_url: "<URL или ''>"
# works заполняется автоматически — /new-anime cross-update шаг 7.5 дописывает
# при создании аниме со staff[]. При создании person с нуля можно оставить []
# или заполнить вручную из web research (формат как в Metadata_schema).
works: []
---
```

**Соглашения по пустым значениям:** см. аналогичный блок в `/new-anime` SKILL.md шаг 4. Кратко: строки → `""`, числа/даты → `null`, списки → `[]`, enum → `null`.

**Ставить `co_authored: <model-id>`** при создании. ID текущей модели в волтовом формате `claude-<family>-<major>.<minor>` (например `claude-opus-4.7`, `claude-sonnet-4.6`, `claude-haiku-4.5`); источник — системный промпт, строка «The exact model ID is …» (последний дефис → точка). См. `feedback_co_authored`.

**Теги.** Подбирать пошагово (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня
2. ТЕМЫ — стиль, эпоха (если применимо: `golden-age-anime`, `studio-ghibli-era`)
3. **НЕГАТИВНАЯ ПРОВЕРКА** — убрать теги, дублирующие `roles`/`country`/`note_kind`. Имя человека НЕ ставится тегом (правило 7 из Tag_taxonomy).
4. **ПРОБЕЛЫ в таксономии** — обязательно добавить новые теги в `SYSTEM/Tag_taxonomy.md`.

Person без обязательных тегов — норма (kind не требует, в отличие от `anime`).

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <name_primary> (<name_native>)

**<Имя на русском или ромадзи>** — <1-2 предложения: основные роли, страна, чем известен>.

## Биография

<Краткая биография: рождение, ключевые вехи карьеры, важные коллаборации. 2-4 абзаца. Без воды.>

## Стиль и темы

<Творческий стиль, повторяющиеся темы, узнаваемые приёмы. Опционально — если про человека есть что сказать концептуально.>

## Студии

<Опциональная секция — добавлять только если человек работал в студиях, у которых есть карточки в `STUDIOS/`. Gallery-формат, ширина 60 (по дефолту конвенции для студий). Источник — `vault_backlinks(note="<thisPerson>")` отфильтрованный по `sourcePath: STUDIOS/*` (студии, в чьих картах человек упомянут с WikiLink в секции «## Основатели и ключевые люди»).>

```
- ![[<Studio_File>_cover.<ext>|60]] **[[<Studio_File>|<Имя>]]** — <роль / период / контекст>.
```

<Сортировка хронологическая (от ранних мест работы к поздним). При пустом наборе backlinks секцию опустить полностью.>

## Ключевые работы

<Список ключевых работ в gallery-формате (см. `Linking_guidelines.md` секция «Gallery-стиль для backlink-списков»), ширина 80. Записи дублируют `works[]` из frontmatter — здесь развёрнуто и визуально, там структурно. Сортировка хронологическая (по году релиза).>

```
- ![[<Anime_File>_cover.<ext>|80]] **[[<Anime_File>|<Название>]]** (<год>) — <роль(и) этого человека>, <одна строка контекста: значимость, награды, коллаборации>.
```

<Правила:
- Если карточка тайтла **есть и `images.cover` заполнен** → миниатюра + bold-WikiLink + описание.
- Если карточка **есть, без `images.cover`** → только bold-WikiLink + описание (без миниатюры).
- Если карточки **нет** → plain bold-имя + описание (без ссылки).

Для каждого WikiLink — Read целевой карточки → парс frontmatter → проверить `images.cover`.>

## Ссылки

<Внешние ссылки: MAL, AniList, Wikipedia, официальный сайт/блог, при наличии.>
```

Размер: 40–150 строк в зависимости от значимости персоны.

### 6. Проверить ссылки и собрать gallery-формат

Обязательные WikiLinks для `note_kind: person`:
- Каждый `works[].title` → карточка в `ANIME/` + запись в gallery-секции «## Ключевые работы».
- Каждая `STUDIOS/<S>.md` с этим персонажем в `founders[]` или с WikiLink на него в `## Основатели и ключевые люди` → запись в gallery-секции «## Студии» (если такие студии существуют). Сборка через `vault_backlinks(note="<thisPerson>")` + фильтр по `sourcePath` начинается с `STUDIOS/`.
- `birth_place` (если есть карточка локации) — упомянуть в теле, обычно в секции «## Биография» (не gallery-формат, это одиночная контекстная ссылка).

**Gallery-формат для `## Ключевые работы`** — см. шаблон в шаге 5 и полные правила в `Linking_guidelines.md` секция «Gallery-стиль для backlink-списков». Для каждого WikiLink-цели — Read frontmatter, посмотреть `images.cover`, выбрать один из трёх форматов (миниатюра+линк / линк / plain bold).

**Принцип единственного канала.** Если тайтл присутствует в gallery `## Ключевые работы`, его упоминания в прозе биографии остаются **plain text**, даже если карточка существует. Gallery-секция = единственное навигационное место. Симметрично правилам в `/new-anime` и `/new-character`.

Проверки:
- Все обязательные WikiLinks присутствуют (если соответствующие заметки в волте).
- Нет дубликатов (`vault_duplicate_links`).
- Все target существуют (Glob или `vault_query`).
- Все embed'ы миниатюр резолвятся в реальные файлы в `attachments/`.

### 6.5. Постер / фото (интерактивно)

Опционально — спросить пользователя структурированным multi-choice вопросом: «Добавить фотографию персоны?»
- **Да** — попросить локальный путь к файлу, выполнить `/add-images <person> poster <path>` (или прямо повторить его логику). Постер фотографии — обычно портрет → ширина 300.
- **Нет** — пропустить. Можно добавить позже через `/add-images`.

### 7. Создать файл

Записать в `PERSONS/<имя_файла>.md`.

### 7.5. Обновить связанные карточки (production-роли)

Разбить записи `works[]` на две группы по `roles`:
- **Production-роли** (всё кроме `voice-actor`) → шаг 7.5a, апдейт `ANIME/<title>.md`.
- **VA-роли** (`voice-actor`, с обязательным `character`) → шаг 7.5b, апдейт `CHARACTERS/<character>.md`.

#### 7.5a. Production-cross-update

Для каждой production-записи в `works[]`, у которой есть карточка в `ANIME/`:

1. Прочитать `ANIME/<title>.md`.
2. Проверить `staff[]` — есть ли запись `{person: <этот человек>, role: <r>}` для каждой роли из `works[*].roles` (исключая `voice-actor`). Если **отсутствует** — добавить во frontmatter.
3. В секции «## Студия и команда» добавить запись в gallery-формате (если её ещё нет; см. шаг 6). Текущая person-карточка может или не может иметь `images.cover`:
   - **Есть** (например, был передан портрет через шаг 6.5):
     ```markdown
     - ![[<thisPerson>_cover.<ext>|60]] **[[<thisPerson>|<Имя>]]** — <роль(и) в этом тайтле>, <одна строка контекста>.
     ```
   - **Нет** — gallery-формат без миниатюры:
     ```markdown
     - **[[<thisPerson>|<Имя>]]** — <роль(и)>, <контекст>.
     ```
4. **Идемпотентность.** Если в секции уже есть запись на этого person'а — не добавлять вторую. Можно проапгрейдить plain-WikiLink-запись в gallery-формат (in-place edit), если у person'а появился `images.cover`.
5. **Минимальный патч.** Не трогать другие поля. Не нормализовывать. Не «попутно чинить».

Если связанных аниме-карточек нет — пропустить подшаг.

#### 7.5b. VA-cross-update

Для каждой VA-записи в `works[]` (`roles: [voice-actor]` + `character: <C>`), у которой есть карточка в `CHARACTERS/<C>.md`:

1. Прочитать `CHARACTERS/<C>.md`.
2. Проверить `voice_actors[]` — есть ли запись `{anime: <title>, person: <этот человек>}`. Если **отсутствует** — добавить во frontmatter (`voice_actors[]` сортируется хронологически по anime — порядок определяется по `featured_in` той же карточки).
3. В секции `## Сэйю` тела персонажа добавить/обновить запись (если секции нет — создать). Формат — gallery-стиль (см. `Linking_guidelines.md`, ширина 60):
   - У текущей person-карточки есть `images.cover` → миниатюра + WikiLink + язык/тайтл:
     ```markdown
     - ![[<thisPerson>_cover.<ext>|60]] **Japanese ([[<title>|<Title>]]):** [[<thisPerson>|<Имя>]] — <одна строка контекста, опционально>.
     ```
   - Нет cover'а → без миниатюры:
     ```markdown
     - **Japanese ([[<title>|<Title>]]):** [[<thisPerson>|<Имя>]] — <контекст>.
     ```
   - Если этот же VA озвучивал персонажа во всех известных `featured_in[]` → можно свернуть в одну запись: `**Japanese (все тайтлы):** [[<thisPerson>|<Имя>]]` (без перечисления тайтлов).
4. **НЕ трогать `anime.staff[]`** — voice-actor туда не входит (см. enums.yaml::staff_role).
5. **Идемпотентность.** Не добавлять дубль `{anime, person}` в `voice_actors[]`. Если plain-text запись `## Сэйю` уже есть на эту пару — апгрейдить в gallery in-place.

Если карточки персонажа нет — пропустить (она появится позже через `/new-character`, который сам поднимет VA-связи через forward-check).

#### 7.5c. VA-cross-update — anime-таблицы `## Персонажи`

После 7.5b — для каждой VA-записи `{title, character}` дополнительно проапгрейдить col 2 (Сэйю) в анимешной таблице `## Персонажи`. Это обеспечивает forward-канал «новая VA-карточка появилась в волте → анимешный кастинг-лист отражает её сразу».

**Алгоритм для каждой VA-записи (если карточки `ANIME/<title>.md` и `CHARACTERS/<character>.md` обе существуют):**

1. **Прочитать** `ANIME/<title>.md`.
2. **Найти секцию `## Персонажи`.** Определить формат: 2-колоночная таблица (`| Персонаж | Сэйю |`) или legacy bullet-list. Скилл апдейтит **только табличный формат** — legacy bullet таблицу VA в себе не содержит, и пытаться втиснуть VA в bullet-описание не стоит (нарушит single-channel). Если bullet — пропустить с записью в отчёт: «`ANIME/X.md`: legacy bullet-формат, ручная конвертация в таблицу нужна для отображения VA».
3. **Найти строку таблицы**, где col 1 (Персонаж) ссылается на `[[<character>]]` или `[[<character>|<...>]]` (с pipe-escaping `\|`) либо содержит plain-text имя персонажа (`name_romaji`/`name_english`/`aliases[]` из `CHARACTERS/<character>.md`).
4. **Проверить col 2 (Сэйю)**:
   - **Уже WikiLink на этого VA** (`[[<thisPerson>]]` или `[[<thisPerson>\|...]]`) — пропустить, идемпотентность.
   - **Plain-text имя этого VA** (`name_primary`/`name_native`/`aliases[]`) — апгрейдить in-place: заменить только содержимое col 2 на gallery-формат (миниатюра 60 при наличии `images.cover`; WikiLink без миниатюры иначе). Pipe-escaping обязательно.
     - Было: `| ![[Yumi_cover.jpg\|60]] **[[Yumi\|Юми]]** | Kana Ueda |`
     - Стало: `| ![[Yumi_cover.jpg\|60]] **[[Yumi\|Юми]]** | ![[Kana_Ueda_cover.jpg\|60]] **[[Kana_Ueda\|Kana Ueda]]** |`
   - **`—` (em dash, пусто)** — заполнить gallery-формат для этого VA (col 2 → WikiLink+миниатюра).
   - **Другой VA имя/линк** — конфликт: в таблице уже указан иной сэйю, чем тот, кого мы добавляем. Это может быть кастинговая смена по сезонам, или ошибка данных. В отчёт: «`ANIME/X.md` строка `<character>`: col 2 содержит `<other VA>`, новый `<thisPerson>` не записан, требуется ручная проверка». Не перезаписывать.
5. **Col 1 не трогать** — она принадлежит `/new-character` reverse-flow.
6. **Обновить `updated`** в frontmatter `ANIME/<title>.md` на сегодня (только если строка реально менялась).

Если карточек `ANIME/<title>.md` или `CHARACTERS/<character>.md` нет — пропустить (отсутствующая anime-карточка означает, что таблицы и не было; отсутствующий character — что forward-check сам подтянет VA при создании character-карточки через `/new-character` шаг 4).

### 8. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** для созданной + обновлённых заметок. Issues — в отчёт дословно.

### 9. Сообщить результат

Показать: имя файла, путь, основное имя, число строк, обновлённые связанные аниме (если были), результат аудита. Предложить `/verify`.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **Ставить `co_authored: <model-id>`** при создании. Формат `claude-opus-4.7` / `claude-sonnet-4.6` / `claude-haiku-4.5` (источник — системный промпт, последний дефис → точка). `/verify` это поле не трогает.
- Web research — обязателен. Не выдумывать факты, особенно даты.
- Имя файла — латиница, без диакритики (`Hayao_Miyazaki`, не `Hayao_Miyazakı` или `宮崎駿`). Кириллица — в `name_native`/`aliases`.
- При добавлении тайтлов в `works[]` — сразу запускать cross-update (шаг 7.5), чтобы граф не съезжал.
- **Сэйю и production-роли смешивать в одной записи `works[]` нельзя.** Если человек — и режиссёр, и VA одного тайтла, это две отдельные записи: одна `roles: [director]` без `character`, другая `roles: [voice-actor], character: X`. Запись с `roles: [voice-actor]` без `character` — невалидна.
- Cross-update для VA-роли ходит в две локации: `CHARACTERS/<character>.md` (шаг 7.5b: `voice_actors[]` + `## Сэйю`) и `ANIME/<title>.md` (шаг 7.5c: только col 2 таблицы `## Персонажи`, frontmatter не трогается). **`anime.staff[]` не апдейтится** — voice-actor туда не входит (см. `enums.yaml::staff_role`).
