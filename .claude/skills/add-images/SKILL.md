# /add-images — Добавление постера и галереи в карточку

Прикрепляет изображения к существующей заметке волта.

- **`poster`** — главный hero-постер. Эмбед сразу после `# <H1>`, до lead-параграфа. Во frontmatter трэкается как `images.cover`.
- **`gallery`** — элементы галереи. Эмбеды в секцию `## Галерея` (создаётся при необходимости, ставится перед `## Ссылки` или в конец тела). Во frontmatter **не трэкаются** — тело является единственным источником истины для галереи. Плагин **Image Toolkit** при клике на любую картинку собирает все эмбеды заметки в gallery navbar автоматически (никакой спец-разметки не нужно).

## Аргументы

```
/add-images <note> <type> <source>
```

1. **`<note>`** — имя карточки без `.md` (`Nausicaa_of_the_Valley_of_the_Wind`), относительный путь от корня волта (`ANIME/Nausicaa_of_the_Valley_of_the_Wind.md`) или абсолютный путь.
2. **`<type>`** — `poster` или `gallery` (case-insensitive).
3. **`<source>`** — путь к одному файлу или папке.

**Перекрытие type:** если `<source>` — папка, **все** изображения внутри трактуются как `gallery` независимо от переданного `<type>`. Об этом сообщается в начале выполнения. Это сделано осознанно: если пользователь указал папку с несколькими файлами, он явно загружает галерею, даже если случайно написал `poster`.

Примеры:
- `/add-images Nausicaa_of_the_Valley_of_the_Wind poster D:/downloads/nausicaa_poster.jpg`
- `/add-images ANIME/Demon_Slayer gallery D:/photos/anime/demon-slayer/`
- `/add-images Tokyo_Ghoul poster D:/screenshots/tokyo-ghoul/` — папка → type перекрыт на `gallery`, копируется всё содержимое.

## Источники правил

1. `SYSTEM/Metadata_schema.md` — конвенции frontmatter (поле `images.cover`).
2. `SYSTEM/Naming_conventions.md` — имена файлов (PascalCase_With_Underscores).
3. `SYSTEM/Linking_guidelines.md` — эмбеды (`![[file]]`) — частный случай WikiLink, действуют те же правила.

## Алгоритм

### 1. Резолв заметки

1. Если `<note>` содержит `/` или `\` — нормализовать к относительному пути от корня волта; убедиться, что файл существует.
2. Иначе — Glob `**/<note>.md` от корня волта:
   - Один матч → использовать.
   - Несколько → показать список структурированным multi-choice вопросом, попросить выбрать.
   - Ноль → ошибка «Заметка `<note>` не найдена».
3. Прочитать файл, извлечь:
   - `basename` — имя без `.md`.
   - frontmatter-блок (между двумя `---`).
   - тело (после второго `---`).
   - позиция первого `# <H1>` heading'а.

Если `basename` содержит символы, которые плохо работают в shell (пробелы, кавычки) — отказать. По `SYSTEM/Naming_conventions.md` имена и так PascalCase_With_Underscores, но проверка lazy-safe.

### 2. Резолв source

1. Source не существует → ошибка.
2. **Файл** → один image. Проверить расширение из `{jpg, jpeg, png, webp, gif, bmp, avif}`. Иначе — ошибка «Неподдерживаемое расширение». **Caveat по типу:** для `gallery` измеримы только `{jpg, jpeg, png, gif, bmp}` — `webp`/`avif` System.Drawing не читает, посчитать aspect (шаг 6.5) нельзя → в роли gallery они отклоняются **сразу по расширению, до копирования** (см. шаг 6.5 п.7). Для `poster` `webp`/`avif` допустимы (aspect не меряется). Effective_type известен после шага 3 — guard применять там же.
3. **Папка** → энумерация файлов **только верхнего уровня** (не рекурсивно — чтобы случайно не подтянуть огромные деревья). Фильтр по расширениям. Сортировка по имени (детерминированность). Если ноль картинок — ошибка.

### 3. Определить effective_type

- Source — файл → `effective_type = <type>` (после lower-case-нормализации).
- Source — папка → `effective_type = "gallery"` (перекрытие). Сообщить пользователю одной строкой: «Source — папка, type перекрыт на gallery. Будет добавлено N изображений в галерею».

`effective_type` ∈ `{poster, gallery}`. Иначе — ошибка.

### 4. Конфликт-чек (только для poster)

Если `effective_type == "poster"` и во frontmatter уже есть `images.cover`:

Структурированным multi-choice вопросом спросить:

- **Заменить.** Скопировать новый, обновить frontmatter + hero-эмбед в теле этой карточки. Дальше — обработка старого файла + cross-update embeds в волте (см. ниже).
- **Перенести старый в gallery.** Переименовать `attachments/<basename>_cover.<ext>` → `attachments/<basename>_gallery_<N>.<ext>` (N = next-free, см. шаг 5). В frontmatter убрать `images.cover`, в теле hero-эмбед удалить (или переписать). Добавить старый файл как gallery-item (эмбед в секцию `## Галерея`). Затем установить новый как cover. **Cross-update также применяется** — embeds в других заметках, ссылающиеся на старое имя, надо перевести на новое (gallery-имя) или на новый cover-файл — решает пользователь (см. ниже шаг 4.5).
- **Отменить.** Прервать. Source-файл не копируется.

### 4.5. Cross-update embeds в других заметках (только для poster при «Заменить» / «Перенести в gallery»)

**Зачем.** Картинка постера может быть зашита в галереи/секции других заметок (типовой пример: миниатюра персонажа в секции `## Персонажи` карточки тайтла). При смене расширения файла (`.jpg` → `.png`) или переносе старого файла под новое имя (cover → gallery) — embeds в этих заметках становятся stale.

**Когда пропустить.** Если новый файл сохраняется под тем же именем что и старый (то же расширение, режим «Заменить») — Obsidian re-render'ит embeds автоматически по filename. Cross-update в этом случае **не нужен**, шаг пропустить.

**Алгоритм:**

1. **Найти embeds старого файла через vault-index** (предпочтительно):
   ```
   # vault-index MCP
   vault_backlinks(note="<old_filename>")
   ```
   Вернёт массив `[{sourcePath, line, displayText}]`. `displayText` содержит ширину из `|<W>` синтаксиса embed'а (либо пусто, если ширина не указана). Это точный, индексированный поиск — без regex-сафари.

   **Fallback (если vault-index недоступен):** Grep по волту с pattern'ом `\!\[\[<old_filename>` (экранировать спецсимволы в `<old_filename>`).

2. **Исключить из списка саму карточку**, которую мы редактируем — она уже обработана шагом 7/8.

3. **Если список пуст** — cross-update не нужен, пропустить шаг.

4. **Если список не пуст** — показать пользователю список структурированным multi-choice вопросом:
   ```
   Старое имя `<old_filename>` встречается в N заметках:
   - ANIME/Demon_Slayer.md:42 (ширина 60)
   - ANIME/Demon_Slayer_Mugen_Train.md:38 (ширина 60)
   - PERSONS/Hayao_Miyazaki.md:71 (без ширины)

   Обновить ссылки на <new_filename>?
   - Да, обновить все (Recommended)
   - Нет, оставить как есть (картинки в этих заметках будут broken)
   ```

5. **Если «Да»** — для каждой строки из списка:
   - Edit: заменить `![[<old_filename>` → `![[<new_filename>` (сохраняя `|<displayText>]]` хвост).
   - Не трогать остальное в файле, минимальный патч.
   - Обновить `updated` во frontmatter затронутой заметки на сегодня.

6. **Удаление старого файла.** Только после успешного cross-update'а:
   - Режим «Заменить» + изменение расширения → удалить `attachments/<old_filename>` через `rm` (Bash).
   - Режим «Заменить» + то же расширение → файл уже перезаписан копией нового, удалять нечего.
   - Режим «Перенести в gallery» → старый файл уже переименован шагом 4 (а не удалён).
   - Если пользователь выбрал «Нет, оставить как есть» на шаге 4 — НЕ удалять старый файл (broken embeds должны хотя бы рендериться старым изображением).

### 5. Подготовка `attachments/` (корень волта)

**Конвенция волта — единая папка `attachments/` в корне** (Obsidian-настройка «Attachments folder» указывает на неё; все заметки используют одну и ту же папку, без подразделения по note-папкам). Все пути `attachments/...` в этом скилле — относительные от корня волта.

1. Проверить существование `attachments/` от корня. Если нет — `mkdir -p attachments` (создаёт корень-уровневую папку; `cd` в vault root обязателен перед запуском).
2. Для `gallery` — найти next-free индекс:
   - Glob `attachments/<basename>_gallery_*.<ext>` (по всем расширениям).
   - N = max(вытащенных номеров) + 1, при пустом match'е N = 1.

### 6. Копирование

Для каждого source-файла:
1. Целевое имя:
   - `poster` → `<basename>_cover.<src_ext>` (расширение источника сохраняется).
   - `gallery` → `<basename>_gallery_<N>.<src_ext>`, N инкрементируется для каждого следующего файла в пачке.
2. **Anti-clobber ДО `cp`:** если `attachments/<target>` уже существует (`Test-Path`) — это не наш чистый случай. Для `poster` коллизию разруливает шаг 4 (через frontmatter); но если файл лежит на диске **без** записи `images.cover` (сирота/рассинхрон) — **стоп**, спросить пользователя структурированным multi-choice вопросом (заменить / выбрать другое имя), не перезаписывать молча. Для `gallery` next-free индекс (шаг 5) и так гарантирует свободное имя.
3. `cp "<source>" "attachments/<target>"` через Bash.
4. Проверки: exit code 0, размер > 1 KB (защита от 0-byte / placeholder).

**Атомарность пачки.** Если на k-м файле copy упал — откатить (`rm`) предыдущие k-1 скопированных файлов **из этого вызова**. Frontmatter и тело не трогать до успешного копирования всей пачки.

### 6.5. Aspect-compatibility check (только для `gallery`)

**Зачем.** Obsidian-эмбед `![[file|W]]` масштабирует картинку до точной ширины W; display-высота считается из aspect исходника (`display_h = W / aspect`). Если в одной строке таблицы оказываются картинки с разными aspect'ами, их display-высоты разные → нижний край строки рваный.

**Обойти это на уровне embed-синтаксиса нельзя.** Полная таблица того, что есть в Obsidian:

| Синтаксис | Семантика | Почему не подходит |
|---|---|---|
| `![[file]]` (без bound) | Натуральный размер | Нативные 1920×1080 раздувают ячейку |
| `![[file\|W]]` | Точная ширина W | При разных aspect'ах — разные display-h, ragged bottom |
| `![[file\|WxH]]` | Точные оба | Квадратит не-square (искажение пропорций) |
| `![[file\|0xH]]` | height-only | **Сломан** в текущем рендере: ставит width=0, картинка исчезает |
| `![[file\|xH]]` | height-only (невалидный) | Obsidian парсит `xH` как `width=xH` (не число), молча игнорирует, натуральный размер |

**Единственное оставшееся решение — гейт на уровне загрузки:** принимаем только файлы, aspect которых совпадает с aspect'ом существующей галереи (или первого файла пачки для новой секции) в пределах tolerance. Display с фиксированной шириной W (см. шаг 6.6) даёт тогда aspect-uniform строку с ровным нижним краем. **Image Toolkit при клике всё равно грузит оригинал** — bound применяется только к preview, исходный файл не модифицируется.

**Алгоритм** (для каждого source-файла, **до копирования** в `attachments/`):

1. Открыть source через `System.Drawing.Image` (если формат поддерживается).
2. Считать `w, h`, посчитать `aspect = w / h`.

3. **Определить `gallery_aspect_norm`** — aspect-ratio, с которым сравниваем:
   - Если `## Галерея` уже существует и содержит эмбеды: открыть каждый gallery-файл из `attachments/<basename>_gallery_*` через `System.Drawing.Image`, посчитать aspect каждого. Норма = **медиана** массива (робастно к одному аутлаеру в малых галереях).
   - Если галерея пуста (новая секция) или это первый файл пачки: norm = aspect текущего source-файла.

4. **Tolerance:** `TOLERANCE = 0.15` (15%, hardcoded). `deviation = |new_aspect - norm| / norm`. Если `deviation > 0.15` — отказ всей пачки.

5. **Отказ:** сообщить: «Файл `<source>` имеет aspect `<X>` (соотношение `<w>:<h>`), что отклоняется от нормы галереи `<Y>` (соотношение `<w>:<h>`) на `<Z>%` (> 15%). Галерея остаётся aspect-uniform для ровного рендера — смешивание landscape/portrait в одной строке даёт рваный нижний край, обойти на уровне embed-синтаксиса Obsidian невозможно. Перекодируйте/обрежьте файл под этот aspect, либо разделите на отдельные секции `## Галерея (портретные)` / `## Галерея (альбомные)`.» **Откатить** все уже скопированные в этой пачке файлы (правила атомарности шага 6). Никаких эмбедов в тело не вставлять.

6. **Edge case — несколько файлов в пачке с разными aspect'ами:** первый файл устанавливает norm. Последующие проверяются относительно него. Если в пачке 2 landscape + 1 portrait — первый landscape станет norm, portrait откажется. **Пользователь сортирует пачку сам** (или разбивает на 2 вызова /add-images в разные секции).

7. **Поддерживаемые форматы для измерения:** JPG, PNG, GIF, BMP (System.Drawing). **WebP и AVIF не поддерживаются** — System.Drawing не содержит их кодеков, WIC-кодеки в базовой поставке Windows тоже отсутствуют → aspect измерить нельзя. **Fail-fast (до открытия и копирования пачки):** проверить расширения всех source-файлов для `gallery` ещё до копи-цикла; если есть `.webp`/`.avif`:
   - **Стоп**, сообщить: «Формат `<ext>` не поддерживается для галереи на этом стеке — нельзя измерить aspect для проверки однородности. Перекодируйте файл(ы) в JPG/PNG и повторите.»
   - **Ни одного файла пачки не копировать**, откатить уже скопированное (атомарность шага 6).

   > Примечание: это ограничение измерения, а не самого Obsidian — он webp/avif в галерее рендерит. Когда появится измеритель кодек-независимый (ImageMagick `identify` / Python PIL), gate можно расширить на webp/avif — см. Vault_Roadmap §10.

**Почему tolerance 15%.** Достаточно плотно, чтобы поймать явные аутлаеры (portrait vs landscape, ~60% deviation), и достаточно свободно, чтобы принять близкие семейства: 16:9 (1.78) и 3:2 (1.5) — deviation 16%, **на грани**; 16:9 и 4:3 (1.33) — deviation 25%, **за границей**; 2:3 (0.67) и 9:16 (0.56) — deviation 16%, **на грани**. Для hand-picked галерей скриншотов и кадров из аниме — рабочая величина, но может потребовать корректировки под конкретную секцию. Если нужна другая граница — менять константу `TOLERANCE` в скилле.

**`poster` не проверяется по aspect** — cover единственный, гомогенный, ratio-based расширение до 350/400 отменено (см. шаг 6.6). Aspect gate — только для gallery.

### 6.6. Определить синтаксис эмбеда

**Галерея → фиксированная ширина `|W`**, где `W = 150` (default, hardcoded). Это **вдвое меньше** ширины cover'а (`|300`, см. ниже): cover'у нужна крупная hero-картинка, галерее — компактные миниатюры в ряд из 6 ячеек. Все эмбеды в одной `## Галерея` используют **одну и ту же W**. При aspect-uniform содержимом (см. шаг 6.5) это даёт:
- display-высоты всех картинок ≈ одинаковые (в пределах 15% tolerance, т.е. ±15% от среднего),
- нижний край строки таблицы ровный,
- ячейки одинаковой ширины (W=150), 6 ячеек × 150px = 900px общей ширины превью строки — умещается в читаемую колонку Obsidian,
- Image Toolkit при клике грузит оригинальный файл в полном разрешении — bound применяется только к preview, файл не модифицируется.

Внутри таблицы pipe **экранирован** `\|` — в `![[file\|150]]` разделитель `|` находится внутри ячейки markdown-таблицы, без экранирования `|` интерпретируется как разделитель колонок и ломает структуру. Стандартное правило `Linking_guidelines.md` → «Pipe-escaping». Разделители колонок самой таблицы `|` (между ячейками) — **не экранируются**, это часть синтаксиса markdown-таблицы.

**Постер → фиксированная ширина 300 (потолок волта).** По `SYSTEM/Linking_guidelines.md` → «Ширина embed собственной обложки/постера карточки» cover-embed кэпится на `|300`: hero-картинка не должна занимать пол-экрана. Ratio-based расширение до 350/400 (для около-квадрата/ландшафта) **отменено** — даже широкий key visual в роли cover ставится `|300`, не больше. Измерять размеры для выбора ширины не нужно. Если по содержательным причинам нужна узкая обложка — допустимо `<300`, но никогда `>300`.

**Backfill-миниатюры (шаг 8.5) — ширина 60/80**, по правилам `Linking_guidelines.md` → «Gallery-стиль для backlink-списков» (anime→80, прочее→60). Это **не** галерейные карточки заметки, а миниатюры в чужих `## Персонажи` / `## Связи` / etc., и для них продолжает действовать width-based bound (компактные маркеры привязки, а не превью-карточки). Скилл не должен путать эти два контекста.

### 7. Обновление frontmatter

**Для `poster`:**
- Если ключа `images:` нет — добавить `images:` блок перед закрывающим `---`.
- Установить `images.cover: "attachments/<filename>"`.

**Для `gallery`:**
- Frontmatter **не трогать**. Тело — единственный источник истины.

Минимальный патч: не сортировать ключи, не нормализовывать кавычки, не переписывать другие поля. Только нужные строки.

### 8. Обновление тела

**Эмбед-синтаксис.** Различается по типу:
- **Постер** — всегда с bound'ом: `![[<filename>|300]]` (фикс. потолок по ширине, шаг 6.6). Голый `![[file]]` без ограничения для постера **не использовать** — без bound'а Obsidian растянет картинку на ширину заметки.
- **Галерея** — фиксированная ширина `![[<filename>\|150]]` (внутри таблицы pipe экранирован `\|`). Bound обязателен: без него Obsidian рендерит картинку в нативном разрешении (например, 1920×1080 → 1920px), что растягивает табличную ячейку на всю ширину заметки и ломает grid из 6 колонок. W=150 (вдвое меньше cover'а 300) держит ячейки компактными, 6 ячеек × 150px = 900px строки. Aspect-uniform содержимого (см. шаг 6.5) обеспечивает ровный нижний край строки.

**Для `poster`:**

Найти первый `# <H1>` heading.

Если сразу под H1 (с учётом пустых строк) уже стоит hero-эмбед вида `![[<basename>_cover.*]]` (с любой шириной или без) — заменить его на `![[<new_filename>|<W>]]` inline (один Edit).

Иначе — вставить блок между H1 и первой непустой строкой ниже:

```markdown
# <H1>

![[<new_filename>|<W>]]

**<Lead>** — ...
```

(Пустая строка до и после эмбеда.)

Если H1 отсутствует — эмбед в самое начало тела (после frontmatter), с пустой строкой после.

**Для `gallery` — рендер таблицей-сеткой.** Картинки укладываются в markdown-таблицу по **6 в ряд**, дальше переносом вниз. Это единственный беc-плагинный способ пустить эмбеды горизонтально; без таблицы они растягивают заметку вертикальной «простынёй».

1. Найти секцию `## Галерея`:
   - Есть → работаем с её текущим содержимым (от заголовка до следующего `##` или конца файла).
   - Нет → создать. Размещение: есть `## Ссылки` → вставить `## Галерея` сразу перед ней; нет → в самый конец тела.
2. **Собрать полный упорядоченный список галерейных файлов** = эмбеды, уже присутствующие в секции (в текущем порядке; вытащить имена из `![[...]]`, в т.ч. из старого построчного формата и из ячеек таблицы), **плюс** новые из этой пачки (в порядке копирования). Дедуп по имени файла: уже в списке → не добавлять второй раз (упомянуть в отчёте).
3. **Перестроить секцию целиком** как одну таблицу из полного списка (`N` файлов):
   - `cols = min(6, N)` — число колонок.
   - Header-ряд: `cols` пустых ячеек. Delimiter-ряд: `cols` ячеек `---`.
   - Data-ряды: эмбеды слева направо, по `cols` в ряд. Неполный последний ряд **добить пустыми ячейками** до `cols` (чтобы таблица осталась валидной).
   - Каждая ячейка с картинкой: `![[<filename>\|150]]` — фиксированная ширина 150, pipe внутри таблицы экранирован `\|`. **Не `![[file|150]]` без escape** — иначе markdown-парсер интерпретирует `|` как разделитель колонок и таблица рассыпается.

Шаблон (пример для N=8 → 6+2, последний ряд добит пустыми):

```markdown
## Галерея

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| ![[<f1>\|150]] | ![[<f2>\|150]] | ![[<f3>\|150]] | ![[<f4>\|150]] | ![[<f5>\|150]] | ![[<f6>\|150]] |
| ![[<f7>\|150]] | ![[<f8>\|150]] |  |  |  |  |
```

Для N≤6 — одна строка из `N` ячеек (без добивки до 6, заголовок тоже из `N` колонок).

**Перестроение, а не append** — потому что добавление картинок может дополнять последний неполный ряд и сдвигать перенос; проще и надёжнее собрать таблицу заново из полного списка. Порядок старых файлов сохраняется, новые идут в хвост.

**Image Toolkit navbar** собирает все эмбеды заметки независимо от того, в таблице они или нет — клик по миниатюре по-прежнему открывает полноразмерную галерею.

**Защита от дублей** обеспечивается дедупом на шаге 2: файл, уже эмбедящийся в секции, повторно не попадёт в список.

### 8.5. Backfill миниатюр в gallery-style backlinks (только для poster)

**Зачем.** В волте действует gallery-конвенция (`SYSTEM/Linking_guidelines.md`, секция «Gallery-стиль для backlink-списков»): запись в любом списке-ссылок имеет форму `- ![[<target>_cover|<W>]] **[[<target>|<имя>]]** — описание`. Когда у `<target>` появляется первый cover (или меняется расширение → новое имя файла), gallery-записи в **других** заметках, ссылающиеся на этот target, остаются без миниатюры. Этот шаг чинит асимметрию в момент, когда меняется единственный артефакт — `images.cover`.

**Когда применяется.**

- `effective_type == "poster"`, и
- Это либо первая установка `images.cover` для заметки (вариант «новый cover»), либо смена имени файла cover'а (например, при замене с другим расширением — в этом случае шаг 4.5 уже обновил embed-ссылки старого файла на новый, но gallery-записи без миниатюр сюда не попадали и должны быть подтянуты).

При одном и том же имени cover-файла (замена `.jpg` → `.jpg`) gallery-записи и так указывают на корректный файл — шаг пропустить.

**Алгоритм:**

1. **Найти backlinks через vault-index:**
   ```
   # vault-index MCP
   vault_backlinks(note="<basename>")
   ```
   Это вернёт WikiLinks на `<basename>` (не embeds — это другая категория). Список — `[{sourcePath, line, displayText}]`.

2. **Исключить саму редактируемую карточку** (на случай self-link).

3. **Для каждой записи** прочитать соответствующую строку source-файла и определить её gallery-формат. Поддерживаются **два** формата — `vault_backlinks` возвращает оба, ограничиваться bullet'ом **нельзя**:

   **a) Bullet** (`## Связи`, `## Тайтлы`, `## Ключевые работы`, `## Студии` и т.п.):
   ```
   ^(\s*)-\s+(?:!\[\[[^\]]+\]\]\s+)?\*\*\[\[<basename>(\|[^\]]+)?\]\]\*\*
   ```
   list-item (с любым отступом) + опциональная миниатюра + bold WikiLink на target.

   **b) Table-cell** — 2-колоночная `## Персонажи` в anime-карточках (`| Персонаж | Сэйю |`, см. `Linking_guidelines.md` → table-format gallery). Признак: строка — ряд таблицы (начинается с `|`), и в **какой-то её ячейке** стоит bold-WikiLink на target с **экранированным** pipe (`**[[<basename>\|<имя>]]**`):
   ```
   | **[[<basename>\|<имя>]]** | <сэйю> |
   ```
   Backfill идёт **в ту ячейку, где стоит target**: для character-target — col 1 «Персонаж»; для person/VA-target — col 2 «Сэйю». Соседнюю ячейку и разделители `|` не трогать.

   **Идемпотентность (оба формата):** если перед `**[[<basename>...]]**` уже стоит `![[...]]` — миниатюра есть, пропустить. Строки, не подходящие ни под один формат (упоминание в прозе) — пропустить.

4. **Определить ширину миниатюры** на основе `note_kind` **target'а** (т.е. редактируемой заметки — той, чей cover ставим). Таблица из `Linking_guidelines.md` («Ширина миниатюры»):
   - target `note_kind: anime` → **W = 80** (карточки тайтлов в подборках — крупнее, чтобы различать обложки)
   - target `note_kind: character` → **W = 60**
   - target `note_kind: person | studio | прочее` → **W = 60**

   Ширина одна на всю пачку — она зависит только от target'а. Source-заметки могут быть любого типа, миниатюра во всех будет одинаковой.

5. **Если набралась non-empty группа кандидатов** → показать пользователю структурированным multi-choice вопросом:
   ```
   Найдено N gallery-style backlinks без миниатюры:
   - ANIME/X.md:42  (anime → W=80)
   - STUDIOS/Y.md:15 (studio → W=60)
   - ...

   Добавить миниатюры (`![[<new_cover>|W]] `) перед bold-ссылками?
   - Да, добавить все (Recommended)
   - Нет
   ```

6. **Если «Да»** — для каждой записи, **по её формату**:
   - **Bullet** → вставить `![[<new_cover_filename>|<W>]] ` (пробел после `]]`, pipe **не** экранируется) непосредственно перед `**[[<basename>`.
   - **Table-cell** → вставить `![[<new_cover_filename>\|<W>]] ` (pipe **экранирован** `\|` — иначе ломается структура таблицы) перед `**[[<basename>` **в той же ячейке**; разделители `|` и соседнюю ячейку (сэйю) не трогать.
   - Минимальный патч: ничего больше в файле не трогать. Обновить `updated` во frontmatter затронутой заметки на сегодня.

7. **Идемпотентность.** Повторный запуск /add-images на ту же карточку с тем же cover'ом не должен ничего менять — паттерн в шаге 3 отсеивает строки, где миниатюра уже стоит.

8. **Отчёт.** Добавить в финальный отчёт секцию (см. шаг 10).

### 9. Самоаудит

1. Вызвать `vault_lint` (vault-index MCP `vault_lint`) с `target` = путь к заметке.
2. Если есть WARN/ERROR — упомянуть в отчёте дословно. Не пытаться чинить.
3. Дополнительно: пройти Glob проверкой, что все эмбеды резолвятся в существующие файлы в `attachments/`.

### 10. Отчёт

```

source:          <абс. путь>
тип (исходный):  <poster|gallery>
тип (effective): <poster|gallery>   (если перекрыт — пометить «перекрыт»)

Скопировано:
  • attachments/<file_1>  ← <source_1>  (<size> KB, <WxH>, ratio <r>, embed |<W>)
  • attachments/<file_2>  ← <source_2>  (<size> KB, <WxH>, ratio <r>, embed |<W>)
  ...

Frontmatter:

  или: «не изменён» (gallery)

Тело:

Cross-update (если был):

  новое имя:   <new_filename>

    • ANIME/Demon_Slayer.md:42  (|60 → |60)
    • PERSONS/...

Backfill миниатюр в backlinks (если был, только poster):

    • STUDIOS/Topcraft.md:62

```

## КРИТИЧНО

1. **Source-файлы НЕ удалять.** Только копировать. Если source — папка из загрузок, оригинал остаётся у пользователя на случай rollback или если он хочет ту же папку использовать для другой карточки.
2. **Атомарность пачки** (см. шаг 6). Mid-batch failure → откат скопированных файлов, frontmatter/тело не трогать.
3. **Не идти рекурсивно** в подпапки source (шаг 2). Чтобы зацепить subfolder — пользователь передаёт его явно.
4. **Минимальный патч frontmatter.** Только нужные строки. Не сортировать, не нормализовывать.
5. **Не дублировать эмбеды.** Файл уже эмбедится в теле — пропустить вставку, отчёт.
6. **Image Toolkit предполагается включённым в волте.** Без него эмбеды отрендерятся как обычные изображения — деградация прозрачная, гарантий не теряется.
7. **Cover collision — только структурированным multi-choice вопросом.** Никакой автозамены, никаких `--force`. Перезапись постера может быть значимым решением (старый файл может оказаться единственной копией).
8. **Поле `images` — не часть формального `note_kind`-схемы.** Скилл генерик, работает с любым видом заметок. Если в карточке уже есть свой образец `images:` (например, `dict[type→path]` у будущих kind типа mushroom-card) — не ломать его, аккуратно добавить ключ `cover`.
9. **Cross-update при замене постера (шаг 4.5).** Картинки персонажей зашиты в галереи аниме, обложки тайтлов могут попадать в подборки — при смене расширения новый файл получит новое имя, embeds в других заметках протухнут. Шаг 4.5 находит их через `vault_backlinks` (Grep как fallback) и обновляет с сохранением `|<width>`. Старый файл удаляется **только после** успешного cross-update'а и подтверждения пользователя. При том же расширении cross-update пропускается — Obsidian re-render'ит embeds сам по filename.
10. **Backfill миниатюр (шаг 8.5).** Симметричный к 4.5 случай: cover у target'а ставится впервые → gallery-записи в других заметках ссылаются на target plain-WikiLink'ом, без миниатюры. Шаг 8.5 находит их через `vault_backlinks` (это **WikiLinks**, не embeds, vault-index их различает) и фильтрует по gallery-паттерну в **обоих** форматах: bullet (`- **[[X|...]]**`) **и** table-cell (`| **[[X\|...]]** |` в anime `## Персонажи`, col 1). Table-вставка **экранирует** pipe (`![[cover\|W]]`), иначе ломает таблицу. Ширина миниатюры — по `note_kind` **target'а** (того, чей cover ставим: anime→80, character/person/studio→60), **не** source-заметки. Идемпотентно: строки с уже стоящей миниатюрой отсеиваются.

## Известные ограничения

- **Один тип за вызов.** Если хочется и постер, и галерею — два вызова. Это сознательно: меньше переключений контекста и аргументов.
- **Файл-источник по абсолютному пути.** Относительные пути от текущего CWD не поддерживаются (CWD CC может отличаться от того, что ожидает пользователь). Лучше явный абсолютный путь.
- **HTTP/HTTPS URL как source — НЕ поддерживается.** Скилл копирует с локальной ФС. Скачивание из сети — это задача `/new-anime` шага 6.5 (там есть curl-логика для cover) или отдельного будущего скилла. Если поступил URL — отказать с сообщением: «URL не поддерживается, скачайте файл локально, затем передайте путь».
