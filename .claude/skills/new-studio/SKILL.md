# /new-studio — Создание карточки студии

Создаёт карточку анимационной студии-производителя с заполнением всех обязательных полей frontmatter согласно `note_kind: studio` и каркасом тела заметки.

> [!important] Vision-gate
> Скачивание логотипа (6.5) подчиняется секции `## Правила скачивания и валидации изображений` в `CLAUDE.md`. После скачивания пройди **активную** проверку мультимодальности (Read-probe) — не классифицируй картинку по URL / имени файла. Карточка — «фото-часть»: не воспринимаешь логотип → он пропускается с пометкой, текстовая часть доделывается.

## Аргумент

`$ARGUMENTS` — название студии на английском, ромадзи, японском или русском.

Примеры:
- `/new-studio Ufotable`
- `/new-studio "Studio Ghibli"`
- `/new-studio MAPPA`
- `/new-studio Уфотейбл`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек Студий (`note_kind: studio`)». Полный список обязательных и опциональных полей с семантикой.
2. **`SYSTEM/enums.yaml`** — секция `note_kinds.studio`. Машинно-читаемые множества значений для `status`.
3. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: studio`. Обязательные WikiLinks (parent_company, successor, founders) + gallery-конвенция для backlink-секций.
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Glob по `STUDIOS/*.md` — поискать совпадение по `name_primary` и `aliases` (если карточка уже есть под другим именем). Если найдено:
- **Stub (<20 строк)** — не создавать с нуля, дополнить.
- **Наполненная (>50 строк)** — сообщить пользователю, спросить что обновить.

### 2. Сбор данных

Web search → актуальные сведения. Приоритетные источники:
- **Wikipedia** — главный источник для founded year, headquarters, founders, parent_company, status. Англоязычная статья даёт каноничный infobox.
- **MyAnimeList company page** (`https://myanimelist.net/anime/producer/<id>`) — backlinks тайтлов, год основания. URL берётся как `mal_url`.
- **AniList Studio API** — структурированный год основания, headquarters.
- **Anime News Network encyclopedia** (`https://www.animenewsnetwork.com/encyclopedia/company.php?id=<id>`) — fallback по неосновным студиям.
- Официальный сайт студии — для текущего статуса и контактов.

**КРИТИЧНО про числовые/датовые поля** (`founded_year`, dissolved year). Брать **только из прямого web fetch конкретной страницы**, не из summary поисковика. См. `feedback_numeric_facts_from_fetch`. Если в Wikipedia встречаются противоречивые даты («1985» в одном месте, «1984» в другом) — флагать в отчёте, оставлять более авторитетное значение.

**Дополнительно — URL логотипа.** Wikipedia infobox обычно содержит логотип студии (`<img>` в верхней правой части). Скачивать через `Invoke-WebRequest` шаг 6.5.

Собрать данные для всех обязательных полей frontmatter (шаг 4) + вспомогательные для тела (история, ключевые работы для предзаполнения gallery, founders).

### 3. Определить имя файла

PascalCase с подчёркиваниями вместо пробелов, латиница. По умолчанию — `name_primary` с заменой пробелов на `_`. Аббревиатуры — как есть: `MAPPA.md`, `WIT_Studio.md`. Многословные — `Studio_Ghibli.md`, `Production_IG.md`. Без диакритики и кириллицы.

Кириллица, японская письменность — в `name_native` и `aliases[]`, не в имя файла.

### 4. Собрать frontmatter

**Прежде чем подставлять значения** enum-полей (`status`) — прочитать `SYSTEM/enums.yaml::note_kinds.studio` и проверить. Если из web-research приходит значение вне enum'а — структурированным multi-choice вопросом: Добавить / Пропустить / Заменить.

Заполнить ВСЕ обязательные поля:

```yaml
---
type: reference
domain: anime
stability: stable
priority: medium
quality: draft
note_kind: studio
tags:
  - <тематические теги по Tag_taxonomy>
aliases:
  - <русское написание>
  - <аббревиатура или историческое имя>
created: <сегодня>
updated: <сегодня>
name_primary: "<Имя студии в латинице>"
name_native: "<Имя в родной письменности>"
founded_year: <YYYY>
country: "<Страна>"
status: <active | defunct | on-hiatus>
# Рекомендуемые — ВСЕГДА присутствуют во frontmatter (политика «always-present»).
# Если данных нет — null / "" / []. НЕ опускать поле целиком.
headquarters: "<Город, район или ''>"
founders: []                          # plain strings; WikiLinks ставятся в теле
parent_company: "<Имя холдинга или ''>"
successor: "<Имя студии-преемника или ''>"   # только для status: defunct
website: "<URL или ''>"
wikipedia_url: "<URL или ''>"
mal_url: "<URL или ''>"
images:
  cover: "attachments/<File_Name>_cover.<ext>"
---
```

**Соглашения по пустым значениям:** строки → `""`, числа/даты → `null`, списки → `[]`.

**Теги.** Подбирать пошагово (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ — если уместно (`legendary-studio`, `independent`).
2. ТЕМЫ — стиль/специализация (`hand-drawn-animation`, `cgi-heavy`, `seinen-specialist`, `shoujo-specialist`).
3. **НЕГАТИВНАЯ ПРОВЕРКА** — убрать теги, дублирующие `status`/`country`/`note_kind`. Имя студии НЕ ставится тегом.
4. **ПРОБЕЛЫ в таксономии** — обязательно добавить новые теги в `SYSTEM/Tag_taxonomy.md`.

Studio без тегов — норма (kind не требует обязательных).

**Ставить `co_authored: <model-id>`** при создании. Значение — ID текущей модели в волтовом формате `claude-<family>-<major>.<minor>` (например `claude-opus-4.7`, `claude-sonnet-4.6`, `claude-haiku-4.5`). Источник истины — системный промпт, строка «The exact model ID is …» (последний дефис → точка). См. `feedback_co_authored`.

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <name_primary> (<name_native>)

![[<File_Name>_cover.<ext>|300]]

**<Имя на русском или ромадзи>** — <1-2 предложения: год основания, страна, специализация/репутация>.

## История

<Основание (год, основатели, контекст индустрии), ключевые вехи (важные релизы как точки роста), важные коллаборации и кризисы (реструктуризация, банкротство, поглощение). 2-4 абзаца. Без воды.>

## Стиль и репутация

<Что отличает студию: визуальный язык, тематические предпочтения, узнаваемые приёмы, технологии. Опционально — если есть что сказать концептуально. Если это «средняя» студия без выраженного стиля, секцию пропустить.>

## Ключевые работы

<Gallery-формат (см. `Linking_guidelines.md` секция «Gallery-стиль»), ширина 80. ANIME/* foreign-keys из anime.studio backlinks. Сортировка хронологическая или по значимости.>

<Для предзаполнения: vault_backlinks(note="<File_Name>") вернёт все ANIME-карточки, у которых studio = этот studio. Из них собрать список с миниатюрами по три случая (карточка + cover / только WikiLink / plain bold для тех, кого ещё нет).>

```
- ![[<Anime_File>_cover.<ext>|80]] **[[<Anime_File>|<Название>]]** (<год>) — <одна строка контекста: жанр, значимость>.
```

## Основатели и ключевые люди

<Gallery-формат, ширина 60, PERSONS/* для основателей и значимых сотрудников (президент, главный режиссёр, ключевой композитор и т.п.).>

```
- ![[<Person_File>_cover.<ext>|60]] **[[<Person_File>|<Имя>]]** — <роль в студии + контекст>.
```

<Если у основателя/сотрудника нет карточки в PERSONS/ — plain bold (gallery-формат, третий случай).>

## Ссылки

- [Wikipedia](<wikipedia_url>)
- [MyAnimeList](<mal_url>)
- [Официальный сайт](<website>)
```

Размер: 40–120 строк в зависимости от значимости студии. Studio Ghibli — 100+, малоизвестная — 40-50.

### 6. Проверить ссылки и собрать gallery-формат

Обязательные WikiLinks для `note_kind: studio` (если соответствующие карточки в волте):
- `parent_company` → `STUDIOS/<name>.md`
- `successor` → `STUDIOS/<name>.md` (только при `status: defunct`)
- `founders[]` → `PERSONS/<name>.md` — упоминание в gallery-секции «## Основатели и ключевые люди»
- Каждая `ANIME/<X>.md` с `studio: <thisStudio>` → запись в gallery-секции «## Ключевые работы»

**Gallery-формат для всех вышеперечисленных секций.** Для каждого WikiLink — Read frontmatter цели, проверить `images.cover`:
- Есть → префиксировать миниатюрой `![[<cover_filename>|<W>]]`. Ширина: 80 для тайтлов, 60 для персон.
- Нет → bold-WikiLink без миниатюры.
- Карточки нет вообще → plain bold имя.

См. полные правила: `Linking_guidelines.md` секция «Gallery-стиль для backlink-списков».

**Принцип единственного канала.** Если тайтл/персона есть в gallery-секции — его упоминания в прозе «## История» и «## Стиль и репутация» остаются **plain text**, даже если карточка существует. Gallery = единственный навигационный канал.

Проверки:
- Все обязательные WikiLinks присутствуют (если соответствующие заметки в волте).
- Нет дубликатов (`vault_duplicate_links`).
- Все target существуют (Glob или `vault_query`).
- Все embed'ы миниатюр резолвятся в реальные файлы в `attachments/`.

### 6.5. Скачать логотип автоматически

**Правило:** карточка студии создаётся **с логотипом всегда, когда возможно**. Логотип — основной визуальный идентификатор студии; без него карточка не появится как миниатюра в `## Студия и команда` секциях карточек тайтлов.

**Алгоритм:**

1. **Взять URL** из шага 2 (приоритет: Wikipedia infobox logo > MAL company page logo > официальный сайт студии favicon).
2. **Определить расширение** — из URL. Логотипы обычно `.png` (с прозрачным фоном) или `.svg`. SVG желательно конвертировать или скипнуть — не все рендереры дружат с SVG в `![[...]]`.
3. **Скачать** в `attachments/<File_Name>_cover.<ext>` через PowerShell:
   ```powershell
   Invoke-WebRequest -Uri "<URL>" -OutFile "attachments/<File_Name>_cover.<ext>" -UserAgent "AnimeVaultBot/0.1 (https://github.com/anthropics/claude-code; <user email из ~/.claude/projects/.../memory или CLAUDE.md>)"
   ```

   **Wikimedia Commons User-Agent.** Если URL на `upload.wikimedia.org` — Wikimedia требует [User-Agent с описанием бота и контактом](https://meta.wikimedia.org/wiki/User-Agent_policy). Generic `Mozilla/5.0` приводит к 429 даже с одного запроса. Использовать описательный UA вида `<BotName>/<version> (<repo or contact URL>; <email>)`. Для других CDN (MAL, AniList) обычный `Mozilla/5.0` работает.

   **Wikimedia thumbnail size policy.** Запросы `/thumb/.../<width>px-<filename>` принимаются только для размеров из [whitelisted список](https://w.wiki/GHai). Кастомные размеры (300px и т.п. для нестандартных файлов) дают 400. Безопаснее качать оригинал по URL без `/thumb/`: `https://upload.wikimedia.org/wikipedia/commons/<hash>/<filename>` — это всегда работает.
4. **Проверить файл** — тип (реальное изображение, не HTML/ошибка — по сигнатуре или `file`) + размер > 0. Если фейл — попробовать следующий источник из приоритета.
4b. **Vision-gate (обязательно).** До записи `images`/эмбеда — **активная** проверка мультимодальности (канон: `## Правила скачивания и валидации изображений` в `CLAUDE.md`). Открыть валидный растровый файл через `Read`, извлечь визуальный факт (форма/цвет логотипа, текст). **Воспринимаешь** → продолжай к шагу 5. **Не воспринимаешь** (Read без визуала / ошибка / только метаданные) → **СТОП на логотипе**: удалить файл, `images`/эмбед **не** добавлять, классифицировать по URL/имени **запрещено**; в отчёте (шаг 9) пометить «логотип пропущен — модель не мультимодальна; добавить позже через `/add-images`». Текстовую часть карточки доделать. *(SVG как растр через `Read` не проверяется — для него действовать по «Замечанию про SVG» ниже, gate неприменим.)*
5. **Прописать в frontmatter:**
   ```yaml
   images:
     cover: "attachments/<File_Name>_cover.<ext>"
   ```
6. **Вставить embed** в теле под заголовком: `![[<File_Name>_cover.<ext>|300]]`.

**Если ни один источник не дал годной картинки:**
- Опустить поле `images` и embed-строку.
- В отчёте (шаг 9) явно сказать: «логотип не найден — карточка не появится как миниатюра в галереях тайтлов до ручного добавления через `/add-images`».

**Замечание про SVG.** Если единственный доступный логотип — SVG, скачать его, но иметь в виду что в `![[file.svg]]` embed Obsidian рендерит, а некоторые экспортёры — нет. Альтернатива: пропустить SVG, оставить без логотипа.

### 7. Создать файл

Записать в `STUDIOS/<File_Name>.md`.

### 7.5. Reverse-check / cross-update

Этот kind требует **двусторонней** сверки:

**A. ANIME/* которые ссылаются на эту студию через `anime.studio` (foreign-key):**

Для каждой `ANIME/<X>.md` с `studio: <thisStudio>` (находим через `vault_query(folder="ANIME", hasField="studio")` + фильтр по значению, или Grep по frontmatter):

1. **Прочитать карточку** через Read.
2. **Найти секцию `## Студия и команда`.** Если её нет — пропустить с заметкой в отчёт.
3. **Через Grep проверить** наличие WikiLink на эту студию в секции:
   - **Уже WikiLink** → ничего не делать.
   - **Plain-text упоминание имени студии** в секции → in-place upgrade. Заменить голое имя на bold-WikiLink + миниатюру (логотип только что скачан шагом 6.5):
     - Было: `- **Topcraft** — студия-производитель; упразднена в 1985 после поглощения...`
     - Стало: `- ![[Topcraft_cover.png\|60]] **[[Topcraft|Topcraft]]** — студия-производитель; упразднена в 1985 после поглощения...`
     - Описание сохранить без изменений.
   - **Не упомянута вообще** → добавить запись в начало bullet-списка секции (студия обычно идёт первой):
     ```markdown
     - ![[<thisStudio>_cover.<ext>|60]] **[[<thisStudio>|<Имя>]]** — студия-производитель.
     ```

4. **Обновить `updated`** в frontmatter затронутой карточки (только если карточка реально менялась).

**B. PERSONS/* — основатели и ключевые люди этой студии:**

Целевой набор персон — `founders[]` из frontmatter **+** все WikiLinks на `PERSONS/<X>` в gallery-секции «## Основатели и ключевые люди» только что созданной карточки студии. Для каждой `PERSONS/<X>.md`, которая существует в волте:

1. **Прочитать карточку** через Read.
2. **Найти секцию `## Студии`**. Если её нет — создать (вставить перед `## Ключевые работы`, поскольку это симметричная backlink-секция и логично идёт первой).
3. **Через Grep проверить** наличие WikiLink на эту студию в секции:
   - **Уже WikiLink** → ничего не делать.
   - **Не упомянута** → добавить запись в gallery-формате (логотип скачан шагом 6.5):
     ```markdown
     - ![[<thisStudio>_cover.<ext>|60]] **[[<thisStudio>|<Имя>]]** — <роль/период/контекст>.
     ```
     Если в карточке студии этот person — `founder` → описание «основатель». Если только в `## Основатели и ключевые люди` без `founders[]` → описание формата «ключевая работа», например «режиссёр <ключевой тайтл>».
4. **Обновить `updated`** в frontmatter затронутой карточки (только если она реально менялась).

**НЕ упоминать студию в `## Ключевые работы`** — там только тайтлы. **НЕ дублировать** упоминание студии в прозе биографии — gallery-секция = единственный канал WikiLink на эту студию.

Если карточки нет — пропустить с записью в отчёт; можно предложить пользователю создать через `/new-person`.

**Не делать:**
- Не превращать plain-text упоминания студии в других карточках волта (за пределами секции `## Студия и команда`) в WikiLinks. Single-channel принцип.
- Не вставлять упоминания в синопсис/сюжет/описание команды (для anime cards). Только в gallery-секции `## Студия и команда`.

### 8. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** для **всех изменённых заметок** (созданная карточка студии + обновлённые карточки аниме из шага 7.5A + опционально person'ы из 7.5B). Issues — в отчёт дословно.

### 9. Сообщить результат

Показать: имя файла, путь, основное имя, число строк, статус, обновлённые связанные заметки (если были), результат аудита. Предложить `/verify` если устраивает.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **Ставить `co_authored: <model-id>`** при создании. Формат `claude-opus-4.7` / `claude-sonnet-4.6` / `claude-haiku-4.5` (источник — системный промпт, последний дефис → точка). `/verify` это поле не трогает.
- Web research — обязательная часть фазы 2. Не выдумывать факты, особенно `founded_year` и `parent_company` (структуры собственности японских анимационных компаний нетривиальны).
- **Логотип скачивать всегда, когда возможно.** Без него студия — невидимая запись в галереях тайтлов; смысл cross-update'а в `## Студия и команда` теряется.
- При обнаружении новых тегов, которых нет в `SYSTEM/Tag_taxonomy.md` — автоматически добавить их туда.
- `status: defunct` без заполненного `successor` — допустимо (не все закрытые студии имели прямого преемника). Но если есть очевидный преемник (Topcraft → Studio Ghibli) — заполнить.
- `parent_company` ≠ `successor`. Parent — нынешний холдинг; successor — кто фактически продолжил традицию после ликвидации. Для активной студии `successor: ""`.
- Карточка студии без `vault_backlinks` (ни одного аниме в волте с этой студией) — допустимый временный случай. Секция `## Ключевые работы` пустая до появления первого тайтла; `/new-anime` cross-update её заполнит.
