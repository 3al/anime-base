# /expand-stub — Расширение заметки-заглушки

Превращает пустую или тонкую заметку в полноценную.

## Аргумент

`$ARGUMENTS` — имя файла, опционально с комментарием через пробел.

Парсинг: первое слово (или путь) — имя файла. Всё остальное — комментарий пользователя.

Примеры:
- `/expand-stub Token` — расширить без указаний
- `/expand-stub Token добавь сравнение BPE vs SentencePiece` — расширить с акцентом
- `/expand-stub RAG обрати внимание на гибридный поиск и reranking` — расширить с фокусом

Если передано имя без пути — найти через Glob.

## Источники правил

Перед расширением прочитай актуальные правила из:

1. **`SYSTEM/Metadata_schema.md`** — обязательные поля frontmatter, поля по `note_kind`
2. **`SYSTEM/Linking_guidelines.md`** — обязательные ссылки по `note_kind` и `type`
3. **`SYSTEM/tag_taxonomy.yaml`** — канон тегов (SSOT); правила/критерий — `.claude/skills/audit-by-creator/references/tag-discipline.md`
4. **`SYSTEM/Naming_conventions.md`** — правила именования

## Алгоритм

### 1. Прочитать заметку и комментарий

Определить:
- Текущее состояние: пустая (только frontmatter или вообще без содержания), тонкая (<20 строк контента), или уже наполненная
- Тип заметки из frontmatter (`type`, `domain`, `note_kind`)
- Существующие теги и ссылки — сохранить их

Если есть **комментарий пользователя** — учитывать как приоритетное указание:
- Запрошенные темы/секции включить обязательно
- Акценты пользователя важнее стандартной структуры по типу
- Если комментарий просит что-то конкретное для уже наполненной заметки — дописать только запрошенное, не переписывая остальное

Если заметка уже наполнена (>50 строк контента) и комментария нет — сообщить пользователю и спросить, что именно дописать.

### 2. Оценить масштаб темы

Размер заметки должен соответствовать объёму темы:
- **Узкий концепт** (один термин, один принцип) — 30–60 строк
- **Средняя тема** (конкретный инструмент, принцип с примерами) — 80–150 строк
- **Большая тема** (большая система, фреймворк, протокол) — 150–200+ строк

Не сжимать большую тему в 50 строк и не раздувать узкий концепт до 200.

### 3. Определить структуру

На основе `type` и `domain` выбрать подходящие секции:

**type: concept** — определение, как работает, применение, связь с другими концептами, ограничения, ссылки

**type: principle** — формулировка, зачем нужен, примеры (хорошие и плохие), антипаттерны, связанные принципы

**type: guide** — назначение, установка/настройка, использование, частые проблемы

**type: reference** — описание, ключевые характеристики, сравнение с аналогами, ссылки

**type: pattern** — проблема, решение, когда применять, когда не применять, примеры

**Специальные `note_kind`** (например, тематические карточки) — если у волта есть отдельный скилл для создания/расширения такого типа заметок (`/new-<kind>`), направить на него и **не использовать** этот скилл.

### 4. Собрать контент

- Для технических тем (актуальные данные, версии): использовать web search
- Для принципов и паттернов: писать из знаний, с примерами кода где уместно
- Для форматов: синтаксис, примеры, частые ошибки

Тон: информативный, лаконичный. Без воды — каждая секция несёт ценность.

### 5. Обновить frontmatter

- Добавить отсутствующие обязательные поля
- Исправить невалидные значения (например `priority: middle` → `medium`)
- Добавить `aliases` если имя файла отличается от естественного названия
- Установить `co_authored: <ID текущей модели>` (expand — содержательный пас, не просто create)
- Установить `quality: draft` (пользователь подтвердит через `/verify`)
- **Канон тегов — auto-update**: для каждого тега, добавляемого в заметку, которого нет в `SYSTEM/tag_taxonomy.yaml`: провести через гейт критерия D (`.claude/skills/audit-by-creator/references/tag-discipline.md`), добавить **структурную запись** в yaml (`- name:`/`group:`/`description:`), перегенерировать `Tag_taxonomy.md` (`core/lib/tag_taxonomy_render.mjs`). **Не** править таблицы md руками (генерируются). НЕ рекомендовать пользователю — делать самому. См. Audit_checklist Шаг 3.2 п.5.
- Обновить `updated` на сегодня

### 6. Добавить ссылки

По правилам из Linking_guidelines:
- Обязательные ссылки по `type`/`note_kind`
- Ссылки на связанные заметки vault (найти через Grep/Glob)
- Без дубликатов, только первое упоминание
- Все target должны существовать

### 7. Записать файл

Перезаписать содержимое файла (сохранив имя).

### 8. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** из него для созданной заметки. Если найдены ошибки — исправить автоматически. Если предупреждения — упомянуть в отчёте.

### 9. Сообщить результат

Показать пользователю: что было (stub/тонкая), что стало (секции, число строк), какие ссылки добавлены, результат аудита. Предложить `/verify` если устраивает.
