# /new-{{kind_slug}} — Создание карточки {{kind_name_ru}}

Создаёт карточку {{kind_name_ru}} с заполнением всех обязательных полей frontmatter согласно `note_kind: {{kind_slug}}` и каркасом тела заметки.

## Аргумент

`$ARGUMENTS` — название {{kind_name_ru}} на русском, английском или оригинальном языке.

Примеры:
- `/new-{{kind_slug}} "<пример названия>"`
- `/new-{{kind_slug}} <пример без кавычек>`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек {{kind_name_ru}} (`note_kind: {{kind_slug}}`)». Полный список обязательных и опциональных полей.
2. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: {{kind_slug}}`. Какие WikiLinks обязательны.
3. **`SYSTEM/Tag_taxonomy.md`** — обязательные теги по `note_kind: {{kind_slug}}` (если определены).
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Найти файл через Glob в `{{folder}}`. Если карточка уже существует:
- **Stub / тонкая (<20 строк контента)** — не создавать с нуля, а дополнить. Сохранить существующие данные.
- **Наполненная (>50 строк)** — сообщить пользователю. Спросить, что обновить.

### 2. Сбор данных

Найти актуальные данные через web search. Источники зависят от темы — приоритет отдавать:
- Каноничным базам/каталогам области (для аниме — MAL/AniList, для книг — OpenLibrary/Goodreads, и т.п.)
- Официальным сайтам / источникам автора
- Wikipedia как fallback

Собрать данные для всех обязательных полей frontmatter (см. шаг 4).

### 4. Собрать frontmatter

Заполнить ВСЕ обязательные поля для `note_kind: {{kind_slug}}`:

```yaml
---
type: {{type}}
domain: {{domain}}
stability: stable
priority: medium
quality: draft
note_kind: {{kind_slug}}
tags:
  - <теги по Tag_taxonomy>
aliases:
  - <альтернативные названия>
created: <сегодня в YYYY-MM-DD>
updated: <сегодня в YYYY-MM-DD>
{{fields_yaml}}
---
```

{{mandatory_tags_hint}}

**Теги подбирать пошагово** (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня
2. ДОМЕН/ВЕНДОР
3. ТЕМЫ содержания
4. ОБЯЗАТЕЛЬНЫЕ по `note_kind: {{kind_slug}}` (см. Tag_taxonomy)
5. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с type/domain
6. **ПРОБЕЛЫ в таксономии — ОБЯЗАТЕЛЬНО** добавить новый тег в `SYSTEM/Tag_taxonomy.md` если используешь тег, которого там ещё нет.

**НЕ ставить `co_authored` при создании.** Поле проставляется только через `/verify` или `/audit-note`, когда пользователь явно подтверждает контентный пас модели (см. общий принцип `feedback_co_authored`).

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <Основное название> (<Оригинальное название если отличается>)

**<Название>** — <1-2 предложения: что это, ключевая характеристика, почему интересно>.

{{body_sections_template}}

## Ссылки

<Внешние ссылки на источники: базы, официальные сайты, Wikipedia>
```

Размер заметки: 60–150 строк в зависимости от глубины темы. Лучше короче и плотнее, чем длиннее и водянистее.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **НЕ ставить `co_authored` при создании.** Только через `/verify` или `/audit-note`.
- При обнаружении новых тегов, которых нет в `SYSTEM/Tag_taxonomy.md` — автоматически добавить их туда. Не оставлять заметку без семантически необходимого тега.
- Web research — обязательная часть фазы 2. Не выдумывать факты, не додумывать данные.
- При наличии связанных заметок в волте (которые попадают в обязательные WikiLinks по правилам Linking_guidelines) — обязательно проставить ссылки.
