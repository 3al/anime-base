# /new-{{kind_slug}} — Создание карточки {{kind_name_ru}}

{{vision_gate_banner}}

Создаёт карточку {{kind_name_ru}} с заполнением всех обязательных полей frontmatter согласно `note_kind: {{kind_slug}}` и каркасом тела заметки.

## Аргумент

`$ARGUMENTS` — название {{kind_name_ru}} на русском, английском или оригинальном языке.

Примеры:
- `/new-{{kind_slug}} "<пример названия>"`
- `/new-{{kind_slug}} <пример без кавычек>`

## Источники правил

Перед созданием прочитай:

1. **`SYSTEM/Metadata_schema.md`** — секция «Поля для карточек {{kind_name_ru}} (`note_kind: {{kind_slug}}`)». Полный список обязательных и опциональных полей.
2. **`SYSTEM/Linking_guidelines.md`** — секция `### note_kind: {{kind_slug}}`. Какие WikiLinks обязательны.
3. **`SYSTEM/Tag_taxonomy.md`** — обязательные теги по `note_kind: {{kind_slug}}` (если определены).
4. **`SYSTEM/Naming_conventions.md`** — формат имени файла.

## Алгоритм

### 1. Проверить существование

Найти файл через glob в `{{folder}}`. Если карточка уже существует:
- **Stub / тонкая (<20 строк контента)** — не создавать с нуля, а дополнить. Сохранить существующие данные.
- **Наполненная (>50 строк)** — сообщить пользователю. Спросить, что обновить.

### 2. Сбор данных

Найти актуальные данные через web search. Источники зависят от темы — приоритет отдавать:
- Каноничным базам/каталогам области (для аниме — MAL/AniList, для книг — OpenLibrary/Goodreads, и т.п.)
- Официальным сайтам / источникам автора
- Wikipedia как fallback

Собрать данные для всех обязательных полей frontmatter (см. шаг 4). **Рекомендуемые (опциональные) поля** заполнять тем, что находится в источниках; те, что не нашлись — оставить пустыми (не выдумывать). **Субъективные / личные поля** (оценка, отзыв, статус прочтения/просмотра — помечены `USER-ONLY`) **не заполнять**: их вписывает пользователь, не модель.

### 3. Определить имя файла

{{filename_pattern}}

**Уникальность basename по всему волту (обязательно).** Obsidian резолвит `[[WikiLink]]` по basename, **без учёта папки** — две заметки с одинаковым именем файла в разных папках делают каждую ссылку `[[<Имя>]]` неоднозначной и молча ломают граф. Поэтому проверять занятость имени **по всему волту**, а не только в `{{folder}}`:
1. Glob `**/<Имя>.md` по всему волту.
2. Если совпадений нет → имя свободно, продолжать.
3. Если найдена заметка с таким же basename (в любой папке) → **СТОП, не создавать молча**. Предложить пользователю дизамбигуированное имя и подтвердить: например, более полное/уточнённое имя (фамилия-вперёд, добавить квалификатор) либо суффикс-маркер kind'а. Сам WikiLink в других заметках при этом ведёт на новое имя.

{{aliases_hint}}

### 4. Собрать frontmatter

Заполнить ВСЕ обязательные поля для `note_kind: {{kind_slug}}`:

```yaml
---
type: {{type}}
domain: {{domain}}
stability: stable
priority: medium
quality: draft
note_kind: {{kind_slug}}
tags:
  - <теги по Tag_taxonomy>
aliases:
  - <альтернативные названия>
created: <сегодня в YYYY-MM-DD>
updated: <сегодня в YYYY-MM-DD>
{{fields_yaml}}
---
```

{{mandatory_tags_hint}}

**Теги подбирать пошагово** (как в `/new-note` шаг 6):
1. КАТЕГОРИЯ верхнего уровня
2. ДОМЕН/ВЕНДОР
3. ТЕМЫ содержания
4. ОБЯЗАТЕЛЬНЫЕ по `note_kind: {{kind_slug}}` (см. Tag_taxonomy)
5. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с type/domain
6. **ПРОБЕЛЫ в таксономии — ОБЯЗАТЕЛЬНО** добавить новый тег в `SYSTEM/Tag_taxonomy.md` если используешь тег, которого там ещё нет.

**НЕ ставить `co_authored` при создании.** Поле проставляется только через `/verify` или `/audit-note`, когда пользователь явно подтверждает контентный пас модели (см. общий принцип `feedback_co_authored`).

**Рекомендуемые поля** (помечены комментарием `# опционально`, **но БЕЗ** `# USER-ONLY`) — заполнить найденными в источниках значениями; ненайденные оставить пустыми или удалить строку, но **не выдумывать**.

**`USER-ONLY`-поля** (помечены комментарием `# USER-ONLY` — личная оценка, отзыв, статус) — оставить **пустыми** (или с дефолтом, если он указан в комментарии). Это субъективные данные пользователя; модель их не заполняет (та же дисциплина, что у `co_authored` и фактчекинга). **НИКОГДА не удалять `USER-ONLY`-строку — даже если поле опциональное и значение не найдено.** Поле рендерится пустым специально, чтобы пользователь увидел его и заполнил; удаление прячет поле и убивает discoverability. Правило «удалить ненайденную строку» выше — только про не-`USER-ONLY` рекомендуемые поля.

### 5. Написать содержание

Каркас тела заметки:

```markdown
# <Основное название> (<Оригинальное название если отличается>)

**<Название>** — <1-2 предложения: что это, ключевая характеристика, почему интересно>.

{{body_sections_template}}

## Ссылки

<Внешние ссылки на источники: базы, официальные сайты, Wikipedia>
```

Размер заметки: 60–150 строк в зависимости от глубины темы. Лучше короче и плотнее, чем длиннее и водянистее.

**Секции, помеченные `USER-ONLY`** (личный отзыв, оценка, личные впечатления) — **не заполнять контентом**. Оставить заголовок и плейсхолдер `_Заполняется пользователем._` Модель не выдумывает субъективную оценку за пользователя.

- Нет дубликатов ссылок.

### 6.9. Проверка внутренней согласованности

Перед записью — прогнать карточку по чеклисту самопроверки. Это не косметика: цель поймать решения, которые противоречат сами себе (например поле, описывающее конкретное воплощение сущности, заполнено данными, не соответствующими остальной карточке).

{{consistency_checks}}

Если проверка вскрыла противоречие — **не «дозаполнять» молча**: исправить так, чтобы карточка была согласована, либо (если данных не хватает / решение за пользователем) оставить поле пустым и упомянуть это в отчёте. Не подставлять произвольное значение ради заполненности.

## Важно

- `quality: draft` всегда — пользователь подтвердит через `/verify`.
- **НЕ ставить `co_authored` при создании.** Только через `/verify` или `/audit-note`.
- **`USER-ONLY`-поля и секции** (личная оценка, отзыв, статус) — оставить пустыми. Субъективные данные вписывает пользователь, не модель.
- При обнаружении новых тегов, которых нет в `SYSTEM/Tag_taxonomy.md` — автоматически добавить их туда. Не оставлять заметку без семантически необходимого тега.
- Web research — обязательная часть фазы 2. Не выдумывать факты, не додумывать данные.
- При наличии связанных заметок в волте (которые попадают в обязательные WikiLinks по правилам Linking_guidelines) — обязательно проставить ссылки.
