# SYSTEM touchpoints — карта правок при добавлении note_kind

Точные якоря и паттерны вставки для каждого governance-документа. Используется фазой 4 скилла `/add-note-kind`.

Все правки **аддитивны** (Edit, не Write). Никогда не модифицируем существующие kind — только добавляем.

---

## 1. `SYSTEM/Metadata_schema.md`

### 1.1. Список значений `note_kind`

**Якорь:** строка `**note_kind** — уточнение типа заметки`. Под ней — bullet-список существующих значений вида `- \`mushroom-card\` — карточка вида гриба (\`Mushroom/\`)`.

**Действие:** добавить новую строку перед строкой «Список расширяется по мере необходимости. Поле необязательно — если не указано.»

**Формат строки:**
```markdown
- `<slug>` — <одна строка: что это, в какой папке>
```

Пример для anime:
```markdown
- `anime` — карточка аниме-тайтла (`ANIME/`)
```

### 1.2. Секция полей kind'а

**Якорь:** последняя секция `## Поля для карточек ...` в файле (найти через Grep `^## Поля для`, взять последнее совпадение).

**Действие:** добавить новую `## Поля для ...` секцию **после** последней такой секции (в самый конец файла, если нет других секций ниже).

**Шаблон секции:**

```markdown

## Поля для карточек <название на русском> (`note_kind: <slug>`)

Все поля ниже **обязательны** для карточек <kind>, если не указано иное.

**<field_name>** — <описание>
```yaml
<field_name>: <YAML-пример>
```
<если enum — список допустимых значений с описаниями>

<повторить для каждого обязательного поля>

### Рекомендуемые поля

**<field_name>** — <описание> (рекомендуемое)
```yaml
<field_name>: <YAML-пример>
```

<повторить для каждого опционального поля>
```

**Важно:** ровно одна пустая строка перед `## Поля для ...` (иначе обсидиановский парсер схлопнёт).

## 2. `SYSTEM/Linking_guidelines.md`

### 2.1. Секция `### note_kind: <slug>`

**Якорь:** последняя секция вида `### note_kind: ...` в файле (Grep `^### note_kind:`, взять последнее совпадение, найти конец секции — следующий `^##` или `^###`).

**Действие:** добавить новую `### note_kind: <slug>` секцию **после** последней такой секции, **перед** следующим `## ...` (обычно это `## Синтаксис`).

**Шаблон секции:**

```markdown

### note_kind: <slug>

Карточка <kind> **должна** содержать следующие WikiLinks (если соответствующие заметки существуют в vault):

| Ссылка | Зачем |
| --- | --- |
| <Описание целевой заметки> | <Зачем эта связь> |
| <...> | <...> |

<Опционально: специфические правила взаимности, лимиты, исключения>
```

**Дисциплина таблиц:** все pipes `\|`, никаких лишних `\` перед `_` и `**` (см. CLAUDE.md «Правила форматирования»).

## 3. Канон тегов (`SYSTEM/tag_taxonomy.yaml`) + обязательные теги (манифест)

> Канон тегов — машиночитаемый `SYSTEM/tag_taxonomy.yaml` (yaml = SSOT). `Tag_taxonomy.md` — **генерируемый** вид (`core/lib/tag_taxonomy_render.mjs`), руками не править. Правила/критерий приёма — `.claude/skills/audit-by-creator/references/tag-discipline.md`.

### 3.1. Обязательные теги по `note_kind` → манифест (НЕ md)

**Якорь:** `.claude/vault-manifest.yaml`, ключ `required_tags_by_kind`.

**Действие:** **только если** на фазе 3.3 определены обязательные теги для kind — добавить запись:
```yaml
required_tags_by_kind:
  - { kind: <slug>, tags: [<tag1>, <tag2>] }
```
Бэкстоп — `missing-required-tag` (`vault_lint`). Иначе шаг пропустить (enum-поля часто закрывают обязательность).

### 3.2. Новые теги в каноне

Если правило 3.1 ссылается на теги, которых ещё нет в каноне (логично попадают в существующую группу — съедобность, гименофор, регион, биотоп, сезон и т.п.) — провести через гейт (критерий D) и добавить **структурной записью** в `SYSTEM/tag_taxonomy.yaml`:
```yaml
tags:
  - name: <tag>
    group: category | theme | format
    description: <когда использовать>
    applies_to_kinds: [<slug>]   # опц. — если тег осмыслен только для этого kind
```
Затем перегенерировать `Tag_taxonomy.md` (`core/lib/tag_taxonomy_render.mjs`). Алгоритм подбора — `/new-note` шаг 6 / `Audit_checklist.md §3.2`.

### 3.3. Формат-шаблонные теги (новое измерение)

Иногда новому kind нужны теги **формат-шаблона**, а не закрытого множества (хронология `YYYY-MM`, версии `v<major>.<minor>`, GPS-bucket'ы). Это `group: format`.

**Действие:**
1. Добавить записи в `tag_taxonomy.yaml` с `group: format`. В `name` для шаблона использовать угловые скобки: `name: "<YYYY-MM>"` — чтобы было видно, что это паттерн, а не конкретный тег.
2. В `description` прямо сказать «формат-шаблон, не закрытый список значений» — иначе аудит будет пытаться добавлять конкретные значения как новые канон-теги.

**Не плодить лишнее**: если тег ужимается в существующую группу — лучше так. `group: format` — для принципиально иного измерения классификации.

## 4. `SYSTEM/Vault_architecture.md`

### 4.1. Таблица структуры папок

**Якорь:** `## Структура папок` + таблица под ним (формат `| Папка | Содержимое | Пример |`).

**Действие:** добавить строку **перед** строкой `SYSTEM/` (чтобы пользовательские папки шли до системных). Если в таблице нет такого упорядочения — добавить в конец.

**Формат строки:**
```markdown
| `<FOLDER>/` | <описание> (`note_kind: <slug>`) | `<FOLDER>/<Example_File>.md` |
```

### 4.2. Секция «Куда класть новую заметку?»

**Якорь:** `## Куда класть новую заметку?` + нумерованный список.

**Действие:** добавить пункт **перед** последним «Не подходит ни одна категория?» (он всегда последний).

**Формат:**
```markdown
<N>. Это <описание объекта kind>? → `<FOLDER>/`
```

Пример для anime: `Это карточка аниме-тайтла? → \`ANIME/\``

## 5. `CLAUDE.md`

### 5.1. Таблица «Структура папок»

**Якорь:** `## Структура папок` (в CLAUDE.md, не в Vault_architecture.md) + таблица под ним (формат `| Папка | Содержимое |`, **две колонки**, без «Пример»).

**Действие:** добавить строку перед строкой `SYSTEM/` (или в конец).

**Формат строки:**
```markdown
| `<FOLDER>/` | <короткое описание> |
```

## 6. `.claude/vault-manifest.yaml`

### 6.1. Секция `folders[]`

**Действие:** добавить запись в массив `folders`. Сохранить отступы (2 spaces) и порядок остальных полей.

**Формат записи:**
```yaml
  - path: <FOLDER>/
    description: "<одна строка: что хранится>"
    keywords: [<2-5 ключевых слов, на русском и английском, lower-case>]
    create_skill: /new-<slug>
```

**Важно:** `create_skill` указывает на скилл, который будет сгенерирован шагом 4.8. Имя должно совпадать.

## Порядок применения

Шаги выполнять в порядке 1 → 6, потом mkdir папки, потом генерация скилла. Причина:
- Manifest правится последним из YAML-файлов — там есть зависимость на имя скилла.
- Скилл генерируется в самом конце, после всех governance-правок: если он использует Read для SYSTEM-файлов при первом запуске — там уже всё на месте.

## Откат

Идеального отката нет (не транзакционная файловая система). При падении на шаге N:
1. Сообщить пользователю что́ успело применится.
2. Указать какие файлы оставить, какие отменить вручную (или через `git checkout`).
3. Не пытаться автоматически откатывать — слишком велик риск повредить уже корректные правки в SYSTEM-файлах.

Идемпотентность: повторный запуск `/add-note-kind <тот же slug>` должен на фазе 0 поймать что kind уже есть и предложить «дополнить или прервать».
