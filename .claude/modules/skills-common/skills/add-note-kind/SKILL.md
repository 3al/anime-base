# /add-note-kind — Добавление нового типа заметок

Расширяет волт поддержкой нового специализированного типа заметок (`note_kind`). Делает три вещи одновременно:
1. Патчит governance-документы в `SYSTEM/` (Metadata_schema, Linking_guidelines, Tag_taxonomy, Vault_architecture) и `CLAUDE.md`.
2. Регистрирует папку в `.claude/vault-manifest.yaml` с `create_skill` — после этого `/new-note` будет автоматически делегировать на новый kind.
3. Генерирует пользовательский скилл `/new-<kind>` из встроенного шаблона, без `.managed` маркера.

**Это творческий, не механический скилл.** Не повторяет один и тот же шаблонный набор полей — исследует тему через web и предлагает осмысленную структуру для конкретного домена.

## Аргумент

`$ARGUMENTS` — опционально slug нового kind в kebab-case (`anime`, `book`, `tv-series`). Если не указан — спросить интерактивно на фазе 1.

Примеры:
- `/add-note-kind anime`
- `/add-note-kind` — полностью интерактивно
- `/add-note-kind tabletop-game`

## Источники правил

Перед запуском прочитай:

1. **`SYSTEM/Metadata_schema.md`** — текущий список `note_kind` и формат секций для специфичных полей.
2. **`SYSTEM/Linking_guidelines.md`** — формат секций `### note_kind: <kind>`.
3. **`SYSTEM/Tag_taxonomy.md`** — таблица «Обязательные теги по `note_kind`».
4. **`SYSTEM/Vault_architecture.md`** — таблица структуры папок и список `note_kind`.
5. **`SYSTEM/Naming_conventions.md`** — формат имени папки и slug.
6. **`.claude/vault-manifest.yaml`** — секция `folders`.
7. **`references/system-touchpoints.md`** (этот скилл) — карта точек правки в каждом из вышеперечисленных файлов.
8. **`references/frontmatter-patterns.md`** (этот скилл) — паттерны типизации полей (enum, list, dict, foreign-key).

## Алгоритм

### Фаза 0. Sanity

**0.1.** Распарсить `$ARGUMENTS`:
- Если пусто — `KIND_SLUG=null`, спросить на фазе 1.
- Если задан — нормализовать в kebab-case (lower, дефисы вместо пробелов/подчёркиваний).

**0.2.** Если `KIND_SLUG` задан — проверить нет ли уже такого kind:
- Grep `note_kind` в `SYSTEM/Metadata_schema.md` (искать в списке значений и в заголовках секций).
- Grep `### note_kind: <slug>` в `SYSTEM/Linking_guidelines.md`.
- Если найдено хотя бы одно совпадение — структурированным multi-choice вопросом спросить: «Kind `<slug>` уже частично объявлен. Дополнить/исправить существующее объявление, или прервать?». Если «прервать» — выйти с сообщением.

### Фаза 1. Концепт

Структурированным multi-choice вопросом (один или несколько вопросов в одном блоке) собрать:

**1.1. Slug** (если не задан в аргументе):
- Спросить имя kind на русском или английском.
- Самому предложить kebab-case slug и подтвердить.
- Slug — это значение для поля `note_kind:` во frontmatter. Должен быть коротким, lower-case, kebab-case.

**1.2. `type` для frontmatter**:
- Предложить из списка `Metadata_schema.md` (concept, rule, pattern, guide, project, reference).
- Для карточек объектов (фильм, книга, гриб, место) — обычно `reference`.
- Для процессов/практик — `guide` или `pattern`.

**1.3. `domain`**:
- Предложить либо реюзить существующий (Grep по волту `^domain:` — какие есть), либо новый.
- Если новый — короткий slug (e.g. `anime`, `cooking`, `gaming`).

**1.4. Папка**:
- Предложить имя по `Naming_conventions.md` (обычно `<KIND_UPPER>/` — `ANIME/`, `BOOK/`).
- Можно использовать существующую папку если она тематически подходит.

**1.5. Краткое описание kind** (одно предложение для description в манифесте и заголовков секций).

**1.6. Единица учёта и основной сценарий** (нудж против семантических пробелов): что именно одна карточка представляет и как пользователь её использует. Например для книги — «карточка = произведение» vs «карточка = конкретное прочитанное издание» (от этого зависит, нужен ли `isbn`/`edition` и поля статуса прочтения). Это дизайн-решение пользователя, а не дефолт — но один уточняющий вопрос здесь предотвращает кривую схему полей на фазе 2. Если из названия kind ответ очевиден — не переспрашивать.

### Фаза 2. Web research + черновик полей

**2.1. Исследование темы:**
- Сделать 1–2 web search запроса, нацеленных на структуру данных предметной области, например:
  - `"<kind>" card metadata fields catalog database`
  - `"<kind>" infobox wikipedia fields`
  - Для anime: «MyAnimeList anime entry fields», «AniList GraphQL Media schema»
  - Для книги: «Goodreads book fields», «OpenLibrary book schema»
  - Для рецепта: «Schema.org Recipe properties»
- Прочитать 1 наиболее релевантную страницу через web fetch.
- **Не уходить в дебри** — цель не идеальная схема, а понимание ключевых полей, которые обычно есть у такого объекта.

**2.2. Сформировать draft frontmatter spec:**

Структура для каждого поля:
- Имя поля (snake_case)
- Тип (см. `references/frontmatter-patterns.md`):
  - `string` — свободный текст
  - `enum` — одно значение из закрытого списка
  - `list[enum]` — список значений из закрытого списка
  - `list[string]` — список строк
  - `date` — `YYYY-MM-DD`
  - `dict[type→path]` — словарь типизированных файлов (как `images` у mushroom-card)
  - `list[wikilink]` — список имён других заметок волта (foreign keys)
  - `list[dict]` — список структурированных записей (foreign key + per-record метаданные)
  - `number` / `range`
  - `url`
- Обязательное / опциональное (= рекомендуемое)
- **Субъективное / личное?** Поля вроде оценки, рейтинга, личного отзыва, статуса прочтения/просмотра — это `USER-ONLY`: их заполняет пользователь, не модель. Пометить явно (см. F5 / шаг 4.8).
- **Scope: intrinsic или instance-specific?** *Intrinsic* — свойство самой сущности (верно для неё в любом воплощении). *Instance-specific* — свойство **конкретного воплощения** сущности (издания, релиза, версии, копии, постановки). Это общий принцип «сущность vs её манифестация», без апелляции к конкретному kind. Связать с единицей учёта (probe 1.6): **если единица учёта kind = абстрактная сущность, а поле instance-specific — это структурное напряжение.** Сущность не имеет одного «правильного» значения instance-поля; заполнить его произвольным воплощением → карточка молча привязывается к случайному экземпляру и противоречит сама себе. Решить на дизайне (вынести в варианты для пользователя, см. 2.4):
  - убрать поле (instance-данные не описывают сущность), либо
  - увести в тело заметки (как свободное упоминание, а не структурированное поле), либо
  - оставить, но в генерируемом скилле дать **runtime-guard**: «привяжи к конкретному, явно названному воплощению, либо оставь пустым для пользователя — не подставляй произвольное». Тогда это же напряжение попадёт в `{{consistency_checks}}` (шаг 4.8).
- Краткое описание
- Для `enum` / `list[enum]` — допустимые значения
- YAML-пример

**Объём:** 4–10 обязательных полей + 2–6 опциональных. Не плодить поля «на всякий случай».

**Оба класса полей попадают в генерируемый скилл.** Обязательные — заполняются всегда; рекомендуемые (опциональные) — рендерятся в шаблоне frontmatter как пустые/закомментированные, чтобы пользователь о них знал и `/new-<kind>` заполнял web-находимые. Не рендерить только обязательные — иначе про рекомендуемые поля (`isbn`, `pages`, `series`, …) никто не узнает (F6).

**2.3. Проверить конфликты имён полей с существующими kind:**

Grep по `SYSTEM/Metadata_schema.md` для каждого предложенного имени поля. Если поле с тем же именем уже определено для другого kind — проверить:

- **Семантика та же, тип тот же** → это нормально, имя реюзается (например, `aliases`, `created`, `updated`).
- **Семантика та же, тип отличается** → **семантический конфликт.** Решить:
  - переименовать поле в новом kind (`found_species` → `finds`), либо
  - явно зафиксировать в Linking_guidelines что у разных kind разная семантика поля (тогда обязательно описать поток данных между ними — как с observation ↔ picking-location).
- **Семантика разная, имя случайно совпало** → переименовать, не плодить путаницу.

Этот шаг важен: тип `list[dict]` нового kind, столкнувшийся с `list[wikilink]` существующего — типичный сценарий, который рождает архитектурный спор. Лучше поймать сейчас, чем в фазе 4.

**2.4. Показать пользователю draft** и структурированным multi-choice вопросом (или просто текстом «правки?») собрать корректировки. Итерировать пока пользователь не подтвердит.

**На развилках дизайна — предлагать варианты, не railroad'ить.** Этот скилл творческий (см. начало). Где есть осмысленный выбор — scope поля (intrinsic vs instance-specific и что с ним делать), единица учёта (сущность vs воплощение), типизация связи (`list[wikilink]` vs `list[dict]`), набор и границы секций тела — **не выбирать молча первый вариант.** Сформулировать 2–3 разумных варианта с короткими tradeoff'ами и дать пользователю выбрать. Одна рельса = обеднённый дизайн kind; пользователь часто видит нюанс своей предметной области, который из общих принципов не выводится.

### Фаза 3. Связи и структура тела

Через серию вопросов (можно одним блоком multi-choice вопросов с multi-select где уместно):

**3.1. Обязательные WikiLinks** — какие связи должны быть в каждой заметке этого kind. Например:
- Для anime: studio (если есть карточка), franchise, related-titles.
- Для книги: author, series.
- Для гриба: lookalikes (двойники).

Самому предложить 2–3 разумных варианта на основе фазы 2.

**3.2. Секции в теле заметки** — какой каркас markdown будет у каждой заметки. Предложить осмысленную структуру из общего знания темы, например для anime:
- Краткий синопсис
- Сюжет (по сезонам)
- Главные персонажи
- Студия и команда
- Личный отзыв / оценка
- Ссылки (MAL, AniList, официальный сайт)

Для каждой секции — заголовок второго уровня (`##`) и подсказка что туда писать.

**Пометить субъективные секции как `USER-ONLY`.** Секции личной оценки/впечатлений («Личный отзыв», «Оценка», «Мои впечатления») модель **не заполняет** — это субъективные данные пользователя. В каркасе тела (`{{body_sections_template}}`) такая секция получает только заголовок + плейсхолдер `_Заполняется пользователем._`, без сгенерированного контента. Иначе `/new-<kind>` выдумает отзыв за пользователя (F5).

**3.3. Обязательные теги** (опционально):
- Нужны ли теги, обязательные для всех заметок этого kind? Например, у `mushroom-card` обязательные теги по съедобности.
- Если да — какие.

**3.4. Aliases-паттерн** — какие альтернативные имена обычно бывают (русское название, оригинальное на другом языке, аббревиатура).

**3.5. Галерея / изображения** — нужна ли поддержка изображений в заметке этого kind:
- Если да — какие типы (как у `mushroom-card`: top, side, habitat) или один общий (`cover` для книги/аниме).
- Будет ли поле `images:` во frontmatter и секция «Галерея» в теле.
- **Если да → включается vision-gate.** Любая работа с картинками (скачивание, подбор, классификация, визуальная проверка) требует мультимодальной модели. Это значит три вещи на фазе 4: (а) в волт сеется секция «Правила скачивания и валидации изображений» (если её ещё нет), (б) генерируемый скилл получает баннер vision-gate и inline-проверку в фото-шаге, (в) генерируемый скилл объявляет escape-hatch `--no-images` целостно: флаг в `argument-hint`, шаг детекции в фото-секции, упоминание в баннере. См. шаги 4.5b и 4.8 + КРИТИЧНО #12.
- **`--no-images` — целостно или никак.** Если фото-шаг ссылается на `--no-images` (а vision-gate из `CLAUDE.md` его ожидает как путь обхода), флаг обязан быть в `argument-hint` сгенерированного скилла **и** иметь реальный шаг детекции. Полу-объявленный флаг (упомянут, но не объявлен/не детектируется) — хлипкий escape-hatch (F4).

### Фаза 4. Применение — атомарная транзакция

**Не начинать пока пользователь не подтвердил черновики на фазах 2 и 3.** Все шаги выполнить за один проход; если падает на шаге N — откатить шаги 1..N-1 (или хотя бы сообщить пользователю что́ успело примениться).

Прочитать `references/system-touchpoints.md` — там точные якоря для каждого Edit. Дальше — шаги:

**4.1. Patch `SYSTEM/Metadata_schema.md`:**
- Добавить `- \`<slug>\` — <краткое описание kind>` в список значений `**note_kind**` (после существующих values).
- Добавить новую секцию `## Поля для карточек <название> (\`note_kind: <slug>\`)` в конец файла (после последней `## Поля для ...`). Содержание секции — все поля из фазы 2 с YAML-примерами и допустимыми значениями enum.

**4.2. Patch `SYSTEM/Linking_guidelines.md`:**
- Добавить секцию `### note_kind: <slug>` после последней такой секции. Содержание — правила WikiLinks из фазы 3.1 (что обязательно, что опционально, на какие kind ссылаться).

**4.3. Patch `SYSTEM/Tag_taxonomy.md`** (только если на фазе 3.3 определены обязательные теги):
- В таблицу «Обязательные теги по `note_kind`» добавить строку: `\| <slug> \| <теги через запятую> \|`.
- Каждый новый тег которого ещё нет в таксономии — добавить в соответствующую группу (по тому же алгоритму что `/new-note` шаг 6).

**4.4. Patch `SYSTEM/Vault_architecture.md`:**
- В таблицу структуры папок добавить строку с путём, описанием и привязкой к `note_kind: <slug>`.
- Если есть отдельный список `note_kind` (см. Grep) — добавить и туда.

**4.5. Patch `CLAUDE.md`:**
- В таблицу «Структура папок» добавить строку с папкой и описанием. Эта таблица — пользовательская справка, повторяет Vault_architecture в сжатом виде.

**4.5b. Seed правил изображений в `CLAUDE.md`** (только если на фазе 3.5 решили включить изображения):
- Проверить, есть ли в `CLAUDE.md` секция-заголовок `## Правила скачивания и валидации изображений`.
- **Уже есть** → не трогать (волт уже работает с картинками, канон един; повторно не сеять и не дублировать).
- **Нет** → прочитать `templates/image-handling.claude-fragment.md` и вставить его содержимое в пользовательскую часть `CLAUDE.md` (вне managed-block маркеров `<!-- BEGIN/END: managed by /init-vault -->`), перед секцией про плагины/окружение, если она есть, иначе в конец пользовательской части. Это канонический текст vision-gate + curl/валидация; генерируемый скилл будет на него ссылаться.

**4.5c. Seed `SYSTEM/spec_changelog.yaml`** (enforcement-at-source для epoch-conformance):
- Этот файл фиксирует **дату ввода** каждого требования create-спеки. Его читает `/audit-by-creator` (`ledger-protocol.md §4.1`): требование с `introduced > created` карточки не штрафует автора (эволюция спеки ≠ недоработка модели). Без записи требование считается пред-существовавшим → штрафуется; поэтому новый kind логируем **в момент создания**.
- Если файла нет — создать с заголовком и `v: 1`. Схема:
  ```yaml
  # SYSTEM/spec_changelog.yaml — когда введено каждое требование create-спеки.
  # Контент — vault-side; схему задаёт фреймворк (ledger-protocol §4.1).
  v: 1
  kinds:
    <slug>:
      - requirement: <имя поля / "## Заголовок секции">
        introduced: <сегодня YYYY-MM-DD>
        kind_of: field | section | format
  ```
- **Собери список обязательных требований kind ОДИН РАЗ** — это единый источник для двух артефактов (changelog здесь + spec-requirements блок скилла, шаг 4.8). В список входят: обязательные frontmatter-поля (из фазы 2), обязательные секции тела (из фазы 3.2), любые format-правила (стабильный label, напр. `cover_ext_matches_content`). Опциональные/USER-ONLY — НЕ включать (аудит их не требует). Каждый элемент = `{requirement, kind_of}`.
- Добавить под ключ `kinds.<slug>` запись на **сегодня** (`introduced: <YYYY-MM-DD>`) для **каждого** элемента списка.
- **Этот же список** рендерится в `{{spec_requirements_list}}` шаблона на шаге 4.8 — changelog и spec-requirements блок обязаны быть 1:1 (это проверяет `vault_spec_drift`). Разойдутся — аудит либо ложно штрафует, либо пропускает требование.

**4.6. Patch `.claude/vault-manifest.yaml`:**
- Добавить запись в `folders[]`:
  ```yaml
  - path: <FOLDER>/
    description: "<описание из фазы 1.5>"
    keywords: [<2-5 ключевых слов на русском и английском>]
    create_skill: /new-<slug>
  ```
- **Это ключевой шаг для интеграции** — после этого `/new-note` будет автоматически делегировать создание заметок такого kind в новый скилл.

**4.7. Создать папку** — `mkdir <FOLDER>/`.

**4.8. Сгенерировать `/new-<slug>` скилл:**
- Прочитать `templates/new-kind-skill.template.md`.
- Подставить placeholders (см. шаблон для полного списка):
  - `{{kind_slug}}` — slug. **Должен в точности совпадать с `create_skill: /new-<slug>` из шага 4.6.** Это один и тот же идентификатор — используется как имя скилла И как routing-ключ в манифесте. Расхождение ломает delegation из `/new-note`.
  - `{{kind_name_ru}}` — название на русском
  - `{{argument_hint}}` — значение frontmatter-поля `argument-hint`. По умолчанию `<название {{kind_name_ru}}>`. **Для image-kind (фаза 3.5 = да) — добавить escape-hatch:** `<название {{kind_name_ru}}> [--no-images]`. Так флаг из фото-шага реально объявлен в сигнатуре скилла (F4).
  - `{{folder}}` — папка с `/`
  - `{{domain}}` — domain
  - `{{type}}` — type
  - `{{filename_pattern}}` — формат имени файла. Описание + пример. По умолчанию `PascalCase_With_Underscores.md` (без пробелов, без кириллицы). Для temporal kind типа observation — `<YYYY-MM-DD>_<Location_Slug>.md`. Для других случаев — определить во время фазы 1 или 3 и зафиксировать здесь.
  - `{{fields_yaml}}` — рендеренный YAML-блок полей frontmatter с placeholder-значениями. **Рендерить и обязательные, и рекомендуемые (опциональные) поля** (F6) — иначе пользователь о рекомендуемых не узнает. Разметка:
    - Обязательные — с placeholder-значением: `pages: <число>`.
    - Рекомендуемые — пустые/закомментированные с пометкой: `isbn:            # опционально`.
    - Субъективные / личные (оценка, рейтинг, статус) — пометить `# USER-ONLY`: `rating:          # USER-ONLY — личная оценка, не выдумывать`. Если поле и рекомендуемое, и субъективное — обе пометки: `# опционально, USER-ONLY` (F5).
    - **`USER-ONLY`-поля рендерятся ВСЕГДА и НИКОГДА не удаляются при создании — даже если опциональные (F7).** Это поле-приглашение: пользователь должен увидеть его пустым, чтобы заполнить. «Удалить ненайденную строку» (см. ниже инструкцию шага «Сбор данных») допустимо **только** для не-`USER-ONLY` рекомендуемых полей — борьба с захламлением. `USER-ONLY` ≠ захламление. В генерируемом скилле явно прописать: `USER-ONLY` поле остаётся пустым в frontmatter, не удаляется.
    - Пример:
      ```yaml
      author: <автор>
      year: <год>
      isbn:            # опционально
      publisher:       # опционально
      rating:          # опционально, USER-ONLY — личная оценка
      ```
  - `{{required_links_section}}` — текст про обязательные WikiLinks (из фазы 3.1)
  - `{{body_sections_template}}` — markdown-каркас тела заметки (из фазы 3.2). **Субъективные секции (личный отзыв, оценка, впечатления) рендерить как `USER-ONLY`:** только заголовок `##` + плейсхолдер `_Заполняется пользователем._`, без сгенерированного контента и без подсказки «напиши отзыв». Цель — `/new-<kind>` оставит секцию пустой, а не выдумает отзыв за пользователя (F5).
  - `{{mandatory_tags_hint}}` — подсказка про обязательные теги (из фазы 3.3)
  - `{{aliases_hint}}` — подсказка про aliases (из фазы 3.4)
  - `{{vision_gate_banner}}` — **если на фазе 3.5 включили изображения**: callout-баннер vision-gate в начале скилла, например:
    ```
    > [!important] Vision-gate (обязательно)
    > Этот скилл работает с изображениями. Если передан флаг `--no-images` / `-n` — фото-шаг пропускается целиком (флаг не часть названия). Иначе перед классификацией/подбором фото пройди **Vision-gate** из `CLAUDE.md` → «Правила скачивания и валидации изображений»: **pre-flight probe ДО любого скачивания** — открой ассет `.claude/assets/vision_probe.png` (Read) и убедись, что реально видишь содержимое. Если модель не мультимодальна (Read дал ошибку/метаданные, не визуал) → **СТОП**, ноль скачиваний: не оценивай фото по имени файла/URL, потребуй переключения на мультимодальную модель или запуска с `--no-images`.
    ```
    Если изображений нет — **пустая строка**.
  - `{{gallery_section}}` — секция про изображения (если на фазе 3.5 решили включать) или пустая строка. Для image-kind она **обязана** содержать: **(0) детекцию `--no-images` / `-n` в `$ARGUMENTS`** — если флаг есть, пропустить весь фото-шаг (gate/probe не нужны, см. `CLAUDE.md` п. 1) и при разборе названия игнорировать этот токен; (1) **pre-flight probe** ДО скачивания — Read ассета `.claude/assets/vision_probe.png`, вердикт мультимодальности по поведению Read (слепая → СТОП/пропуск, ноль скачиваний), со ссылкой на `CLAUDE.md` → «Правила скачивания и валидации изображений»; (2) скачивание/валидацию фото (только если probe прошёл) со ссылкой на тот же канон — **включая anti-clobber** (не перезаписывать молча чужой ассет: проверить занятость целевого пути, при коллизии с другой заметкой дизамбигуировать имя) и **расширение по реальному типу содержимого** (не хардкодить `.png`); (3) заполнение `images:` + вставку в галерею, где **cover/постер кэпится `|300`** (`![[<файл>|300]]`), а галерейные миниатюры — узкие по конвенции волта. Эти три правила (anti-clobber, ext-by-content-type, cover-cap) — канон из `CLAUDE.md` → «Правила скачивания и валидации изображений» (посеян на шаге 4.5b); генерируемый скилл на них **ссылается**, не переписывает. Текст — natural-language, без CC-specific tool-имён (см. КРИТИЧНО #11). Флаг из (0) должен совпадать с тем, что объявлен в `{{argument_hint}}` (F4).
  - `{{cross_updates_section}}` — шаги по обновлению связанных карточек. Если на фазе 3 определились правила обновления (например observation → location.found_species union) — описать здесь явно: какие файлы прочитать, какие поля как обновить, что НЕ трогать. Если cross-updates не нужны — поставить «Этот kind не требует обновления других заметок. Шаг пропустить.»
  - `{{consistency_checks}}` — чеклист самопроверки внутренней согласованности для шага 6.9 генерируемого скилла. **Это мета-инструкция «как вывести», а не готовый список:** на этапе генерации **выведи из семантики ИМЕННО ЭТОГО kind** конкретные точки, где карточка может противоречить сама себе, и впиши их сюда как `- [ ]` проверки. Откуда брать точки:
    - **scope-напряжения** из фазы 2.2 — instance-specific поле на kind с единицей учёта «сущность»: проверить, что значение не привязывает карточку к произвольному воплощению, противоречащему остальным (intrinsic) полям;
    - **пары/наборы полей**, которые логически связаны и могут разойтись (поле A подразумевает диапазон/тип поля B);
    - **поле vs секция тела** — структурированное поле и текст не должны противоречить;
    - **enum-инварианты** — значение одного enum исключает/требует другое.
    Каждая проверка — конкретная и проверяемая для этого kind, **не** общая фраза «проверь согласованность» (она бесполезна — слабая модель её проигнорит, сильной не нужна). Если осмысленных точек противоречия у kind нет — короткий базовый чеклист (обязательные поля на месте, ссылки/таргеты валидны, нет выдуманных USER-ONLY). **Конкретику не хардкодить в этот скилл — она выводится здесь и живёт в порождённом `/new-<slug>` (см. КРИТИЧНО #14).**
  - `{{spec_requirements_list}}` — YAML-список обязательных требований kind для managed spec-requirements блока в конце скилла. **Источник — тот же единый список, что и записи changelog (шаг 4.5c)**, без `introduced` (дата — забота changelog). Рендерить по элементу:
    ```yaml
      - requirement: <имя поля | "## Заголовок секции" | format-label>
        kind_of: field | section | format
    ```
    Отступ: `-` на 2 пробелах, `kind_of` на 4. Список обязан быть 1:1 с записями `kinds.<slug>` в `spec_changelog.yaml` — иначе `vault_spec_drift` флагнёт рассинхрон.
- Записать результат в `.claude/skills/new-<slug>/SKILL.md`.
- **Не записывать `.managed` маркер** — это пользовательский скилл. Переустановки `skills-common` его не тронут (см. ops/install.mjs).

**4.9. Сгенерировать Opencode wrapper (безусловно).**

Принцип волта — harness-agnostic by default: новый скилл должен работать в любом harness без отложенной настройки. Wrapper создаётся **всегда**, независимо от того, используется ли Opencode прямо сейчас. CC игнорирует `.opencode/commands/`; Opencode при следующем заходе подхватит готовый wrapper без ручных шагов.

Создать `<vault>/.opencode/commands/new-<slug>.md` (mkdir -p при необходимости) со следующим содержимым:

```yaml
---
description: <description целиком из SKILL.md frontmatter, verbatim>
---
Invoke the `new-<slug>` skill via the skill tool NOW.

User's arguments: $ARGUMENTS

Execute the skill's instructions IMMEDIATELY using the arguments above. Do not just acknowledge — perform the actions and report the result.
```

**`description`** — берётся verbatim из SKILL.md frontmatter, не сокращать. Расхождение между SKILL.md и wrapper description ломает implicit-matching/autocomplete в Opencode. Лимит Opencode для `description` — 1024 chars, наш description обычно укладывается.

**Императивный язык** (`NOW`, `IMMEDIATELY`, `do not acknowledge`) — необходим из-за известного upstream-bug ([sst/opencode#26185](https://github.com/sst/opencode/issues/26185)): slash-вызов одновременно инжектит SKILL.md в user message И оставляет skill tool exposed, на слабых моделях (GLM/Qwen/DeepSeek) это вызывает passive load.

Wrapper — derived artifact, регенерация при изменении frontmatter SKILL.md идемпотентна. См. полный контракт в `docs/Vault_Bootstrap_Architecture.md § Slash UX в Opencode`.

## КРИТИЧНО

1. **Не применять до подтверждения.** Все правки на фазах 2 и 3 — это draft. Применение (фаза 4) — только после явного «да» от пользователя.

2. **Атомарность.** Если шаг 4.N упал — сказать пользователю что успело применится, не оставлять полусделанным. Идемпотентность важна: повторный запуск с тем же slug должен либо штатно обновить (если пользователь выбрал «дополнить» на 0.2), либо сразу сказать «уже есть».

3. **Slug дисциплина.** `note_kind` — это **закрытый словарь**. Slug должен быть kebab-case, lower, без пробелов, без кириллицы. Не плодить варианты: `tv-series` или `tv_show` или `series` — выбрать один и придерживаться. Если близкий slug уже есть в Metadata_schema — спросить, может пользователь хочет реюзить.

4. **Веб-исследование обязательно.** Фаза 2 — без шортката. Если для kind «аниме» предложить тот же набор полей что для гриба — это провал скилла. Каждая тема имеет свою специфику.

5. **Не редактировать чужие модульные файлы.** Скилл правит только:
   - `SYSTEM/*` (после первого посева core они user's)
   - `CLAUDE.md` (управляемый блок — но мы правим **вне** managed-block-маркеров, в пользовательской части)
   - `.claude/vault-manifest.yaml` (секция folders — общий протокол с new-note)
   - Создаёт `.claude/skills/new-<slug>/` (пользовательский, без .managed)

6. **Не трогать существующие note_kind**. Все правки — добавление новой секции/строки, не модификация существующих.

7. **WikiLinks-таблицы:** при формировании Linking_guidelines секции — соблюдать форматирование таблиц из CLAUDE.md (экранирование `\|`, не трогать `_` и `**`).

8. **co_authored:** в генерируемом /new-<slug> скилле — НЕ ставить автоматически. По общему правилу `feedback_co_authored`: модель не видна агенту.

9. **Минимальный патч.** Каждый Edit правит **только то, что нужно для нового kind**. Не «попутно чинить» опечатки, не «заодно обновлять» соседние строки, не нормализовывать форматирование вне зоны правки. Если в файле виден дрейф (опечатки, рассогласованность) — это **отдельная задача**, не часть `/add-note-kind`. Иначе diff коммита смешивается, и невозможно увидеть что именно сделал скилл.

10. **`create_skill` ↔ имя сгенерированного скилла.** В манифесте `create_skill: /new-<slug>` (шаг 4.6) и имя сгенерированной директории `.claude/skills/new-<slug>/` (шаг 4.8) должны буквально совпадать — единственный источник истины это `{{kind_slug}}`. Никаких творческих вариаций («new-mushroom» vs «create-mushroom», «new-observation» vs «new-trip»). Если на фазе 1 пользователь выбрал slug, он зафиксирован до конца — переименовывать на лету нельзя.

11. **Сгенерированный `/new-<slug>` скилл — harness-agnostic.** При заполнении placeholders (`{{body_sections_template}}`, `{{required_links_section}}`, `{{cross_updates_section}}` и т.д.) НЕ использовать CC-specific tool-имена. Конкретно:
    - ❌ Не писать «через AskUserQuestion спросить», «через WebFetch получить», «`mcp__vault-index__vault_query(...)`», «вызови TaskCreate».
    - ✅ Использовать natural-language references: «структурированным multi-choice вопросом спросить», «через web fetch получить», «vault-index MCP, tool `vault_query(...)`», «через TODO-список».
    
    Этот скилл может быть вызван из любого harness (Claude Code или Opencode) и порождает скиллы, которые тоже должны работать в обоих. Полные правила — `docs/Vault_Bootstrap_Architecture.md § Конвенция формата skills → Body content`.

12. **Vision-gate для image-kind (обязательно).** Если на фазе 3.5 включены изображения — три вещи неразрывны и должны примениться вместе:
    - **Seed правил** (шаг 4.5b): секция «Правила скачивания и валидации изображений» обязана быть в `CLAUDE.md` волта (вставить из `templates/image-handling.claude-fragment.md`, если её нет). Без неё ссылки в скилле висят в пустоту.
    - **Баннер** (`{{vision_gate_banner}}`) + **inline-gate** в фото-шаге (`{{gallery_section}}`): генерируемый скилл активно проверяет мультимодальность **pre-flight probe'ом** (Read ассета `.claude/assets/vision_probe.png` ДО любого скачивания), а не декларативно и не пробой на боевом постере.
    - **Запрет эвристики:** ни баннер, ни фото-шаг не должны допускать подбор/классификацию изображений по имени файла, URL или подписи как «запасной путь». Текстовая модель → СТОП или пропуск фото-шага с пометкой, а не брак на эвристике.
    - **`--no-images` целостно (F4):** если фото-шаг ссылается на флаг обхода — он обязан быть в `{{argument_hint}}` **и** иметь шаг детекции (`{{gallery_section}}` п. 0). Упомянуть флаг, но не объявить/не детектировать — хлипкий escape-hatch.
    Цель — новый image-kind ведёт себя ровно как эталонные фото-скиллы волта (`mushroom-card`-семейство), без расхождения.

13. **`USER-ONLY` дисциплина для субъективных полей/секций (F5).** Поля и секции с личной/субъективной оценкой (отзыв, рейтинг, статус прочтения/просмотра, личные впечатления) — это **данные пользователя, не модели**. На фазах 2–3 пометить их `USER-ONLY`; при генерации (шаг 4.8) разметить так, чтобы `/new-<kind>` оставлял их пустыми (плейсхолдер `_Заполняется пользователем._` в теле, `# USER-ONLY` в frontmatter), а не выдумывал. Та же дисциплина, что у `co_authored` и фактчекинга: модель не сочиняет за пользователя то, что он один может знать/оценить. **`USER-ONLY` поле рендерится всегда и не удаляется при создании, даже опциональное (F7)** — иначе пользователь о нём не узнает.

14. **add-note-kind остаётся ОБЩИМ генератором — не сборником частных решений.** Kind-specific логика (проверки согласованности `{{consistency_checks}}`, runtime-guard'ы instance-specific полей, набор секций, типизация связей) **выводится на этапе генерации** из семантики конкретного kind и **живёт в порождённом `/new-<slug>`**, а не хардкодится в этом скилле. В add-note-kind — только общие принципы и мета-инструкции «как вывести» (scope intrinsic/instance, USER-ONLY, единица учёта, как сформировать чеклист самопроверки). Примеры конкретных kind (книга, аниме, гриб) допустимы **только как иллюстрации принципа**, не как зашитые правила. Если ловишь себя на «добавлю проверку специально для kind X в add-note-kind» — стоп: это выводится при генерации и пишется в скилл X. Этот скилл должен одинаково хорошо порождать kind, который ещё никто не видел.

15. **Collision-prevention обязателен у КАЖДОГО генерируемого kind (не выводится — встроен).** В отличие от kind-specific логики (#14), это инварианты безопасности графа/данных, общие для всех kind. Они уже зашиты в шаблон `new-kind-skill.template.md` и канон `CLAUDE.md`; при генерации их **нельзя ослаблять или вырезать**:
    - **Глобальная уникальность basename** — шаг 3 шаблона: Glob `**/<Имя>.md` по всему волту (не только по `{{folder}}`) + СТОП-с-предложением дизамбигуированного имени при коллизии. Obsidian резолвит WikiLinks по basename без учёта папки → одноимённые заметки в разных папках молча ломают граф.
    - **Anti-clobber + ext-by-content-type + cover-cap `|300`** для image-kind — часть `{{gallery_section}}` (см. выше) со ссылкой на канон `CLAUDE.md`. Не перезаписывать чужой ассет молча; расширение из реального типа, не хардкод; cover не растягивать без `|N`.
    Эти правила theme-neutral и применяются к любому kind, включая ещё не виданный. Источник инцидента: слабая модель создала одноимённую заметку в другой папке (граф сломался) и перезаписала чужую обложку (потеря данных) — оба класса теперь предотвращаются на уровне генератора, а не только в рукописных vault-скиллах.

16. **spec-requirements блок ↔ changelog 1:1 (epoch-conformance, ledger-protocol §4.1).** Порождённый `/new-<slug>` несёт managed-блок `spec-requirements` (шаблон, `{{spec_requirements_list}}`) — машиночитаемую декларацию обязательных требований kind. Он и записи `kinds.<slug>` в `SYSTEM/spec_changelog.yaml` (шаг 4.5c) строятся из **одного списка** и обязаны совпадать 1:1. Источник проблемы, которую это закрывает: при **ручной** правке create-скилла (ввод нового поля/секции/format-правила) дисциплина changelog держалась только на памяти агента → молчаливый пропуск, аудит ложно штрафует старые карточки. Теперь: блок делает требования явными, `vault_spec_drift` (vault-index MCP) детерминированно ловит рассинхрон блок↔changelog, а в Claude Code правка блока триггерит PostToolUse-reminder (harness-claude-code). **Не выкидывать блок при генерации и не давать ему разойтись с changelog.** В Opencode live-reminder недоступен (upstream #13574) — там инвариант держит только backstop `vault_spec_drift`, поэтому он критичен.
