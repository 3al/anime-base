# /new-note — Создание новой заметки

Создаёт новую заметку, определяя её тематическую папку через manifest-driven routing.

## Аргумент

`$ARGUMENTS` — тема заметки, опционально с комментарием через пробел.

Парсинг: первое слово (или словосочетание в кавычках) — тема. Всё остальное — комментарий пользователя.

Примеры:
- `/new-note Docker Compose` — создать заметку
- `/new-note "Dependency Injection" с примерами на Python` — создать с акцентом
- `/new-note LSP` — создать заметку о Language Server Protocol

## Источники правил

Перед созданием прочитай:

1. **`.claude/vault-manifest.yaml`** — секция `folders` (источник routing-таблицы)
2. **`SYSTEM/Metadata_schema.md`** — обязательные поля frontmatter
3. **`SYSTEM/tag_taxonomy.yaml`** — канон тегов (SSOT); правила + критерий приёма — `.claude/skills/audit-by-creator/references/tag-discipline.md`
4. **`SYSTEM/Linking_guidelines.md`** — правила WikiLinks
5. **`SYSTEM/Naming_conventions.md`** — формат имён файлов
6. **`SYSTEM/Vault_architecture.md`** — описание структуры волта (для пользователя; манифест — для машин)

## Алгоритм

### 1. Проверить существование

Найти через Glob по теме. Если заметка уже существует:
- **Stub / тонкая (<20 строк контента)** — перенаправить: «Заметка уже есть как stub — используйте: `/expand-stub <файл>`»
- **Наполненная (>50 строк)** — сообщить: «Заметка уже существует: `<путь>`. Для аудита: `/audit-note <файл>`»

### 2. Routing — выбор папки

**2.1. Прочитать `.claude/vault-manifest.yaml`** и распарсить секцию `folders`. Формат:

```yaml
folders:
  - path: SOMEFOLDER/
    description: "Что хранится в этой папке"
    keywords: [keyword1, keyword2]      # опционально
    create_skill: /new-something        # опционально, тематический create-handler
```

**2.2. Если `folders` пуст или отсутствует** — перейти к **подпрограмме «Folders not declared»** (см. ниже).

**2.3. Классификация:**
- Сопоставить тему + комментарий пользователя с описаниями (`description`) и ключевыми словами (`keywords`) каждой папки.
- Использовать **семантическое понимание**, не только keyword match: тема «Жульен из лисичек» подходит к COOKING/, даже если в keywords нет «жульен».

**2.4. По результату классификации:**

| Сценарий | Действие |
|---|---|
| Однозначное совпадение, у папки есть `create_skill` | **Delegate**: «Эта заметка относится к `<path>`, используйте `<create_skill> <тема>`». Skill завершается. |
| Однозначное совпадение, без `create_skill` | Идти на шаг 3 с этой папкой. |
| Несколько кандидатов с близкой релевантностью | **Спросить пользователя** структурированным multi-choice вопросом: показать кандидатов с обоснованиями, запросить выбор. |
| Никакая папка явно не подходит | Перейти к **подпрограмме «Propose new folder»**. |

### Подпрограмма «Folders not declared»

Если `folders` пуст:
1. Сообщить пользователю: «В манифесте нет объявленных папок. Перед созданием заметки нужно выбрать или создать папку».
2. Просканировать существующие top-level директории волта (исключая `.claude/`, `.obsidian/`, `attachments/`, `SYSTEM/`).
3. Если найдены подходящие — предложить пользователю одну из них для текущей темы.
4. Если ничего не найдено — перейти к **«Propose new folder»**.
5. По результату — добавить запись в `folders` манифеста (см. подпрограмму «Register folder»).

### Подпрограмма «Propose new folder»

1. На основе темы предложить:
   - Имя папки (по `Naming_conventions.md`: `UPPER_CASE` для root-папок).
   - Краткое description (одна строка).
   - Обоснование: почему ни одна существующая не подходит.
2. Структурированным multi-choice вопросом запросить подтверждение / коррекцию.
3. После подтверждения — выполнить **«Register folder»** и продолжить с шага 3 алгоритма.

### Подпрограмма «Register folder»

1. Прочитать `.claude/vault-manifest.yaml`.
2. Добавить запись в `folders[]`:
   ```yaml
   folders:
     - path: NEW_PATH/
       description: "<согласованное описание>"
   ```
3. Сохранить манифест с сохранением порядка остальных полей (используй простой YAML-парсер или построчное редактирование с минимальной модификацией).
4. Прочитать `SYSTEM/Vault_architecture.md`, добавить строку в folder-таблицу.
5. Создать саму папку (`mkdir`).
6. Сообщить пользователю что папка зарегистрирована.

### 3. Определить имя файла

По правилам из `Naming_conventions.md`: `PascalCase_With_Underscores.md`. Без пробелов, без кириллицы. Кириллическое название — в `aliases`.

### 4. Определить тип и домен

На основе темы и комментария + `Metadata_schema.md`:
- **type**: concept, guide, reference, principle, rule, pattern, project (или другие, разрешённые в schema)
- **domain**: — поле обозначает тематическую область; смотри в `Metadata_schema.md` какие значения допустимы; если тема не вписывается в существующие domain'ы — спросить пользователя (нужен ли новый domain).

Если есть **комментарий пользователя** — учитывать как приоритетное указание для типа, домена, акцентов.

### 5. Собрать контент

- Если тема — **внешний софт/библиотека/протокол**: web search для актуальных данных. Если доступен MCP `context7` и тема — библиотека/фреймворк — использовать его.
- Если тема — **принцип/паттерн**: писать из знаний, с примерами кода где уместно.
- Если тема — **формат**: синтаксис, примеры, частые ошибки.

Если есть **комментарий пользователя** — запрошенные темы/секции включить обязательно. Акценты пользователя важнее стандартной структуры.

#### Масштаб

- **Узкий концепт** (один термин, один принцип) — 30–60 строк
- **Средняя тема** (конкретный инструмент, принцип с примерами) — 80–150 строк
- **Большая тема** (большая система, фреймворк, протокол) — 150–200+ строк

#### Структура по типу

**type: concept** — определение, как работает, применение, связь с другими концептами, ограничения

**type: principle** — формулировка, зачем нужен, примеры (хорошие и плохие), антипаттерны, связанные принципы

**type: guide** — назначение, установка/настройка, использование, частые проблемы

**type: reference** — описание, ключевые характеристики, сравнение с аналогами, ссылки

**type: pattern** — проблема, решение, когда применять, когда не применять, примеры

Тон: информативный, лаконичный. Без воды — каждая секция несёт ценность.

### 6. Собрать frontmatter

Заполнить обязательные поля + рекомендуемые:

```yaml
---
type: <определённый тип>
domain: <определённый домен>
stability: <stable|evolving|experimental>
priority: <high|medium|low>
quality: draft
tags:
  - <каноничные теги из SYSTEM/tag_taxonomy.yaml, max 10>
aliases:
  - <если имя файла ≠ естественное название>
created: <сегодня>
updated: <сегодня>
---
```

**НЕ ставить `co_authored` при создании.** Поле проставляется только через `/verify` или `/audit-note`, когда пользователь явно подтверждает контентный пас модели. На этапе `/new-note` авторство модели формальное — заметка ещё не верифицирована.

**Теги подбирать по алгоритму из `Audit_checklist.md`** — пошагово:
1. КАТЕГОРИЯ верхнего уровня
2. ДОМЕН/ВЕНДОР
3. ТЕМЫ содержания
4. ОБЯЗАТЕЛЬНЫЕ по `note_kind`
5. НЕГАТИВНАЯ ПРОВЕРКА — убрать тег, совпадающий с именем заметки; убрать дубли с type/domain
6. **РАСШИРЕНИЕ КАНОНА — через гейт (критерий D).** Для **каждого** нового тега, которого нет в `SYSTEM/tag_taxonomy.yaml`:
   - Применить критерий приёма D (`.claude/skills/audit-by-creator/references/tag-discipline.md`): переиспользуемая ось по природе ∧ ортогональна enum-полям ∧ описывает содержание. Не прошёл → тег не заводить.
   - Прошёл → добавить **структурную запись** в `SYSTEM/tag_taxonomy.yaml` (Edit): `- name: <tag>` + `group:` + `description:`. **Не** править таблицы `Tag_taxonomy.md` (генерируются). Затем перегенерировать md (`core/lib/tag_taxonomy_render.mjs` / `/init-vault --update`).
   - **Никаких «рекомендуется пользователю добавить»** в отчёте — канон и теги синхронны по контракту. Скилл нашёл нужду → скилл сам и записал в yaml.

Если после шагов 1–3 получилось <3 тегов — пересмотреть, скорее всего пропущена категория.

### 7. Добавить ссылки

По правилам из `Linking_guidelines.md`:
- Обязательные ссылки по `type`
- Ссылки на связанные заметки волта (найти через Grep/Glob)
- Без дубликатов, только первое упоминание
- Все target должны существовать

### 8. Создать файл

Записать в выбранную папку.

### 9. Самоаудит

Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** для созданной заметки. Если найдены ошибки — исправить. Если предупреждения — упомянуть в отчёте.

### 10. Сообщить результат

Показать: имя файла, путь, тип, число строк, ссылки, результат аудита. Предложить `/verify` если устраивает.

## КРИТИЧНО

1. **Никогда не создавать заметку вне зарегистрированных папок**. Если ни одна не подходит — сначала зарегистрировать новую через подпрограмму выше.
2. **Не создавать заметку если есть `create_skill`** для подходящей папки — всегда delegate.
3. **При регистрации новой папки** — обновить и манифест, и `Vault_architecture.md`. Они должны быть согласованы.
4. **Не редактировать manifest вне секции `folders`** — другие секции (modules, config, language) принадлежат другим скиллам.
