# /audit-note — Полный аудит заметки

Проверяет заметку по всем правилам vault, перепроверяет факты, улучшает содержание, оценивает качество модели-автора (в леджер) и выдаёт отчёт.

## Аргумент

`$ARGUMENTS`:
- **Имя/путь заметки** (обязательно). Без пути — найти через Glob.
- `--report-only` — никаких правок: только отчёт (+ оценка/запись в леджер, если не `--no-ledger`). Контент, frontmatter, `quality`/`updated` не мутируются. Dry-run для безопасного теста.
- `--no-ledger` — выполнить аудит без записи в леджер (ручной разовый прогон / пауза леджера). Симметрично `audit-by-creator`.
- `--no-network` — пропустить network-шаг проверки внешних ссылок (Шаг 4.6). По умолчанию шаг выполняется при наличии сети; флаг отключает его явно (offline-прогон / не дёргать внешние сайты).

Флаги композируются: `--report-only --no-ledger` = чистый отчёт без мутаций и без телеметрии.

## Алгоритм

1. Найти файл по `$ARGUMENTS` (Glob если нужно)
2. Прочитать файл. **Прочитать на входе** (для леджера, шаг 8): `co_authored` = origin-автор (субъект оценки) и `content_hash` по контенту ДО любых правок. **Этот скилл `co_authored` НЕ перезаписывает** (см. «Порядок авторства» ниже) — он судья, не автор.
3. Прочитать `SYSTEM/Audit_checklist.md` и выполнить **все шаги** из него — буквально, **со следующим переопределением** (см. «Порядок авторства» ниже): **НЕ выполнять установку `co_authored` из `§4.8 «Результат»` вообще** (аудит не авторствует). `quality`/`updated` из §4.8 — только в конце (шаг 10), и только при наличии правок. Остальные шаги:
   - **Шаг 1**: lint — MCP-tool `vault_lint` с `target: <файл>`. Если в `.claude/vault-manifest.yaml` задан `link_cap` — передать его как `linkCap` (число → порог `too-many-links`; `null` → отключить проверку). Поле отсутствует → параметр не передавать (дефолт 15).
     - Прокинуть lint-backstop ключи манифеста, если заданы (одноимённые camelCase-параметры): `cover_field`/`cover_embed_suffix` → `cover-ref-mismatch`; `name_surface_pairs` → `name-surface-mismatch`; `required_tags_by_kind` → `missing-required-tag`; `max_tags` → `maxTags` (`too-many-tags`, дефолт 10); `user_only_sections`/`user_only_stub_whitelist` → `user-only-fabricated`; `prose_script` → `mixed-script-prose`; `tooling_vocab.field_names`/`tooling_vocab.state_phrases`/`tooling_vocab.flag_skill_commands` → `toolingVocabFieldNames`/`toolingVocabStatePhrases`/`toolingVocabFlagSkillCommands` (`tooling-vocab-in-prose`). Отсутствующие — не передавать.
     - В выводе различать класс: `summary.structural_green` (нет structural-ERROR) — главный детерминированный сигнал. `empty-tags` (structural WARN) — детерминированный сигнал пустого `tags`. Это **structural-дефект**: скорится на structural-оси (`audit-by-creator`, `audit_type: structural`), а на **контентной — НЕ скорится** (иначе один root-cause штрафуется дважды — cross-axis double-jeopardy; оси разведены по `audit_type` именно чтобы дефект считался один раз на оси своего класса, `ledger-protocol §3`, §4 cross-axis). На контентной оси `empty-tags` можно **разрешить как улучшение** (добавить теги по Шагу 3.2 чеклиста), но в content-`score` (Шаг 7) **не включать** — это не контентная находка. **`heuristic`-поток** (`mixed-script-prose`, `user-only-fabricated`, `tooling-vocab-in-prose`) — это детерминированные бэкстопы к ручным шагам аудита: `mixed-script-prose` подсвечивает кандидатов для §4.0 (языковой мусор), `tooling-vocab-in-prose` — кандидатов для §4.0/§4.3 (внутренний bookkeeping волта в прозе: имена полей, `карточек нет`, имена скиллов), `user-only-fabricated` — кандидата для §4.2 (фабрикация субстанции). Это **нечёткие WARN**: верифицировать восприятием, не доверять слепо; находки разрешать на соответствующем контентном шаге, а не штамповать.
   - **Шаг 1.5**: ссылки — MCP-tool `vault_broken_links` и `vault_duplicate_links`. Также вручную проверить: нет ли WikiLinks в заголовках (`##`, `###`)
   - **Шаг 2**: прочитать governance-документы (`SYSTEM/Metadata_schema.md`, `SYSTEM/tag_taxonomy.yaml` — канон тегов; правила/критерий — `.claude/skills/audit-by-creator/references/tag-discipline.md`; `SYSTEM/Linking_guidelines.md`)
   - **Шаг 3**: семантические проверки (frontmatter, теги, ссылки, имя файла)
   - **Шаг 4**: контентный аудит — **это ключевой шаг**:
     - **Шаг 4.0**: очистка языкового мусора — убрать из текста:
       - Вкрапления CJK-символов (китайские иероглифы, японские хирагана/катакана, корейские хангыль), которые не являются частью осознанного названия на оригинальном языке
       - Случайные Unicode-артефакты (零宽字符, 全角半角混用 и т.п.)
       - Ложные «китайские» апострофы, скобки, пробелы (全角空格, 全角逗号 и т.п.), заменяя их на ASCII-аналоги
       - Опечатки в markdown-разметке (лишние `*`, `_`, неверные заголовки, битые списки)
       - Опечатки и стилистические ошибки в тексте (опечатки, повторы слов, лишние пробелы перед знаками препинания)
     - Web search для проверки фактов (даты, цифры, версии)
     - **Шаг 4.2 — смысловой фактчек СОДЕРЖАНИЯ (приоритетный, НЕ опциональный).** Контентная ось первым проходом ловит механику (мусор/теги/даты/`featured_in`-якорь), но систематически пропускает **фабрикацию субстанции**. Поднять в ядро как обязательный шаг:
       1. **Фактчек описаний связей** в `## Связи` (и любой связевой секции) — не только существования линка, а **описания** связи. Для каждой связи прочитать карточку цели/источника и сверить характер связи (`Magnolia как «коллеги CH»` — ложь, если по карточкам они клиенты). Ложная связь = **дефект контентной оси** (как `WRONG_FORMAT` в находках): исправить/удалить описание, **НЕ пропагировать** ложную связь в реципрокность (`/fix-links asymmetric` не должен достраивать обратную ногу к фабрикации).
       2. **Substance-grounding прозы** — характеристики, биография, плот должны быть подтверждаемы (web/source), а не правдоподобный вымысел. Неподтверждаемую субстанцию — удалить или пометить, как и любой недоказанный факт.
       - **ЖЁСТКО: никакой новой оси/score-измерения.** Это остаётся `audit_type: content`, находки якорятся в ту же контентную формулу (`ledger-protocol.md §4`). Усиливается **процедура судьи** (этот framing), а не схема леджера. Отдельной «оценки качества прозы» не заводить.
     - **Шаг 4.5 — обязательный prose-clarity пасс (НЕ опциональный).** Однопроходный судья систематически пропускает мутные/невнятные формулировки (durable-фикс — форсинг-чеклист, а не «постараюсь внимательнее»). Пройти **каждую** прозовую секцию явно и проверить: нет ли смазанных, двусмысленных, ничего-не-говорящих фраз; читается ли утверждение однозначно. Мутную фразу — переписать ясно или удалить. Детерминированный бэкстоп к технической части мусора (`tooling-vocab-in-prose` из Шага 1) покрывает имена полей/скиллов/`карточек нет`; этот пасс покрывает **смысловую внятность**, которую regex не ловит. **Без новой оси/score** (как §4.2): остаётся `audit_type: content`, усиливается процедура судьи.
     - **Шаг 4.6 — внешние ссылки: liveness + адекватность (network, opt-in).** Выполняется при наличии сети; пропустить при флаге `--no-network` (явно пометить в отчёте «network-шаг пропущен»). Для каждого внешнего URL заметки — URL-поля frontmatter (`mal_url`, `anilist_url` и любые `*_url`) и ссылки в связевых секциях (`## Ссылки` и т.п.):
       1. **Liveness (детерминированно, но сетезависимо).** web fetch → HTTP 200/3xx — живой; 4xx/5xx/таймаут — в отчёт как кандидат «мёртвая ссылка». **Сетезависимо и флапает** (Fandom отдаёт 403 даже валидному fetch): помечать «возможен флап», **не** штамповать мёртвой по одному ответу. **НЕ гейтит `structural_green` и НЕ пишется в леджер как structural-дефект** (liveness — не детерминированный offline-сигнал).
       2. **Адекватность (эвристика §4.2-класса).** Страница ведёт на **того** субъекта: id в URL (`character/<id>`) совпадает с frontmatter (`mal_url`/`anilist_url`), и контент страницы — про этого субъекта. Несовпадение id / чужой субъект = **контентный дефект** (как ложная связь в §4.2): исправить или удалить URL, не пропагировать.
       - При `--report-only` — только зафиксировать находки для отчёта, без правок.
     - **Шаг 4.7 — осмысленность тега (HEURISTIC-флаг, критерий D, БЕЗ score/гейта).** Структурную тег-консистентность (canon-membership, facet-пропагация) судит `/audit-by-creator` (structural-ось, §4.2) — **здесь её не дублировать** (cross-axis double-jeopardy). Контентная ось добавляет только **нечёткое суждение**, которого детектор не даёт: достоин ли тег канона по **критерию D** (`.claude/skills/audit-by-creator/references/tag-discipline.md §3`) — переиспользуемая ось по природе vs сущность-узкий шум. Сигнал-кандидаты (НЕ вердикт) — `vault_tag_health`: `singletons` (count==1) и `ghost` (канон с 0 использований) подсвечивают теги на ревизию; счёт — **слабый сигнал** (широкая ось может быть синглтоном в малом корпусе — не демоутить по счёту). Вердикт «сущность-узкий ярлык → демоут / широкая ось → оставить» — суждение судьи по D.
       - **ЖЁСТКО: флаг, не правка и не score.** Это **vault-долг канона** (демоут/чистка тега — решение по всей таксономии, не по одной карточке), не author-defect. Отдельная секция отчёта `TAG-MEANINGFULNESS (кандидаты канона, не в score)`; в content-`score` **не входит**, `structural_green` **не гейтит**, новой оси НЕ заводить (как §4.2/§4.6). Демоут тега из канона — отдельное осознанное действие (правка `tag_taxonomy.yaml` + регенерация md), не молчаливая правка карточки.
     - Удаление домыслов, разговорного стиля, дублирования
     - Дополнение пропущенных важных аспектов
     - Улучшение формулировок
4. **Применить все правки** к файлу (frontmatter + содержание) — **кроме** метаданных ревизии (`quality`/`updated`), они идут последним шагом. **`co_authored` не трогать вовсе** (аудит не авторствует). **При `--report-only` правки НЕ применять** (только зафиксировать находки для отчёта и леджера).
5. **Type-conditional проверки** (если в `Audit_checklist.md` определены чек-листы для конкретных `note_kind`):
   - Определить `note_kind` из frontmatter заметки
   - Если в checklist'е есть секция для этого `note_kind` — выполнить её шаги
   - Иначе пропустить (skill универсален; type-specific логика — в checklist'е, не в этом скилле)
6. **Пост-редакционная валидация**:
   - Повторно вызвать MCP-tools `vault_broken_links` и `vault_duplicate_links`
   - Проверить: нет ли WikiLinks в заголовках
   - Для `[[Target|text]]` — убедиться что Target соответствует display text
7. **Вычислить оценку 1–10** по контентной оси из находок аудита (ошибки/предупреждения как дефекты). Формула — `.claude/skills/audit-by-creator/references/ledger-protocol.md §4` («якорь к находкам»: штраф от базы, число выводится из находок, не из «ощущения»).
8. **Записать в леджер** (`audit_type: content`), **если не `--no-ledger`**, по `.claude/skills/audit-by-creator/references/ledger-protocol.md §2–§5`: `authored_model` = **нормализованный** (§3.1) origin-автор из `co_authored` (с шага 2), `authored_model_raw` = сырое значение; `judge_model` = нормализованная текущая модель; `score`, `findings` (вкл. `schema_drift` — обычно 0 для контентной оси, но если требуемая секция введена в спеку после `created`, применить epoch-реклассификацию §4.1), `reasons`, `content_hash` (с шага 2), `invariants_total`; `fixes_applied` = были ли правки на шагах 4–6 (при `--report-only` → `false`); `pristine` = нет ли более ранней записи по этой `note` с `fixes_applied=true` (§5.B). Дедуп точных дублей (§5.A).
9. **Сформировать отчёт** (формат из чеклиста + краткое описание контентных изменений).
10. **Метаданные ревизии — ПОСЛЕДНИМ шагом, БЕЗ авторства** (порядок по `ledger-protocol.md §6`). **`co_authored` не трогать никогда** (аудит — судья, не автор; его след — в `ledger.judge_model`). **Только если на шагах 4–6 реально были правки** (т.е. не `--report-only` и было что менять): `quality: draft` (всегда draft — пользователь подтвердит через `/verify`), `updated: <сегодня>`. Правок не было / `--report-only` → frontmatter не менять.

## Порядок авторства (override §4.8 чеклиста — КРИТИЧНО)

Это переопределение **обязательно** и имеет приоритет над текстом `SYSTEM/Audit_checklist.md`:

- **Аудит НЕ авторствует.** `/audit-note` ревьюит и улучшает, но **не становится автором** карточки. Поэтому `co_authored` он **не перезаписывает вообще** — ни в середине (как велит устаревший `§4.8` vault-owned чеклиста), ни в конце. Носитель актуального правила — **этот SKILL.md** (managed, ресинкается), он переопределяет чеклист.
- **Почему:** перезапись `co_authored` на судью **уничтожает субъект оценки** для последующих аудитов. Пример: `/audit-note` затёр `co_authored = opus` → `/audit-by-creator` затем записывает структурные косяки слабой модели **на opus**. Capture-at-entry это не спасает (живёт один прогон). Единственный фикс — origin-автор **никогда** не затирается аудитом; след судьи живёт в `ledger.judge_model`.
- **Серая зона §4.4** («дописать секции»): даже там `co_authored` остаётся origin, а правка ставит `fixes_applied=true` → последующие замеры карточки помечаются `pristine=false` и не засчитываются автору (`ledger-protocol.md §5–§6`). Настоящее авторское дописывание тонкого стаба — путь `/expand-stub` (он производит контент и ставит `co_authored`), не аудит.
- Правильный порядок (он же `ledger-protocol.md §6`): **(2)** прочитать origin-автора на входе → **(3–6)** аудит и правки (`co_authored` не трогая) → **(7–8)** оценка и запись в леджер → **(10)** только `quality: draft` + `updated`, если были правки.

## Важно

- Аудит — это не просто lint. Скилл должен **улучшить заметку**, а не только найти проблемы.
- Факты, не подтверждённые web search — удалить или пометить.
- `quality` после правок — **всегда `draft`**, даже если заметка была `verified` до аудита. Если правок не было — `quality`/`updated` не трогать вовсе (см. шаг 10). `co_authored` не трогать **никогда** (аудит не авторствует).
- Леджер пополняется **попутно**, как побочный эффект обычного аудита (не отдельный скилл, не background-инфраструктура). Схема записи, нормализация ID, формула score, дедуп/контаминация (`pristine`) и правило «аудит не авторствует» — **единый источник** `.claude/skills/audit-by-creator/references/ledger-protocol.md`, не дублировать здесь. `audit_type: content` отличает контентную ось от структурной (`audit-by-creator`).
- Type-conditional шаги (Шаг 5) определяются в `Audit_checklist.md` волта, не в этом скилле. Если волт хочет специфичный аудит для своих `note_kind` (например, `mushroom-card` — проверка фотографий) — это правится в checklist'е.
- После отчёта спросить: «Устраивает? /verify для подтверждения».
