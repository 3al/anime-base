# /vision-gate-audit — Ретрофит vision-gate в рукописные image-скиллы

Доводит **рукописные тематические скиллы**, работающие с изображениями, до соблюдения **vision-gate** — конвенции, что классификация/подбор/визуальная проверка фото возможны только при реальном восприятии картинки мультимодальной моделью.

**Зачем это нужно.** Канон image-handling засевается автоматически только в скиллы, сгенерированные через `/add-note-kind` (фаза 3.5). **Рукописные** create/enrich/audit-скиллы (`new-<entity>`, `add-image`, `audit-images`, `fix-*` с фото) его не получают — и не-мультимодальная модель (GLM/Qwen/DeepSeek) встраивает изображение вслепую по имени файла/URL, что канон прямо запрещает. Это частный случай общей проблемы: **framework-конвенции не доходят до pre-existing/hand-written тематических скиллов** (фреймворк намеренно не трогает unmanaged-скиллы). Скилл закрывает разрыв через явный ретрофит-проход с подтверждением.

**SSOT канона — секция `## Правила скачивания и валидации изображений` в `CLAUDE.md` волта** (always-on контекст). Скиллы **ссылаются** на неё, не дублируют. Этот аудитор: (1) гарантирует, что секция в CLAUDE.md есть и актуальна; (2) гарантирует, что каждый image-touching скилл имеет баннер + активный gate-шаг, ссылающийся на канон.

См. также: `docs/Theme_Skills_Patterns.md § Vision-gate (скиллы с изображениями)`, `modules/skills-common/skills/add-note-kind/templates/image-handling.claude-fragment.md` (источник canonical-текста секции).

## Аргумент

`$ARGUMENTS` (опционально):
- Пусто — полный аудит волта (default).
- Имя файла / путь — аудит одного скилла.
- Glob — аудит подмножества (например, `.claude/skills/new-*`).

## Scope

**Сканируется:**
- `.claude/skills/<name>/SKILL.md` — **только без `.managed` маркера** (managed принадлежат фреймворку, перетрутся на `/init-vault`; gate в них уже встроен авторами фреймворка).
- `CLAUDE.md` — проверка/засев canonical-секции image-handling.

**НЕ сканируется:**
- `.claude/modules/` — source-копии модулей.
- `.obsidian/`, `attachments/` — конфиг и бинари.
- Content-папки (`MUSHROOM/`, `CHARACTERS/` и т.п.) — это заметки, не скиллы.
- Managed-скиллы skills-common — gate уже на месте.

## Что считается «image-touching» скиллом

Скилл попадает в scope, если в SKILL.md есть **сигналы работы с изображениями** (искать любой):
- Скачивание/вставка картинок: `curl`, `![[`, `.jpg`/`.jpeg`/`.png`/`.webp`/`.gif`, `attachments/`.
- Лексика галереи/фото: «галере», «миниатюр», «обложк»/«постер»/«cover», «фото», «изображени», «image», «gallery», «cover».
- Frontmatter/описание с `--no-images` / `-n` флагом.
- Назначение скилла из `description`: создание карточки с галереей, `add-image`/`add-photo`, `audit-images`, `fix-*` с фото в таблицах.

Скилл **вне scope** (не флагать), если изображений не касается вовсе (например, `add-found-species`, `rename-note`).

## Что считается «есть vision-gate»

Скилл уже соответствует (НЕ флагать), если присутствуют **оба**:
1. **Баннер** vision-gate сверху — `> [!important]` / `Vision-gate` / явная отсылка к секции `## Правила скачивания и валидации изображений` CLAUDE.md.
2. **Активный gate-шаг** в точке начала работы с фото — формулировка вида «открой первое фото через Read, убедись что воспринимаешь визуальный факт; не воспринимаешь → СТОП / пропуск фото-шага с пометкой, без классификации по имени/URL».

Если скилл image-touching, но один из двух отсутствует → **флаг на ретрофит**.

### Drift: gate есть, но инлайнит канон (мягкий флаг)

Отдельный класс — gate **присутствует** (баннер + шаг есть, idempotency не вставляет повторно), но скилл **дублирует тело канона** вместо ссылки: инлайнит правила скачивания/валидации (`curl -f`, «проверить тип/размер», обработку `"Could not process image"`) или ссылается на **старый путь** `add-note-kind/templates/image-handling.claude-fragment.md` вместо секции в `CLAUDE.md`. Это не разрыв gate, а **риск дрейфа** «копия vs канон» при будущей эволюции правил.

→ Мягкий флаг: предложить **слимить инлайн до ссылки** на `## Правила скачивания и валидации изображений` в `CLAUDE.md` и **репойнтнуть старый путь** на этот SSOT. Не навязывать — показать как опциональную правку; владелец мог инлайнить намеренно. Если gate работает и владелец отказался — оставить как есть (gate важнее чистоты ссылки).

## Алгоритм

### Этап 1. Discovery

**1.1.** Собрать non-managed скиллы: для каждой `<vault>/.claude/skills/<name>/` без `.managed` → `SKILL.md`. Если задан `$ARGUMENTS` — заменить discovery на путь/glob.

**1.2.** Для каждого — определить image-touching (сигналы выше). Не-image-touching отбросить.

**1.3.** Прочитать `CLAUDE.md`, проверить наличие секции `## Правила скачивания и валидации изображений` и её актуальность относительно canonical-текста (см. Этап 3).

**1.4.** Сводка пользователю: «Image-touching скиллов: N. Из них без vision-gate: M. CLAUDE.md-секция: есть/актуальна | дрифт | отсутствует».

### Этап 2. Per-skill confirmation loop

Для каждого флагнутого скилла (детерминированный порядок — по имени):

**2.1.** Прочитать SKILL.md целиком.

**2.2.** Определить **тип** скилла для корректной формы gate (по каталогу `Theme_Skills_Patterns.md`):
- **Целиком про фото** (create с обязательной галереей без `--no-images`, `add-image`, `audit-images`) → при текстовой модели **остановиться**.
- **Фото — лишь часть** (audit заметки, дополнение, fix с фото в таблицах) → текстовую часть сделать, фото-шаг **пропустить** с пометкой.

**2.3.** Собрать предлагаемый ретрофит:
- **Баннер** сразу после заголовка/frontmatter:
  ```markdown
  > [!important] Vision-gate
  > Работа с изображениями (скачивание, подбор, классификация, визуальная проверка) подчиняется секции `## Правила скачивания и валидации изображений` в `CLAUDE.md`. Перед фото-шагом пройди активную проверку мультимодальности — не классифицируй по имени файла/URL/подписи.
  ```
- **Активный gate-шаг** — вставить в точке, где скилл начинает качать/подбирать/оценивать фото (найти этот шаг в теле; если шагов несколько — перед первым). Текст по типу скилла (2.2):
  - флаг `--no-images`/`-n` (если скилл его поддерживает) → пропустить фото целиком, gate не нужен;
  - иначе: скачать+провалидировать (тип+размер) первое изображение, открыть через Read, извлечь конкретный визуальный факт; воспринимаешь → продолжай; не воспринимаешь → СТОП (целиком-про-фото) или пропуск фото-шага с пометкой (фото-часть), **без** эвристики по имени/URL.
- **Ссылку на канон**, не копию правил скачивания/валидации (они в CLAUDE.md).

**2.4.** Показать **per-skill diff** через структурированный multi-choice вопрос:
```
Скилл: <path>  (тип: целиком-про-фото | фото-часть)
Вставки: баннер (после строки X), gate-шаг (перед строкой Y).
[A] Применить  [B] Показать полный текст вставок  [C] Пропустить скилл  [D] Прервать
```

**2.5.** Применить через Edit (вставка баннера + gate-шага). Если точку gate-шага не удалось однозначно найти — вставить gate как отдельный нумерованный шаг в начало секции работы с фото и пометить «проверь размещение вручную».

**2.6. Drift-проход (для скиллов, где gate уже есть).** Если детектнут инлайн канона или ссылка на старый путь (см. «Drift» выше) — отдельным **опциональным** вопросом предложить: заменить инлайненные правила скачивания/валидации на одну ссылку на `## Правила скачивания и валидации изображений` в `CLAUDE.md`; репойнтнуть `add-note-kind/templates/image-handling.claude-fragment.md` → секция CLAUDE.md. `[A] Слимить до ссылки [B] Оставить инлайн (намеренно) [C] Пропустить`. Дефолт при сомнении — спросить, не трогать молча.

### Этап 3. CLAUDE.md canonical-секция

**3.1.** Canonical-текст секции = содержимое `add-note-kind/templates/image-handling.claude-fragment.md` (источник истины). Воспроизвести его в волте дословно.

**3.2.**
- Секции в CLAUDE.md **нет** → вставить (в конец, после module sub-blocks). Отметить `inserted`. Если CLAUDE.md нет — предупредить, предложить создать.
- Секция **есть**, дрифтнула от canonical → предложить обновить (показать diff). Отметить `updated`.
- Идентична → `unchanged`, noop.

**3.3.** Перед записью — подтверждение структурированным multi-choice вопросом (`[A] Применить [B] Показать diff [C] Пропустить`).

> Примечание: засев этой секции выполняет и `/add-note-kind` (фаза 4.5b) при создании image-kind. Этот аудитор покрывает волты, где image-скиллы рукописные и `/add-note-kind` секцию не сеял.

### Этап 4. Summary

```
/vision-gate-audit — Сводка

Image-touching скиллов: N
  Уже с vision-gate: K
  Ретрофит применён: A
  Пропущено по выбору: B
  Размещение gate проверить вручную: C
  Инлайн канона слимлено до ссылки (drift-проход): E

Изменённые файлы:
- .claude/skills/<name>/SKILL.md (баннер + gate)
- CLAUDE.md (секция)

Следующий шаг: git diff для ревью; прогнать обновлённый скилл на одном примере (само-проверка по Theme_Skills_Patterns § Workflow 5).
```

Если применений > 0 — напомнить про коммит в волте с git-управлением.

## КРИТИЧНО

1. **Не применять без подтверждения.** Каждый скилл с разрывом → структурированный вопрос. Никаких silent rewrites.
2. **Не трогать managed-скиллы** (`.managed` в директории). Gate в них уже встроен фреймворком; правка перетрётся на `/init-vault`.
3. **SSOT — в CLAUDE.md.** Скиллы получают баннер + gate-шаг + **ссылку** на секцию, НЕ копию правил скачивания/валидации. Дублирование канона в каждый скилл = drift при эволюции правил. Не создавать конкурирующий `SYSTEM/Image_handling.md`.
4. **Idempotency.** Повторный прогон на уже-ретрофикнутом волте → «всё с vision-gate, изменений не требуется». Баннер/gate детектить по сигналам (см. «Что считается есть vision-gate»), не вставлять второй раз.
5. **Тип скилла определяет форму gate.** Целиком-про-фото при текстовой модели → стоп; фото-часть → пропуск фото-шага с пометкой. Не навязывать «стоп» там, где текстовая часть полезна.
6. **Не дублировать с `/harness-agnostic-audit`.** Тот занимается портативностью tool-имён; этот — vision-gate. Разные концерны, разные проходы.
7. **Не превращать в линтер.** Разовая миграция / периодический аудит, не run-on-save.

## Известные ограничения

- **Эвристика image-touching** может дать ложные срабатывания (скилл упоминает «фото» в проходящем контексте, но реально с картинками не работает) — в сомнительных случаях спросить пользователя, флагать ли.
- **Размещение gate-шага** LLM-driven: если структура скилла нестандартна, точку вставки стоит проверить вручную (помечается в summary).
- **Не покрывает не-managed скиллы из других волтов** — работает только в текущем волте.
