# /vault-rag — RAG-роутер L1 + L2 над Obsidian-волтом

## Аргумент

`$ARGUMENTS` — аналитический запрос на русском или английском. Примеры:
- «сравни Macrolepiota Procera и Chlorophyllum Rhacodes по риску спутывания»
- «обзор условно-съедобных грибов Подмосковья требующих термообработки»
- «двойники в роде Lepiota»
- «что в волте есть про осенние пластинчатые с белыми пластинками»

Если `$ARGUMENTS` пустой — попроси пользователя сформулировать запрос и выйди.

## Контракт (читать ОБЯЗАТЕЛЬНО перед запуском)

Скилл — **L3-роутер** в трёхслойной RAG-архитектуре (см. фрагмент `vault-semantic` в CLAUDE.md и `docs/RAG_Architecture.md` если есть в волте).

- **L1 = `vault-index` MCP.** Structural: frontmatter-фильтры, граф ссылок, lint. Tools: `vault_query`, `vault_backlinks`, `vault_note_profile`, `vault_lookalike_peers`.
- **L2 = `vault-semantic` MCP.** Hybrid search (bge-m3 dense + BM25 с pymorphy3/snowball лемматизацией, RRF-fusion). Tools: `vault_semantic_search`, `vault_semantic_stats`, `vault_semantic_reindex`, `vault_semantic_warmup`.
- **L3 = этот скилл.** Классифицирует запрос, выбирает слой(и), делает fan-out, merge'ит результаты, синтезирует ответ.

**Hard dependency:** скилл не работает без vault-semantic. Если ни один MCP tool сервера `vault-semantic` недоступен — выйди с сообщением:

> /vault-rag требует MCP-сервер `vault-semantic`. Добавьте модуль в `.claude/vault-manifest.yaml` и запустите /init-vault.

Не пытайся graceful-degrade до L1-only — пользователю, которому нужен L1 без L2, проще вызвать `vault_query` напрямую.

## Этап 0. Health-check индекса (lazy auto-reindex)

**До любого вызова L2** — vault-semantic MCP, tool `vault_semantic_stats()`.

| Сигнал | Реакция |
|---|---|
| `notes_total == 0` или tool падает с `INDEX_NOT_INITIALIZED` | Индекс пуст — нужен первичный reindex. См. §0.1 ниже: собери персональный ETA, спроси согласие, запусти `vault_semantic_reindex(scope='full')`. **НЕ запускать молча.** |
| `notes_total > 0`, всё ок | Продолжать. Lazy refresh-on-query (v0.2.0+) сам подхватит изменения через `incremental_refresh()` — ручной reindex обычно не нужен. |
| `last_indexed_at` сильно старше последнего изменения в волте (опц.) | Не блокирует — `search` сам обновит дельту. |

### 0.1. Первичный reindex — персональный ETA + согласие

Если `notes_total == 0` (или `INDEX_NOT_INITIALIZED`):

1. **Размер волта** — vault-index MCP, tool `vault_stats()` → берём `totalNotes`. Это источник истины для ещё-не-индексированного волта; L2-stats показывает 0, L1 знает реальное N.
2. **Torch variant** — прочитать `<vault>/.claude/modules/vault-semantic/.installed` (JSON, поле `torch_variant`). Может быть `cpu`, `cuda_cu126` или отсутствовать (старая установка до v0.4.0 — трактовать как `cpu`).
3. **ETA** — оценка по таблице (замеры на RTX 3070 Laptop / cp314+cu126, полная таблица — `modules/vault-semantic/README.md`):

   | Notes | CPU | CUDA |
   |---|---|---|
   | ≤ 25 | ~10 мин | ~2 мин |
   | ~50 | ~20 мин | ~4 мин |
   | ~100 | ~40 мин | ~8 мин |
   | ~250 | ~100 мин | ~19 мин |
   | ~500 | ~200 мин | ~38 мин |
   | ~1000 | ~400 мин | ~75 мин |

   Линейная экстраполяция между точками. Скорость на чужом GPU отличается (4090 быстрее на embedding-шаге, но CPU-bound chunking/lemmatize ставит потолок ~7x); цифры — ориентир, не SLA.

4. **Спросить пользователя** примерно так:

   > Семантический индекс пуст. В волте `N` заметок, текущий torch — `<variant>`. Первичная индексация займёт **~M минут**. Запустить сейчас?

   На CUDA-варианте дополнительно упомянуть, что это разовая операция — после неё `search/stats` сами подхватывают изменения через `incremental_refresh()` (2s debounce).

5. **При согласии** — vault-semantic MCP, tool `vault_semantic_reindex(scope='full')`, дождаться, повторить health-check. При отказе — выйти с сообщением «семантический слой недоступен; уточни запрос для L1-only или вызови `vault_query` напрямую». Не пытаться частично работать без L2 — см. контракт (hard dep).

Если оба источника недоступны (vault-index тоже не отвечает) — спросить юзера N приближённо («сколько у тебя примерно заметок — десятки, сотни, тысячи?») и взять верхнюю границу из таблицы.

## Этап 1. Декомпозиция запроса

Разбей `$ARGUMENTS` на компоненты:

### 1.1. Сущности

Конкретные имена видов, локаций, концептов, латинские названия. Используй их для:
- точного `vault_query` filter'а (`note_kind`, `aliases`, `path`)
- BM25-режима `vault_semantic_search` (mode='bm25')

### 1.2. Темы и фильтры

Маппинг на frontmatter (для L1) и lemma-токены (для L2):

| Тема в запросе | L1 filter (vault_query) | L2 query hint |
|---|---|---|
| Виды грибов, морфология | `note_kind=mushroom-card`, tags `edibility/*`, `biotope/*`, `hymenophore/*` | свободный текст про морфологию |
| Места сбора, биотопы | `note_kind=picking-location`, tags `region/*` | свободный текст про лес/болото/тип леса |
| Двойники, токсичность | `tags=lookalike|toxic|deadly-poisonous` | + `vault_lookalike_peers(path)` если знаешь центральную карточку |
| Приготовление, рецепты | `domain=recipe`, path `COOKING/` | свободный текст про метод/блюдо |
| Гайды, методики | `type=guide` | свободный текст |
| Мета о волте | `domain=system`, path `SYSTEM/` | редко нужен L2 — обычно L1 хватает |

### 1.3. Тип запроса (определяет shape ответа)

- **Сравнение** (X vs Y по критерию) → таблица + текст
- **Обзор группы** (все виды из рода / все локации региона) → список + закономерности
- **Сводка по аспекту** (двойники, сезон, термообработка) → структурированный список
- **Аналитика** (рекомендации, плюсы/минусы) → связный текст с обоснованием
- **Точечный fact-finding** («что в волте про X») → короткий ответ + ссылки

### 1.4. Регион / временной контекст

Если упомянут регион (Московская область, Мордовия, …) — добавь в L1 filter (`extra.foraging_regions` или поиск в text), в L2 — расширь query термином региона. По умолчанию приоритезируй Московскую область и Мордовию (см. CLAUDE.md → context).

Относительные даты («в этом сезоне», «за последний месяц») разверни в абсолютные ISO через `currentDate` из context'а.

## Этап 2. Routing — выбор слоя(ёв)

Применяй эти heuristics последовательно. Первая подходящая ветка побеждает.

### 2.1. L1 only — structural query

**Признаки:** в запросе только точные frontmatter-фильтры или граф-операции; нет нужды читать текст заметок.

| Признак | Что делать |
|---|---|
| «все mushroom-card с тегом X» | vault-index MCP, `vault_query(filter={note_kind: 'mushroom-card', tags: {any: ['X']}}, limit: 50)` |
| «что ссылается на Y» | vault-index MCP, `vault_backlinks(path='Y')` |
| «карточки конкретного рода / latin prefix» | vault-index MCP, `vault_query(filter={path: {prefix: 'MUSHROOM/Lepiota_'}})` либо просто Glob |
| «двойники для X» (есть точная карточка) | vault-index MCP, `vault_lookalike_peers(path='MUSHROOM/X.md')` |

Если результат полностью покрывает запрос — переходи к Этапу 4 (синтез). Если L1 дал список заметок, но запрос ожидает narrative-ответ из их **содержимого** — добавь L2-обогащение (см. 2.4).

### 2.2. L2 only — semantic search

**Признаки:** свободная формулировка, fuzzy match, отсутствие точных fingerprint'ов frontmatter.

```
# vault-semantic MCP
vault_semantic_search(
  query=<перифраза $ARGUMENTS или ключевые термины>,
  mode='hybrid',         # 95% случаев
  k=20,                  # для merge нужно широкое over-fetch
  min_score=0.4          # отсечь шум (для hybrid mode)
)
```

**Когда переключать `mode`:**
- `mode='bm25'` — латинские имена, технические токены (`H2S`), цитата, поиск тега как текста.
- `mode='dense'` — чисто semantic (без BM25-канала, если в hybrid слишком много BM25-noise'а от частых слов).
- `mode='hybrid'` — default.

Если hybrid вернул пусто — повтори `mode='bm25'`. Если и BM25 пусто — попробуй перифразу.

**Если L2 output превысил token limit** (типично при `k≥20` на текстовых волтах): harness сохранит результат в `tool-results/<hash>.txt` и попросит читать по частям. Файл — **корректный UTF-8 JSON**. Извлекать данные строго через **Read tool с `offset/limit`** или (если нужно структурное извлечение) `node -e "..."` / `jq`. **НЕ через `python3 -c "json.load(...); print(...)"` в Bash** — на Windows у python's stdout дефолтный cp1251, кириллица в JSON-тексте превратится в `��` mojibake (сам файл при этом UTF-8 целый; ломается только перекодирование при `print()`). Если без Python никак — добавь `PYTHONIOENCODING=utf-8` в env. Альтернативно — сузь L2 запрос (`k=10`, более специфичный query, `min_score` повыше) чтобы помещаться в token limit.

### 2.3. L1 + L2 fan-out (default для большинства аналитических запросов)

**Признаки:** есть И тематический фильтр, И семантическое содержание. Это **most common case** для запросов типа «сравни», «обзор», «сводка».

```
1. L1: vault_query(filter={...}, limit=50) → набор путей path_set
2. L2: vault_semantic_search(query=..., k=30, mode='hybrid')
3. Merge:
   - Path-intersection: оставляем chunks из L2, чьи note_path ∈ path_set
   - Dedup по chunk_id
   - Display-order — по L2 score (пересечение усиливает релевантность, но не переопределяет ranking)
```

Если intersection слишком пустая (< 3 chunks) — ослабь L1 filter (убери самое узкое поле) и повтори.

### 2.4. L1-обогащение L2-результата (графовый контекст)

После L2 нашёл центральную (top-1) заметку — обогатить контекст:
- `vault_backlinks(path=top.note_path)` — кто ссылается (часто это локации, гайды, обзоры)
- `vault_note_profile(path=top.note_path)` — frontmatter и outgoing-links

Это **дополнение**, не замена L2 ranking'а.

### 2.5. Похожие заметки

Если в запросе есть «похожие на X» / «соседи Y» — vault-semantic MCP, tool `vault_semantic_similar(path='MUSHROOM/X.md', k=10)`.

## Этап 3. Ранжирование и чтение

### 3.1. Сводный список кандидатов

Объедини результаты всех слоёв, дедуплицируй по `note_path`. Для каждого пути зафиксируй:
- score (L2) или relevance grade (L1)
- какие chunks нашлись (section_path, line_start–line_end)
- frontmatter (если L1) или `note_metadata` (если L2)

### 3.2. Ранжирование (0–3)

- **3** — путь прямо отвечает на запрос (ключевая сущность в имени или aliases)
- **2** — путь содержит значимую информацию по теме
- **1** — косвенно релевантный контекст
- **0** — упоминание мимоходом → отбросить

При равной релевантности бустить:
- `quality: verified` > `draft`
- свежий `updated`
- регион из user_regions

### 3.3. Чтение top-K

Прочитай top-15 файлов через Read. Для каждого:
- зафиксируй frontmatter (note_kind, domain, edibility, region, tags, updated)
- извлеки релевантные секции (используй `section_path` из L2-результатов как hint)
- собери WikiLinks на другие заметки релевантной темы (кандидаты на дочитывание)

Файлы >200 строк — читай с offset/limit по нужным секциям (заголовки `##`).

## Этап 4. Синтез ответа

### 4.1. Стиль

- **Язык:** русский (если запрос русский) или язык запроса.
- **Латинские названия** — как есть, курсивом не нужно.
- **WikiLinks** — при **первом** упоминании каждой заметки. Не дублировать.
- **Таблицы** — для сравнений и сводок. В WikiLink display-text внутри таблиц экранировать только `|` как `\|` (см. CLAUDE.md → правила форматирования).
- **Безопасность:** если запрос затрагивает ядовитые виды или двойников — явно отметить степень опасности (`deadly-poisonous`/`poisonous`), не маскировать.
- **Конфликты:** если две заметки противоречат — указать обе версии с пометкой и WikiLinks.
- **Лаконичность:** не пересказывать заметки целиком, агрегировать релевантное.
- **Регионализация:** при упоминании сезонности/биотопа — приоритезировать регионы из user_regions.

### 4.2. Маркировка внешних знаний

Любое утверждение, которое не подтверждено заметками волта (общие знания модели или web search) — помещать в callout:

```markdown
> [!info] Внешние знания
> - Род Lepiota содержит >400 видов мировой флоры — обзор, в волте только N карточек
> - Аматоксины разрушаются только при >280 °C — техническая деталь, не описана в волте
```

Если весь ответ из волта — блок не нужен.

### 4.3. Полнота

Если волт не содержит достаточной информации — чётко указать что отсутствует (нет карточки X, нет описания биотопа Y и т.п.).

## Этап 5. Запись в `RAG/` (опционально)

**По умолчанию НЕ записывать.** Запись делать только если:
- Пользователь явно попросил («сохрани в RAG», «запиши ответ», «/vault-rag --save»).
- Запрос явно analytical и ответ объёмный (>20 строк markdown, агрегация 5+ заметок) — тогда **спросить** пользователя «сохранить ответ в RAG/?» перед записью.

### 5.1. Структура файла

```markdown
---
type: reference
domain: <mushroom|location|safety|recipe|equipment|system>
stability: ephemeral
priority: low
note_kind: rag-answer
tags:
  - rag-answer
  - <дополнительные тематические теги>
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>
query: "<исходный запрос из $ARGUMENTS>"
quality: draft
---

## <краткий заголовок ответа>

<Связный текст. WikiLinks при первом упоминании.>

### Источники (vault)
- [[Note_1]] — что взято
- [[Note_2]] — что взято

> [!info] Внешние знания
> <если есть>
```

**Не записывать `co_authored`** в frontmatter — это поле проставляется только через `/verify` после подтверждения пользователем (см. user feedback).

### 5.2. Именование файла

`RAG/<YYYY-MM-DD>_<Topic_PascalCase>.md`

- `Topic_PascalCase` — 3–5 слов по сути запроса, ASCII-транслитерация, разделитель `_`
- Только ASCII в имени (упрощает работу с git/CLI на Windows)
- Никаких пробелов, тире, кириллицы, точек кроме расширения

Примеры:
- «сравни Macrolepiota Procera и Chlorophyllum Rhacodes» → `2026-05-18_Macrolepiota_vs_Chlorophyllum.md`
- «обзор условно-съедобных Подмосковья» → `2026-05-18_Conditionally_Edible_Moscow.md`
- «двойники в роде Lepiota» → `2026-05-18_Lepiota_Lookalikes.md`

### 5.3. Создание папки

Если `RAG/` нет — создать (`mkdir -p RAG/`). Если в волте нет упоминания папки `RAG/` в `Vault_architecture.md` — добавить строку в таблицу папок.

### 5.4. Reindex после Save

После записи файла — vault-semantic MCP, tool `vault_semantic_reindex(scope='changed')`. Без этого следующий вызов /vault-rag не найдёт только что сохранённый ответ. Lazy refresh-on-query (v0.2.0+) должен подхватить, но явный reindex быстрее и детерминированнее. Если tool недоступен или падает — warning, не фатально.

## Этап 6. Финальный вывод пользователю

Что вывести (вне записанного файла, в чат):

1. **Ответ** — основной результат (если не записан в RAG/) или короткое резюме (3–5 предложений) + путь к файлу (если записан).
2. **Использованные слои** — короткая помета: «L1 only», «L2 only», «L1+L2 fan-out» — для прозрачности routing'а.
3. **Использованные источники** — список WikiLinks верхнего уровня.

## Крайние случаи

- **Ничего не найдено** (L1 пуст, L2 пуст) — сообщить «По запросу не найдено релевантных заметок». Файл в `RAG/` **НЕ создавать**. Предложить уточнить запрос или сменить mode.
- **Слишком много** (>30 релевантных) — приоритизировать по `quality: verified`, `priority: high`, свежему `updated`. В ответе явно отметить «выборка ограничена top-15».
- **Запрос вне волта** («как настроить мотокультиватор») — ответить из волта что есть, остальное — в callout `> [!info] Внешние знания`. Если волт молчит совсем — сообщить и НЕ писать файл.
- **Запрос на идентификацию по фото** — скилл текстовый; перенаправить на `/audit-images` или ручной разбор.
- **Vault_semantic_reindex падает с EMBEDDING_MODEL_UNAVAILABLE** — модель bge-m3 не загрузилась (нет интернета на первом скачивании, или OOM). Сообщить пользователю, не пытаться обойти.

## Пример сценария: «сравни Macrolepiota Procera и Chlorophyllum Rhacodes по риску спутывания»

1. **Health-check:** `vault_semantic_stats()` → `notes_total: 312, last_indexed_at: ...` — ок.
2. **Декомпозиция:** сущности `Macrolepiota_Procera`, `Chlorophyllum_Rhacodes`; тема — двойники, edibility-контраст; тип — сравнение.
3. **Routing — L1+L2 fan-out:**
   - L1: `vault_query(filter={note_kind: 'mushroom-card', path: {any: ['MUSHROOM/Macrolepiota_Procera.md', 'MUSHROOM/Chlorophyllum_Rhacodes.md']}})` + `vault_lookalike_peers(path='MUSHROOM/Macrolepiota_Procera.md')`.
   - L2: `vault_semantic_search(query='Macrolepiota procera Chlorophyllum rhacodes отличия двойники', k=20, mode='hybrid')`.
