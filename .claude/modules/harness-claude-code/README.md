# harness-claude-code module

Патчит `~/.claude.json`: регистрирует MCP-серверы, заявленные другими установленными модулями, под уникальным namespace.

## Что делает (v0.1.0)

- Backup `~/.claude.json` (`<file>.bak.<ISO-timestamp>`, ротация — последние 5).
- Читает manifest, находит установленные модули с `provides.mcp_server`.
- Детектирует layout vault-index'а (legacy / canonical) и подставляет правильный путь к `dist/index.js`.
- Идемпотентно патчит **project-scope** секцию: `projects["<vault_root>"].mcpServers["<server_name>"]`. Если значение совпадает — no-op (backup не пишется).
- Пишет sub-block в CLAUDE.md.

## Зависимости

- `core` (нужен outer managed-block в CLAUDE.md).
- `vault-index` (источник `provides.mcp_server` в текущей версии).

## Namespace стратегия: project-scope

Claude Code хранит MCP-сервера в двух местах:
- Top-level `mcpServers` — глобально доступные из любого проекта.
- `projects["<абс_путь>"].mcpServers` — привязанные к конкретному проекту.

Этот модуль **использует project-scope**, потому что:
1. Project key уже даёт уникальность между волтами на одной машине — не нужен суффикс-slug в имени сервера.
2. MCP-серверы инфраструктуры волта (vault-index и т.п.) логически принадлежат проекту, а не глобально.
3. Совпадает с тем, как Claude Code сам регистрирует серверы при ручном добавлении.

В результате имя сервера остаётся простым `vault-index`, а изоляция между волтами — через project key.

## Ограничения v0.1.0

- Читает `provides.mcp_server` только у `vault-index`. Универсальная итерация — следующая версия.
- Не поддерживает удаление: если модуль исчез из манифеста, его регистрация в `~/.claude.json` остаётся (потребуется ручная чистка или будущий `remove.mjs`).
- Не валидирует, что MCP-сервер фактически запускается — только что путь к binary существует.
